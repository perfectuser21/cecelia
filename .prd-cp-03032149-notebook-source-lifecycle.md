# PRD: NotebookLM Source 生命周期管理 + 对账机制

## 背景

当前 NotebookLM 压缩流程（日→周→月）有两个隐患：

1. **Source 污染**：压缩后旧 source 不删除。月合成时 NotebookLM 同时读取原始日条目 + 周合成结果，导致内容被重复计算、污染权重。
2. **无可观测性**：DB 不存 source_id，不知道 DB 记录和 NotebookLM 实际 source 是否一致。出错了无从发现。

## 目标

1. 压缩后自动删除下级 source（7条日 → 1条周合成后删7条日）
2. 每次写入 NotebookLM 时记录返回的 source_id 到 DB
3. 提供对账 endpoint 让 Cecelia 可以自检
4. 差异超阈值时触发 ALERT 事件到丘脑

## 具体需求

### 1. Migration 109: synthesis_archive 加 notebook_source_id 字段
- `ALTER TABLE synthesis_archive ADD COLUMN notebook_source_id VARCHAR(255)`
- 存写入 NotebookLM 时返回的 source_id

### 2. notebook-adapter.js: addTextSource 返回 source_id
- 当前 `addTextSource` 调用 bridge，但不返回 source_id
- 修改为解析 bridge 响应并返回 `{ success, sourceId }`

### 3. rumination-scheduler.js: 压缩完成后删除下级 source
- 周合成完成后：删除对应7条日 source（从 synthesis_archive 取 notebook_source_id）
- 月合成完成后：删除对应4条周 source（从 synthesis_archive 取 notebook_source_id）
- 删除失败不阻断流程（fire-and-forget，记录日志）

### 4. 对账 endpoint: GET /api/brain/notebook-audit
- 查询 DB 中所有 synthesis_archive 的 notebook_source_id（非 NULL）
- 调用 NotebookLM 获取实际 source 列表（按 notebook 分组）
- 对比差异：DB 有但 NotebookLM 没有（孤儿记录）/ NotebookLM 有但 DB 没有（未追踪）
- 返回 `{ matched, orphaned_in_db, untracked_in_notebooklm, audit_time }`

### 5. 差异告警
- orphaned_in_db > 2 时：fire processEvent('ALERT', { type: 'notebook_audit_drift', ... })

## 成功标准
- 周合成后 DB synthesis_archive 对应日条目的 notebook_source_id 非 NULL
- 周合成后 NotebookLM 中对应日 source 被删除
- GET /api/brain/notebook-audit 返回正确的对账结果
- 测试：rumination-scheduler 单元测试覆盖 source 删除逻辑
- DevGate 通过，CI 通过
