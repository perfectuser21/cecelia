# Task Card: [SelfDrive] 任务失败率根因修复 — error_message 未写入

## 需求（What & Why）

**功能描述**: 修复 execution callback 中 `error_message` 未写入问题，确保失败任务始终有可读的错误原因。

**背景（根因分析结果）**:
过去24h 任务状态分布（总计624条）：
- completed: 263 (42.15%)
- pipeline_rescue quarantined: 174 (27.88%) → 已被 PR #2019 修复（MAX_RESCUE_QUARANTINE）
- content_publish queued 积压: 104 (16.67%) → rescue 风暴期间 slot 被挤占，现 slot=1 正在慢慢消化
- dev quarantined (liveness_dead): 15 (2.40%) → failure_class=auth（API 过载），外部原因
- dev/code_review paused: 21 (3.36%) → Harness 评测任务，正常暂停

**4类根因分类**:
1. **根因A (已修复)** pipeline_rescue 循环风暴 → PR #2019 MAX_RESCUE_QUARANTINE
2. **根因B (外部)** API 过载 liveness_dead → failure_class=auth，smart-retry 应对
3. **根因C (自愈中)** content_publish 积压 → rescue 风暴消退后 slot 自动恢复
4. **根因D (本PR修复)** error_message 未写入 → callback 状态竞争导致 error_message=NULL

**根因D 详细**:
- 当 watchdog 改变任务状态（→quarantined）后 callback 才到达
- 主 UPDATE `WHERE status='in_progress'` 不匹配 → error_message 不写
- 分类 payload UPDATE `WHERE id=$1`（无状态检查）仍运行 → failure_class/failure_detail 写入
- 结果: failure_class=task_error 在 payload，error_message=NULL，无法诊断

**不做什么**: 不处理 content_publish 积压（自愈中），不改动 pipeline_rescue（已修复）

## 成功标准

> [BEHAVIOR] 当任务被 watchdog 先隔离、callback 后到达时，error_message 仍被写入
