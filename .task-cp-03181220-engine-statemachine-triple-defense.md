# Task Card: Engine 状态机三层防御 P0+P1

## 需求（What & Why）

**功能描述**: 堵住状态机的两条绕行路，并引入验签机制形成双重兜底：
1. **bash-guard.sh 补丁**：拦截 AI 用 Bash 工具直接写 `.dev-mode` 文件（如 `echo "step_2_code: done" >> .dev-mode.*`），调用 `verify-step.sh` 做同等验证
2. **verify-step.sh 验签**：步骤通过验证后写 `.dev-seal.${BRANCH}` 文件（`step_N_xxx: verified@timestamp`）
3. **stop-dev.sh 兜底**：Stop Hook 检查每个标记为 `done` 的关键步骤是否有对应验签，无验签则 `exit 2` 强制补验

**背景**:
- 上一个 PR (#1064) 只保护了 PreToolUse:Write/Edit 路径
- Bash 工具绕行：`echo "step_2_code: done" >> .dev-mode.*` 完全绕过 branch-protect.sh
- Stop Hook 不检查步骤验签，即使 AI 绕过所有写入拦截，Stop Hook 也不会发现
- 三层防御才能做到：单层被绕 → 下层兜底

**不做什么**:
- 不实现 Verifier Subagent（那是 P2，需要 agent 协议）
- 不修改 devloop-check.sh 的 PR/CI/Learning 检查逻辑
- 不改变 .dev-mode 文件格式（seal 写入独立的 .dev-seal 文件）

## 成功标准

1. [ARTIFACT] `bash-guard.sh` 包含 `.dev-mode` Bash 写入检测逻辑，命中时调用 `verify-step.sh`
2. [ARTIFACT] `verify-step.sh` 通过验证后写 `.dev-seal.${BRANCH}` 验签文件
3. [BEHAVIOR] Bash 工具执行 `echo "step_2_code: done" >> .dev-mode.*` 时被拦截，verify-step.sh 验证失败则 exit 2
4. [BEHAVIOR] stop-dev.sh 检测到 `step_N: done` 无对应验签时输出警告并 exit 2
5. [GATE] CI 全部通过
