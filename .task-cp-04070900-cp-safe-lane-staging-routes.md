# Task Card: Safe Lane CD — Brain 侧 staging 路由实现

## 需求（What & Why）

**功能描述**: 在 `packages/brain/src/routes/ops.js` 中添加 3 个 staging 专用路由：
1. `GET /api/brain/deploy/staging/status` — 返回 staging deployState 状态
2. `POST /api/brain/deploy`（mode=staging / staging=true）— 部署到端口 5222 的 staging 实例
3. `POST /api/brain/deploy/staging/cleanup` — 清理 staging 环境

**背景**: deploy.yml 的 Safe Lane 路径（高风险 PR：改 thalamus/tick/ops 等核心文件）会：
1. POST /api/brain/deploy（带 staging:true）触发 staging 部署
2. 轮询 GET /api/brain/deploy/staging/status 等待 staging 就绪
3. 对 staging Brain（端口 5222）执行 smoke test
4. staging 成功后才放行 production deploy
当前 ops.js 缺少这 3 个路由，导致 Safe Lane PR 一律 404 失败。

**不做什么**: 不修改现有 production deploy 逻辑；不创建 docker-compose.staging.yml（用脚本方式实现）

## 成功标准

> staging 路由正确响应，Safe Lane 流水线不再 404

## 实现方案

**要改的文件**: `packages/brain/src/routes/ops.js`

**新增内容**:
1. `stagingDeployState` — 独立的 in-memory staging 状态对象（与 deployState 隔离）
2. `POST /api/brain/deploy` 扩展 — 检测 `req.body.staging === true` 或 `req.body.mode === 'staging'`，走 staging 部署分支
3. `GET /api/brain/deploy/staging/status` — 返回 `{ ...stagingDeployState }`
4. `POST /api/brain/deploy/staging/cleanup` — 停止 staging 容器/进程，重置 stagingDeployState

**staging 部署实现**: 用 `scripts/staging-deploy.sh`（不存在则降级为 PM2 port 5222 方式）

**不改什么**: 现有 `deployState`、`GET /deploy/status`、`POST /deploy`（production）、`POST /deploy/rollback` 全部不动
