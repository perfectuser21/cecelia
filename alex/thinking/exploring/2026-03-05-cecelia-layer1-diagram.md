## 拆解的完整生命周期（放大看）

```
一个 KR 从"没有任务"到"代码合并"的完整路径：

1. KR 存在于 goals 表（status='ready'）
         │
2. 拆解检查器发现：这个 KR 下面没有 Task
         │
3. 创建拆解任务：task_type='okr', status='queued'
         │
4. 规划器选中这个拆解任务
         │
5. 执行器派给秋米（/okr）
         │
6. 秋米拆解：
   KR → Project → Initiative → Task（多个）
         │
7. Vivian 审查拆解质量
   ├── 不通过 → 秋米重拆（回到步骤 6）
   └── 通过 → 新 Tasks 状态变为 'queued'
         │
8. 下一轮 Tick，规划器看到新的 queued Tasks
         │
9. 规划器选中一个 dev Task
         │
10. 执行器派给 Caramel（/dev）
         │
11. Caramel 写代码 → PR → CI → 合并
         │
12. 回调 Brain → 任务状态 = 'completed'
         │
13. Initiative 下所有 Task 完成？
    ├── 否 → 下一轮 Tick 继续选下一个 Task
    └── 是 → initiative-closer 自动关闭 Initiative
              │
14. Project 下所有 Initiative 完成？
    ├── 否 → 继续
    └── 是 → Project 关闭 → KR 进度更新
```

## 总结：第 1 层只有这些组件

| 组件 | 类型 | 做什么 |
|------|------|--------|
| Tick | L0 | 心跳，驱动一切 |
| 拆解检查器 | L0 | 检查哪些 KR 需要拆解 |
| 规划器 | L0 | 从队列里选最优任务 |
| 执行器 | L0 | 派发任务给外部 Agent |
| 任务监控 | L0 | 检查进程存活、超时 |
| 丘脑 | L1 | 事件判断，简单自己处理 |
| 皮层 | L2 | 复杂事件深度分析 |
| 保护系统 | L0 | 警觉/熔断/隔离/看门狗 |
| 秋米/Caramel/小检/小审 | 外部 | 被召唤的 Agent |
| Vivian | 外部 | 拆解质量审查 |
| OKR 闭环 | L0 | Initiative/Project/KR 自动关闭 + 进度更新 |
