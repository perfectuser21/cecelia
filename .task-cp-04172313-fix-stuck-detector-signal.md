# C3 — stuck-detector 改看活动信号（git + CI），不只看时间

Brain task_id: `5d877e5a-1fd1-442b-a71a-435ca9c9ed93`

## 背景
今天 3 个任务（A1 `e7fe7fa4` / `75c0e524` / F4 `eace8df9`）被 stuck-detector 误 cancel：PR 在 CI 等待期间没状态变化，被当成 stuck。

## 根因
`packages/brain/src/pipeline-patrol.js` 的 `checkStuck` 只看 `.dev-mode` 文件 mtime 距今的时间窗口（stage_2/1 是 20 分钟，stage_3 是 90 分钟），没看实际是否还活跃。
stage_3_integrate 虽然阈值 90 分钟，但 stage_2_code 的 20 分钟阈值仍然窄：Claude 在写代码时 mtime 不刷新的话会被误判。
更核心的问题：只要在 stuck 判定里任一活动信号存在（git commit / CI check-run），就不该 cancel。

## 修复
在 `pipeline-patrol.js` 的 `runPipelinePatrol` 里，当 `checkStuck` 返回 stuck=true 后，额外调用 async helper `hasRecentActivity(branch)`：
1. `hasRecentGitActivity(branch, minutes=10)` — 查 branch 最近 10min 是否有本地 / origin commit
2. `hasRecentCiActivity(branch, minutes=10)` — 通过 `gh pr view` 查 branch 对应 PR 的 check-run 更新时间

两个都无 + 时间超阈值 = 真 stuck。任一有 = 活跃，跳过 rescue 创建。

若 branch 没有 PR 且 git 查询失败（比如孤儿 branch），退化到旧行为，保持向后兼容。

## 范围
