# Task Card: feature.yaml 自注册机制 — 消除 CI 5 处手动同步

## 需求（What & Why）

**功能描述**：
现在每加一个新功能，需要手动触碰 5 个地方：
1. `packages/engine/features/feature-registry.yml` — 登记功能
2. `regression-contract.yaml` — 登记合约
3. 版本号（多个文件）— 手动 bump
4. 手动跑 `generate-path-views.sh` — 同步派生文档
5. `docs/learnings/` — 每次 PR 一个 Learning

**目标**：每个功能目录放一个 `feature.yaml`，CI 扫描它，自动完成同步验证。新功能加进来不需要改任何中央脚本。

**背景**：清点了 124 个 Learning 文件，版本同步相关失败占 ~45%，registry/contract 漂移占 ~25%。这是系统最高频的失败来源。

**本次 PR 的边界（不做什么）**：
- 不替换现有 24 个必要 CI 检查（这些有真实事故来源，不动）
- 不要求所有现有 skill 立刻迁移，只做 dev/audit/assurance/qa 四个 skill 作为 proof of concept
- 不改变 Learning 文件机制（它是组织记忆，不是技术问题）
- 版本号多文件 bump 问题由 `auto-version.yml` 处理，本 PR 只解决 registry + path-views 两个手动步骤

## 成功标准

1. [ARTIFACT] `packages/engine/skills/dev/feature.yaml`、`audit/feature.yaml`、`assurance/feature.yaml`、`qa/feature.yaml` 四个文件存在，格式符合 JSON Schema
2. [ARTIFACT] `packages/engine/scripts/scan-feature-yamls.mjs` 存在，能扫描所有 `feature.yaml` 并输出摘要
3. [BEHAVIOR] `generate-path-views.sh` 在 `feature-registry.yml` 变更时自动触发（通过 pre-commit hook），不需要手动运行
4. [BEHAVIOR] CI L2 新增 `feature-yaml-lint` job，验证所有 `feature.yaml` 格式正确、与 `feature-registry.yml` 条目对应
5. [GATE] 所有现有 CI 检查仍然通过（L1/L2/L3/L4 全绿）
