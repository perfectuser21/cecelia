# Task Card: devloop-check 会话压缩恢复入口

## 背景

会话压缩重启后，agent 无法知道自己在哪个 stage。
`.dev-mode.*` 被 gitignore，worktree 消失后 stop hook 找不到当前 stage 进度。
当前 `devloop-check.sh` 只是 library（供 stop hook source），没有直接执行入口。

P1（worktree 路径持久化）已在 Engine 13.75.2 合并。本任务为 P2：
让 `bash packages/engine/lib/devloop-check.sh` 直接可用作会话恢复入口。

## 目标

当 agent 会话压缩重启后，第一步执行：
```
bash packages/engine/lib/devloop-check.sh
```
即可看到：
- 当前分支和 .dev-mode 文件路径
- 当前处于哪个 stage（step_1_spec/step_2_code/step_3_integrate/step_4_ship）
- 缺少哪些 seal 文件
- 下一步应该做什么（action）

## 实现方案

**文件改动（2 个）**：

1. `packages/engine/lib/devloop-check.sh` — 添加直接执行入口
   - 在文件末尾加 `if [[ "${BASH_SOURCE[0]}" == "${0}" ]]` 块
   - 自动搜索当前 git worktree 下所有 `.dev-mode.*` 文件
   - 调用 `devloop_check BRANCH DEV_MODE_FILE` 获取状态
   - 格式化输出人类可读的诊断信息（当前 stage + 缺什么 + 下一步）
   - 不改任何现有函数（纯追加，不破坏 source 用法）

2. `packages/engine/features/feature-registry.yml` — 新增 changelog 条目

## 成功标准

会话压缩重启后，agent 执行 `bash packages/engine/lib/devloop-check.sh` 可以在 5 秒内获得：
1. 当前分支名和 stage 状态
2. 缺失的 seal 文件列表（如有）
3. 下一步应执行的具体 action

不破坏现有 stop hook 对 `devloop_check()` 的 source 调用。
