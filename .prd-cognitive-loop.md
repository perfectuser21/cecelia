# PRD: Cecelia 认知闭环架构

## 背景

Cecelia 现在是一堆聪明的模块，但没有形成真正的闭环：

- **信号入口分散**：嘴巴、飞书、Rumination、Goal Evaluator 各走各的路，没有统一大脑
- **决策无记忆**：丘脑 L1 每次从零开始判断，不知道 Cecelia 自己的历史/情绪/用户偏好
- **Plan 与丘脑分裂**：/plan Skill 和丘脑是两个独立的东西，路由逻辑分散
- **L2（Opus 皮层）从未触发**：皮层定义了但从来没被真正激活
- **Rumination/Goal Evaluator 绕过丘脑**：直接写 DB，产出的洞察没有被统一处理

## 目标

让 Cecelia 形成完整认知闭环：**自己的历史影响自己的下一个决策**。

## 架构设计

### 统一信号流（L0/L1/L2 决策层）

```
所有信号来源
（嘴巴 / 飞书 / Rumination / Goal Evaluator / Tick 内部）
        ↓
   L0（纯代码）
   快速判断：能直接处理？→ 处理
   复杂？→ 往下传
        ↓
   L1（丘脑 = 加载了 /plan Skill 的 LLM）
   ├── 调 buildMemoryContext() 注入记忆
   ├── 识别信号类型 + OKR 层级
   ├── 路由到具体行动（coding/research/remember/automate/KR重规划）
   └── 判断是否需要 L2 深度分析
        ↓（仅在需要时）
   L2（Opus 皮层）
   深度战略分析 → 结论返回 L1 重路由
```

### Plan + 丘脑融合

| 现在 | 改后 |
|------|------|
| /plan 是独立 Skill，丘脑是独立 LLM | 丘脑直接加载 /plan Skill |
| 两套路由逻辑并存 | 统一：丘脑 = 会路由的认知大脑 |

### 五条闭环线（4+1）

**第1条：Rumination → 内在状态（RUMINATION_RESULT 协议）**
- Rumination 产出从"直接写 DB"改为发结构化信号给丘脑
- 格式：`{ learnings[], self_updates[], emotion_deltas[], actions[] }`
- 丘脑决定哪些写 memory_stream，哪些触发 action

**第2条：行动结果 → 内在状态（回力镖）**
- 任务完成/失败后，结果作为信号回到丘脑
- 丘脑判断：更新 self_model？触发 Rumination？重规划？

**第3条：对话 → user_profile（另行处理，本期不做）**
- 由 pure-consciousness PR 统一处理，对话层已在另一 terminal 开发中
- 本期不涉及

**第4条：L2 结论 → self_model + policy 沉淀**
- Opus 深度分析后，结论写入 memory_stream（type='self_model'）
- 形成 Cecelia 长期自我认知积累

**第5条（+1）：内在状态 → 决策（brain_context 注入 L1）**
- 每次 L1 被调用前，先调 `buildMemoryContext(signal)`
- 注入：emotion_summary + self_key_facts + user_profile_slice + recent_narrative
- 这是让所有闭环"真正起作用"的那一条线

### brain_context 实现

复用已有的 `buildMemoryContext()`（memory-retriever.js 已实现向量检索 + L0/L1/L2 三层记忆），无需重新造。

```
收到信号
    ↓
buildMemoryContext(query=信号内容)
    ↓ 向量搜索 memory_stream
    ↓ 拿 L1 结构化摘要注入
L1 prompt = 信号 + 相关记忆
    ↓
更有"记忆"的决策
```

## 具体需求

### P0：统一信号入口

1. 嘴巴（orchestrator-chat.js）：发 `USER_MESSAGE` 信号给丘脑，不再自己路由
2. 飞书（feishu event handler）：发 `USER_MESSAGE` 信号给丘脑

### P0：Plan + 丘脑融合

3. 丘脑 L1 prompt 直接加载 /plan Skill 内容
4. /plan 路由扩充：增加 `research / remember / automate / kr_replan` 路由类型（现在只有 coding）

### P1：brain_context 注入

5. 丘脑调 L1 前，先调 `buildMemoryContext(signal)` 并注入 prompt
6. 复用 memory-retriever.js 现有实现，不重新造

### P1：Goal Evaluator → 走丘脑

7. Goal Evaluator stalled 判断后，发事件给丘脑，而非直接 createInitiativePlanForStall()

### P2：Rumination → RUMINATION_RESULT 协议

8. Rumination 产出改为结构化信号发给丘脑
9. 丘脑决定哪些内容写哪个表

### P2：激活 L2（定义明确触发条件）

10. L2 触发条件：
    - KR 连续停滞 > 14 天
    - 任务失败率 > 60%（7天窗口）
    - L1 检测到方向冲突
    - 用户明确要求战略复盘
11. L2 产出写入 memory_stream（type='self_model'）后返回 L1 重路由

## 非目标

- 不改 memory_stream 表结构（L0/L1/L2 三层已有）
- 不引入新的向量数据库（pgvector 已有）
- 不改 Rumination 的每日预算限制逻辑
- Suggestion Dispatcher 本期不动

## 成功标准

1. 所有信号来源统一走丘脑，无旁路
2. 丘脑 L1 每次决策都带 brain_context（可验证：日志有 `[brain_context]` 输出）
3. /plan 路由覆盖 5 种意图类型（coding/research/remember/automate/kr_replan）
4. L2 在满足触发条件时被调用（DB 可查 cortex_route > 0）
5. Rumination 产出能在 memory_stream 中追踪到对应的 L1 结构化摘要
