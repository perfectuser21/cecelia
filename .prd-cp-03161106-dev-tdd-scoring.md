# PRD: /dev 加入 TDD 两阶段探索 + 任务评分

## 背景

/dev 当前问题：
1. 先探索代码再写 DoD → accommodation bias（写容易通过的测试，不是真正验证需求）
2. 任务完成后只有 completed/failed，看不到质量高低，无法优化

## 需求

### 改动 1：TDD 两阶段探索（02-code.md）

把当前 Step 2.1（单次大探索）拆成三个子步骤：

**Step 2.0（新）：浅扫接口层**
- 只读：API 路由文件、类型定义、接口声明、现有 test 文件
- 不读：具体实现逻辑
- 时间限制：≤5分钟，≤5个文件
- 产出：相关接口摘要（endpoint/函数签名/数据结构）

**Step 2.1（新）：Test Designer Subagent**
- 输入：PRD + Step 2.0 接口摘要（不能读实现代码）
- 输出：每条 DoD 对应的 Test: 命令（bash/curl/node，必须当前 FAIL）
- Verifier 确认测试确实 FAIL（如果全部 PASS 说明写错了）
- DoD Test: 字段此时写入并锁定，之后不能修改

**Step 2.2（改名）：深度实现探索**
- 目标变为：如何让 Step 2.1 写的测试通过
- 允许读所有代码
- DoD 已锁定，不能再改 Test: 字段

### 改动 2：评分输出

**Step 1 末尾（01-taskcard.md）：置信度自评**

agent 在 Step 1 完成时输出并写入 .dev-mode：
```
confidence_score: 7
confidence_reason: PRD 清晰，但涉及并发逻辑有不确定性
confidence_risk: tick 循环的并发安全边界条件
```

**Step 5 开始（05-clean.md）：执行质量自评**

agent 在清理前计算并输出：
- CI 效率分（1-5）：1次通过=5，2次=4，3次=3，4+=1
- DoD 诚实度分（1-5）：Test: 命令有没有经历修改
- 循环效率分（1-5）：Stop Hook 循环次数
- Learning 质量分（1-5）：有无根本原因+预防措施

总分 = 四项平均 × 2（换算到 10 分制）

输出格式：
```
执行质量：7/10
- CI 效率：4/5（第2次通过）
- DoD 诚实度：4/5
- 循环效率：3/5
- Learning 质量：5/5
可优化：CI 失败原因是 DoD [BEHAVIOR] 格式，下次 Test Designer 需注意
```

写入 Brain API（如果有 --task-id）：
PATCH /api/brain/tasks/{task_id} 
body: { custom_props: { confidence_score: 7, execution_quality: 7, quality_details: {...} } }

## 成功标准

1. 02-code.md 有明确的三段式：浅扫 → Test Designer Subagent → 深度探索
2. Test Designer Subagent 的 prompt 明确禁止读实现代码
3. DoD Test: 字段在 Step 2.1 后锁定，Step 2.2 不能修改
4. 01-taskcard.md Step 1 末尾有置信度输出格式（写入 .dev-mode）
5. 05-clean.md 有四维评分计算 + 输出格式 + Brain API 写入

## 不做什么

- 不改 Brain DB schema（custom_props 已支持任意字段）
- 不改 Dashboard 显示（那是另一个任务）
- 不改 Stop Hook 逻辑
