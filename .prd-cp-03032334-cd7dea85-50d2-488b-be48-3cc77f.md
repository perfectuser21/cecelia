# PRD: 修复 initiative_plan 任务超时和进程丢失问题

## 背景

从 Brain API 查询的 failed 任务显示，**initiative_plan** 类型的任务持续失败：

### 失败模式

1. **3 个任务因进程消失失败**（`orphan_detected` / `process_disappeared`）
   - Brain 重启后检测到任务 `in_progress` 但进程不存在
   - 被标记为 `failed`

2. **1 个任务超时**（62 分钟，超过 60 分钟限制）

3. **所有任务都是同一个 Initiative**：
   - Initiative ID: `498a953e-17be-4825-ab19-4efc81a6f970`
   - Initiative 名称: "8平台发布卡点摸排"
   - 所属 KR ID: `f59a22f2-d323-4100-bc73-644c105252aa`

### 任务描述

```
请为 Initiative「8平台发布卡点摸排」规划下一个 PR。

你的任务（initiative_plan 模式）：
1. 读取 Initiative 描述（GET /api/brain/projects/<initiative_id>）
2. 读取已完成的 PR 列表（GET /api/brain/tasks?project_id=<initiative_id>&status=completed）
3. 判断 Initiative 目标是否达成
   - 已达成 → 标记 Initiative completed，不创建新任务，结束
   - 未达成 → 规划下一个 PR，写入 tasks 表一条 dev 任务
```

## 问题分析

### 1. 超时原因（推测）

- **initiative_plan 任务可能陷入死循环或无限等待**
- 可能在等待某个 API 响应时卡住
- 可能在执行 `/dev` 子流程时超时

### 2. 进程消失原因（推测）

- Brain 服务重启（可能因为：崩溃、手动重启、OOM）
- 执行中的 initiative_plan 进程被杀死
- Brain 重启后无法恢复任务状态

### 3. 根本原因（待验证）

需要检查：
- `task-router.js` 中 initiative_plan 的路由逻辑
- `executor.js` 中任务执行的超时处理
- Brain 重启后的任务恢复机制

## 目标

**修复 initiative_plan 任务的可靠性问题**

1. **防止超时**：
   - 添加明确的超时控制
   - 分解长任务为小步骤
   - 添加进度检查点

2. **处理进程消失**：
   - Brain 重启时正确恢复任务
   - 或标记为可重试（而非直接 failed）
   - 添加任务状态持久化

3. **提升可观测性**：
   - 添加详细日志记录
   - 记录任务执行的关键步骤
   - 便于定位失败原因
