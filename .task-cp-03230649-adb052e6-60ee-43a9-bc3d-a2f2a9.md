# Task Card: PR3 — 迁移业务代码从 goals/projects 切换到新 OKR 层级表

## 需求（What & Why）
**功能描述**: migration 181 已有触发器同步写入，PR#1461/1462 已切 tick/routes。
还有业务文件在 SQL 里直接查旧 goals/projects 表，需迁移到新表。

**背景分析**:
- 旧表 `goals` → 新表 `objectives`（type='area_okr'）、`visions`（type='vision'）、`key_results`（type='area_kr'等）
- 旧表 `projects` → 新表 `okr_projects`、`okr_scopes`、`okr_initiatives`
- 关键约束：goals.id == objectives.id（UUID 相同，migration 179 原地复制）
- 状态值：objectives 中现有数据与 goals 相同（'pending'/'in_progress'），
  新写入通过 trigger 映射 'in_progress'→'active'，故查询需用 IN ('active','in_progress')
- projects.kr_id 指向 goals.id(area_okr) == objectives.id，不能用 okr_projects.kr_id
- UPDATE/INSERT 继续写旧表（trigger 自动同步到新表）

**PR3 实际迁移范围**（基于列兼容性分析）:
1. `kr-completion.js` — goals→objectives（仅 SELECT，UPDATE 留旧表）
2. `kr-progress.js` — goals→objectives for syncAllKrProgress
3. `entity-linker.js` — goals→UNION(visions+objectives+key_results)；projects→UNION(okr_*)
4. `routes/tasks.js` — request-intent-expand 链路 goals chain→objectives/visions 直接 FK
5. `routes/notion-sync.js` — 统计中增加新表数量

**不做什么（PR4+ 处理）**:
- planner.js、executor.js、focus.js、okr-tick.js 等（依赖 priority/progress/repo_path/SELECT *）
- 共 40+ 个 COMPLEX 文件推后

## 成功标准

1. `kr-completion.js` 中 checkKRCompletion 和 activateNextKRs 的 SELECT 改为查 objectives

4. `routes/tasks.js` request-intent-expand 改为用 objectives→visions FK 链路
