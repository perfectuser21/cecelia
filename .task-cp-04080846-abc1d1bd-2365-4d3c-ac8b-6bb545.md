# Task Card: CI 绿灯率提升 35%→60%（两个根因修复）

## 需求（What & Why）

**功能描述**: 修复导致 CI 绿灯率 KR5 停留在 33% 的两个根因 bug

**背景**: KR5「Engine CI 可信赖 — CI绿灯率≥95%」当前 current_value=33，实际 SQL 应返回 50（4 tasks 中 2 merged）。即使 50 达到了 Week 1 目标 60 的前提条件。

**根因分析**:
1. **kr-verifier.js `$3` 类型推断失败**: 向 `key_results` 写入进度时，`jsonb_build_object('metric_current', $3)` 中 `$3` 类型未知，PostgreSQL 报 `could not determine data type of parameter $3`，导致 KR 进度永远停留在旧值（33）
2. **shepherd.js 遗漏 pr_status=NULL 的任务**: shepherd 只扫描 `pr_status IN ('open', 'ci_pending')` 的任务，pr_url 已有但 pr_status=NULL 的任务（PR #1904、#1896 均已 merged）被跳过，未回写 'merged'，拉低了 merge rate

**不做什么**: 不修改 Deploy 流程（已自愈），不修改 KR verifier 查询逻辑本身

## 成功标准

> [BEHAVIOR] kr-verifier UPDATE key_results 不再报 $3 type 错误
> [BEHAVIOR] shepherd 扫描到 pr_status=NULL 的 PR 任务并更新状态

## 实现方案

**要改的文件**:
- `packages/brain/src/kr-verifier.js` — 第 74 行 UPDATE key_results 查询：删除 `$3` 参数，改用 `$2::text`，同时参数数组从 4 个减到 3 个
- `packages/brain/src/shepherd.js` — 第 127 行查询条件：`pr_status IN ('open', 'ci_pending')` → `(pr_status IN ('open', 'ci_pending') OR pr_status IS NULL)`

**受影响函数/API**:
- `runAllVerifiers()` in kr-verifier.js
- `runShepherd()` in shepherd.js

**不改什么**: 不改 KR verifier SQL 查询本身，不改 Deploy 流程，不改 Brain migrations
