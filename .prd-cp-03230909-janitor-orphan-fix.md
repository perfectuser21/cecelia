# PRD: Janitor 孤儿进程精准识别修复

## 背景

Mac mini 反复崩溃（Kernel Panic / AMCC 内存控制器过压）。
根因：多个 Claude Code agent 并发跑 `/dev` 任务时，如果 agent 异常退出，
其启动的 `npm test` / vitest 子进程变成孤儿进程，持续消耗内存导致机器崩溃。

Janitor 的 frequent 模式本来负责清理这类进程，但有两个问题：
1. **第一个循环（vitest）没有 PPID 检查**：只判断运行时间 ≥ 2 小时，会误杀正在运行中的合法测试
2. **阈值 7200s 太宽**：机器在 40 分钟内就会崩溃，2 小时阈值完全来不及
3. **janitor.sh 不在 git 管理**：只有 SKILL.md 在 repo，实际脚本在 ~/bin/，无法 code review

## 目标

1. 精准识别孤儿进程：追溯祖先 zsh 的 PPID，如果是 launchd(1) = 孤儿
2. 绝不误杀合法进程：祖先链中有活着的 claude 进程 = 合法，不动
3. 把 janitor.sh 纳入 git 管理：放入 packages/workflows/skills/janitor/
4. crontab 频率改为 15 分钟（原 30 分钟）
5. 孤儿阈值改为 10 分钟（原 2 小时）

## 核心判断逻辑

```
对每个 vitest/npm-test 进程：
  向上遍历父进程链 → 找到祖先 zsh/bash
  检查那个 zsh 的 PPID：
    = 1 (launchd) → Claude 已死，这是孤儿 → 等 10 分钟后 kill
    = 活着的 claude 进程 → 合法任务 → 绝对不动
```

## 不在范围内

- daily 模式磁盘清理逻辑不变
- Brain/DB 服务进程不触碰
- 非 cecelia 相关的 node 进程不触碰

## 成功标准

1. janitor.sh 存在于 `packages/workflows/skills/janitor/janitor.sh`（纳入 git）
2. `~/bin/janitor.sh` 是指向 repo 文件的软链接
3. frequent 模式：父进程是活着的 claude 的 vitest 进程不被 kill
4. frequent 模式：祖先 zsh PPID=1 且运行 > 10 分钟的 vitest 进程被 kill
5. crontab 频率为 `*/15`（15 分钟）
