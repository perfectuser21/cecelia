# PRD: feat PR 变更行覆盖率硬门禁

## 背景

AI-first 开发中，同一 AI 写 PRD→DoD→Test→Code，存在自我验证死循环。PR #841 加了 BEHAVIOR 弱测试检测，但无法阻止"写了代码没真测"的情况。

**核心问题**：AI 可以写一个函数，配一个 `grep -c functionName file.js` 的"测试"，CI 全绿但代码从未被执行过。

**解决思路**：用确定性工具（覆盖率）替代 AI 判断 AI。覆盖率工具不会被骗——代码行要么被执行过，要么没有。

## 方案

三个确定性 CI 硬门禁（零 LLM 成本）：

### 门禁 1：feat PR 必须有新增测试文件

commit message 以 `feat:` 开头的 PR，git diff 中必须有新增的 `.test.ts` 或 `.test.js` 文件。
`fix:`、`docs:`、`chore:` 等不受影响。

### 门禁 2：新测试必须 import 新代码

新增的测试文件必须 import 本 PR 新增/修改的源码文件（非测试文件）。
防止写了一个空测试或不相关的测试来敷衍门禁 1。

### 门禁 3：变更行覆盖率（核心）

本 PR 新增/修改的 `.ts`/`.js` 源码行（排除测试文件、配置文件），至少 60% 被测试覆盖。
使用 vitest `--coverage` 输出的 JSON 覆盖报告，对比 `git diff` 的变更行。

## 影响范围

- `packages/engine/scripts/devgate/` — 新增 `check-changed-coverage.cjs`（三个门禁合一脚本）
- `.github/workflows/engine-ci.yml` — L3 Code Gate 新增覆盖率检查步骤

## 成功标准

- feat: PR 无新增测试文件时 CI 拒绝
- 新测试不 import 新代码时 CI 拒绝
- 变更行覆盖率 < 60% 时 CI 拒绝
- fix:/docs:/chore: PR 不受影响（向后兼容）
- 纯配置文件变更（.yml, .json, .md）不计入覆盖率统计
