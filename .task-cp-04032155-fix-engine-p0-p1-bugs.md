# Task Card: Engine P0+P1 Bug 修复（4 个真实确认 Bug）

## 需求（What & Why）
**功能描述**: 修复 4 个经多 agent 审计并逐一代码验证确认的真实 bug
**背景**: 深度代码审计发现，其中1个 P0（CI 门禁失效）+ 3个 P1（误触发告警/日志污染/数据提取不全）
**不做什么**: 不修改未经代码验证的"可疑"bug（P0-1/P0-2/P1-3/P1-6 经核实均为假报告）

## 修复清单

### Bug 1（P0）: check-version-sync.sh — exit 0 掩盖 package.json 读取失败
- **文件**: `packages/engine/ci/scripts/check-version-sync.sh:38`
- **问题**: `BASE_VERSION` 为空时打印 warning 然后 `exit 0`（CI 判定通过），应 `exit 1` 失败
- **影响**: package.json 损坏/删除时版本同步 CI 门禁静默放行

### Bug 2（P1）: devloop-check.sh — date 解析失败时 _se=0 误触发 CI 卡死告警
- **文件**: `packages/engine/lib/devloop-check.sh:192-200`
- **问题**: `_se` fallback 为 `0`，`$(( $(date +%s) - 0 ))` 得到当前 unix 时间戳（~17亿），永远 > 5400，误报"CI 已 pending 90+ 分钟"
- **影响**: 任何 date 解析失败的环境都会无限误报 CI 卡死

### Bug 3（P1）: execution-logger.sh — `sed 's/%/%%/g'` 污染日志
- **文件**: `packages/engine/lib/execution-logger.sh:73`
- **问题**: `%` 在 sed 中无特殊含义，此替换把 `50%` 变成 `50%%`，破坏包含百分号的日志内容
- **影响**: jq 不可用的 fallback 路径下，所有含 `%` 的日志被污染

### Bug 4（P1）: fire-learnings-event.sh — awk 只提取最后一个 `###` 节
- **文件**: `packages/engine/skills/dev/scripts/fire-learnings-event.sh:72-83`
- **问题**: 第一个 awk 每遇到 `### ` 就重置 content，最终只保留最后一个 `###` 节的内容。若 `### 下次预防` 不是最后一节，第二个 awk 找不到它，返回 `[]`
- **影响**: Learning 文件中非最后位置的 `### 下次预防` 内容丢失，Brain 无法获取预防措施
