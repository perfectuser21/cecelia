# PRD: Provider-Agnostic Engine — 单一入口架构

## 背景

Cecelia Engine 的 /dev 完成判断逻辑（PR 创建？CI 通过？PR 合并？Learning 写完？）目前只存在于 `stop-dev.sh`（Claude Code Stop Hook）。当 Codex 等第二个 Provider 需要跑 /dev 时，就要在 runner.sh 里重复同一份逻辑——任何修改都要改两处，这违反 DRY 原则。

## 目标

实现真正的 Provider-Agnostic Engine：换 Provider = 只换 Provider 配置，不改任何业务逻辑。改任何底层逻辑只改一个地方。

## 成功标准

### 单一入口

- 所有完成判断逻辑提取到 `packages/engine/lib/devloop-check.sh`
- stop-dev.sh 改为只做 Claude Code JSON 协议适配（source devloop-check.sh + 读结果 + 输出 JSON）
- runners/codex/runner.sh 改为只做 Codex while-loop 适配（source devloop-check.sh + 读结果 + 驱动 codex exec）
- stop-dev.sh 和 runner.sh 在业务逻辑变化时永远不需要改动

### Brain 路由

- task-router.js 新增 `codex_dev` → `xian`
- executor.js 新增 codex_dev 派发（通过 triggerCodexBridge 以 runner 模式启动）

### 扩展性

- 添加第三个 Provider（Cursor、Gemini CLI）只需新增一个 runners/xxx/runner.sh，source devloop-check.sh 即可

## 不做的事

- 不修改 SKILL.md（业务流程定义不变）
- 不修改现有 hooks（branch-protect.sh 等）
- codex-bridge.cjs 的修改在 infrastructure repo 做，本 PR 只做 engine + brain 部分
