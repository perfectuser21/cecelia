# PRD: Janitor v4.0 — 全资源守护

## 背景

Janitor 是 Cecelia 的横向资源守护员，目标是保障调度器（slot allocator）能正常派发任务。
当前版本存在三个根本缺陷：
1. **crontab 路径错误**：指向 `/Users/administrator/bin/janitor.sh`（不存在），Janitor 从未运行过
2. **claude 孤儿进程无人管**：ppid=1 的孤儿 claude 进程占用 slot，导致 dispatchAllowed=false
3. **只管 vitest/jest**：frequent 模式不扫描 claude 进程，内存/CPU 压力也不响应

## 目标

重写 `packages/workflows/skills/janitor/janitor.sh` frequent 模式，实现：
1. 修复 crontab 路径（软链接到真实文件）
2. claude 孤儿检测：TTY=?? + ppid=1 + 运行>10分钟 + Brain DB 无 in_progress 任务 + 无 .dev-lock → kill
3. 内存压力响应：内存 >90% 时降低阈值到 5 分钟，加强扫描力度
4. CPU 高压上报：CPU >85% 持续触发时，向 Brain 创建告警任务

## 技术方案

### claude 孤儿判定（双重验证，保守原则）

```
条件1: TTY = ??（无终端附属）
条件2: 运行时间 > 600 秒（10 分钟），内存压力下降至 300 秒
条件3: ppid = 1（父进程是 launchd，已成孤儿）
条件4: Brain DB 无对应 in_progress 任务（curl 查询）
条件5: 所有 worktree 中无匹配 .dev-lock 文件

全部命中 → 孤儿 → kill
任一不满足 → 跳过（保守）
```

### 内存压力阈值

- 正常（<90%）：阈值 600 秒
- 高压（≥90%）：阈值 300 秒，扫描范围扩展

### CPU 高压上报

- CPU >85% → 向 Brain POST 告警任务（P1）

## 涉及文件

- `packages/workflows/skills/janitor/janitor.sh`（主要修改）
- crontab（修复路径）

## 成功标准

- crontab 每 15 分钟能正常执行 frequent 模式（验证软链接存在）
- ppid=1 + 无 Brain 任务 + 无 .dev-lock 的 claude 孤儿在超时后被 kill
- 有效的有头 claude 进程（TTY=s001等）绝对不被 kill
- 有效的无头 agent（有对应 in_progress 任务）绝对不被 kill
- 内存 >90% 时阈值降至 300 秒
