# Task Card: 任务失败率根因修复 — liveness_dead 容忍度提升

## 需求（What & Why）

**功能描述**: 分析近7天 dev 任务失败根因，对最高频失败类别实施单点修复。

**背景**: PRD 描述 39% 成功率危机（历史高峰于 4月6-7日），当前已自然回升至 91.7%（7天）。
根因分析结论：
- **87.5% 的 dev 任务失败 = `liveness_dead`**（watchdog OOM/进程消失 → 2次即隔离）
- `QUARANTINE_AFTER_KILLS=2` + `2min` 退避 → 系统内存压力未解时，重试同样被杀 → 过早隔离
- `dev-failure-classifier.js` 缺少 `liveness_dead`/`watchdog` 模式 → 走到 unknown（不重试）
- **PR#1714 `error_message` 缺失问题已彻底修复**（验证：近7天 0% 缺失）

**不做什么**:
- 不改 pipeline_rescue storm 防护（已有 MAX_RESCUE_PER_BRANCH=5 + 72h 冷却）
- 不改 dispatcher 拆分策略（需要更多 LLM 分析，超出单点修复范围）

## 根因库

| 类别 | 占比 | 状态 | 处理 |
|------|------|------|------|
| liveness_dead（OOM/进程消失） | 87.5% | **本次修复** | 增加容忍度+退避 |
| Crisis 压力击杀 | 25% | 部分重叠 | watchdog 已有保护 |
| error_message 缺失 | 0% | **已修复(PR#1714)** | 验证通过 |
| parent_pipeline_failed 级联取消 | <5% | 正常行为 | 无需修复 |

## 成功标准
- [ ] `executor.js` 的 `requeueTask` 对 `liveness_dead` 原因使用 3 次容忍（而非 2）
- [ ] `liveness_dead` 重试退避从 2min 增至 15min，让系统内存有时间恢复
- [ ] `dev-failure-classifier.js` 新增 `liveness_dead`/`watchdog` → TRANSIENT 模式
