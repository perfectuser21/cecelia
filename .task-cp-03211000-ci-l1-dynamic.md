# Task Card: CI L1 动态化 — 读 .dev-mode stage 按需检查

## 需求（What & Why）
**功能描述**: 在 ci-l1-process.yml 中添加 stage 检测逻辑，读取 PR 分支中的 `.dev-mode.*` 文件，根据当前 stage 决定哪些检查项需要执行。
**背景**: 现在 CI 不管 PR 在哪个 stage，所有 PR 都跑同样的检查。Stage 1（Spec 阶段）的 PR 不需要跑 DoD 验证和代码测试，但目前会全部失败，阻塞了开发流程。Pipeline 升级了但 CI 还是死的。
**不做什么**: 不修改 L2/L3/L4 CI 配置；不修改 .dev-mode 文件的格式定义。

## 成功标准
> [ARTIFACT] 产出物 / [BEHAVIOR] 运行时行为 / [GATE] 门禁

1. [ARTIFACT] ci-l1-process.yml 包含 detect-stage job，能读取 .dev-mode 文件并输出 current_stage
2. [BEHAVIOR] Stage 1 PR 只跑 Task Card 格式检查和分支命名检查，跳过 DoD 验证和代码测试
3. [BEHAVIOR] Stage 2+ PR 跑 DoD 验证 + 代码测试
4. [BEHAVIOR] 无 .dev-mode 文件的老 PR 跑全量检查（向后兼容）
5. [GATE] CI 全部通过

## 实现方案
**要改的文件**: `.github/workflows/ci-l1-process.yml`
**Scope 锚定**: 只改 CI workflow 文件，不改 engine/brain 代码

### 方案详情
1. 新增 `detect-stage` job（在 `changes` 之后），checkout 代码后读取 `.dev-mode.{branch}` 文件
2. 解析 step_1_spec ~ step_4_ship 中最高的 done step 作为 current_stage
3. 无文件时默认 stage=4（向后兼容）
4. 各检查 job 添加 `if` 条件：
   - Stage 1：只跑 verify-dev-workflow（分支命名）
   - Stage 2+：跑 dod-check, cleanup-check, check-prd, required-paths-check
   - Stage 3+：全量
   - Stage 4：额外跑 check-learning
5. l1-passed gate job 中对应调整 skipped 判断逻辑
