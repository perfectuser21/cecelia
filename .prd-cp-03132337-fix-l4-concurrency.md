# PRD：修复 L4 并发限制

## 背景

`ci-l4-runtime.yml` 配置了全局 `concurrency: group: l4-runtime-mac`，导致所有 PR 的 L4 只能串行运行。根因是 PostgreSQL 端口硬编码（5432/5433/5434），多个 L4 同时跑会端口冲突。Mac mini M1 有 16GB 内存、8 核，性能足以支撑 2 个 L4 并行（每个 L4 = 3 shard，各占 ~5GB 内存）。

## 目标

1. 去掉全局 concurrency mutex，允许多个 L4 同时运行
2. 动态分配 PostgreSQL 端口和 pgdata 目录（基于 `GITHUB_RUN_ID`）
3. Mac mini M1 从 4 个 runner 扩展到 6 个（支撑 2 个并发 L4）

## 成功标准

- L4 workflow 不再有全局 `concurrency: group: l4-runtime-mac`
- 每个 workflow run 使用独立的 PostgreSQL 端口（不与其他 run 冲突）
- 每个 shard 使用独立的 pgdata 目录（含 run_id 隔离）
- xian-mac-m1 注册 6 个 runner（runner-1 ~ runner-6）
- 2 个 L4 可以同时跑不相互干扰

## 技术方案

### 动态端口计算

```bash
# 在 Set up PostgreSQL step 中
BASE_PORT=$(( 5500 + (GITHUB_RUN_ID % 100) * 3 ))
SHARD_NUM=$(echo "${{ matrix.shard }}" | cut -d/ -f1)
DB_PORT=$(( BASE_PORT + SHARD_NUM - 1 ))
PG_DATA="/tmp/pgdata-${GITHUB_RUN_ID}-${SHARD_NUM}"
```

端口范围：5500~5797，避开 5432（Brain 生产 DB）。100 个 run_id slot × 3 端口 = 足够并发。

### 新增 runner

```bash
ssh xian-m1
cd ~/runners
# 按 runner-1~4 模式安装 runner-5、runner-6
```
