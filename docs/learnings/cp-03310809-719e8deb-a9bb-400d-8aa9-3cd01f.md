# Learning: executeCopywriting 降级策略 + findings 过滤顺序

**Branch**: cp-03310809-719e8deb-a9bb-400d-8aa9-3cd01f
**Date**: 2026-03-31

### 根本原因

1. **结构性错误：hard-fail 在 LLM 调用之前**。原始代码中 `if (top.length === 0) { return fail }` 位于 LLM 调用块之前，导致所有 typeConfig 有 `generate_prompt` 的路径（只要 findings 为空）都提前失败，LLM 从未被调用。
2. **降级策略缺失**。旧版占位符数据的 `brand_relevance: 2` 被 `>= 3` 过滤全部剔除，没有降级路径，导致"亲爱的安先生"等关键词连续 4 次失败。
3. **测试 mock 不完整**。`_loadFindings` 先 `existsSync(rDir)` 检测研究目录是否存在，mock 只覆盖了 `findings.json` 路径，研究目录路径返回 `false` → 循环跳过 → 没有 findings。

### 下次预防

- [ ] **LLM 调用路径不依赖 findings**：findings 是可选上下文，不是 LLM 调用的前提条件。hard-fail 只适用于"既无 typeConfig 又无 findings"的静态模板路径
- [ ] **写 existsSync mock 时**：检查目标函数内部是否先做目录存在检测（`existsSync(dir)`），需要同时 mock 目录路径和文件路径
- [ ] **降级策略模板**：`brand_relevance` 过滤后为空 → 降级用全部 → 让下游阶段（copy-review）做质量判断，不在 copywriting 阶段硬失败
- [ ] **测试文件设计**：test 1-4 全部 typeConfig + generate_prompt，测试预期 LLM 被调用 → 必须确保代码结构中 LLM 块在任何 hard-fail 之前
