# Learning: OKR 数据迁移 migration 179

**Branch**: cp-03231916-1b2e962d-dbf6-400e-880c-51748c
**PR**: #1456
**Date**: 2026-03-23

### 根本原因

1. **migration 177 的语义漂移**：migration 177 把 `goals.type='area_okr'` 迁移到了 `key_results`，而新 PRD 要求 `area_okr` → `objectives`。两次迁移逻辑不一致导致数据分布在错误的表中（7条 area_okr 进了 key_results，应在 objectives）。本 migration 不修复此错误（留给后续 PR），但需在 SQL 注释中明确说明。

2. **FK nullable 链断裂**：旧 `projects.goal_id` 指向 `goals.type='area_okr'`，而 area_okr 现在迁移到 `objectives` 而非 `key_results`，导致 `okr_projects.kr_id` 无法满足 FK 约束（`kr_id → key_results.id`）。解决方案：`kr_id = NULL`（FK nullable），不强行建立断裂的引用。

3. **DoD Test 格式：`[ ]` 混入成功标准**：Task Card 的 `## 成功标准` 使用 `- [ ]` 格式，被 `check-dod-mapping.cjs` 误识别为未验证的 DoD 条目。修复：改用有序列表（`1.`/`2.`/...）。

4. **migration 文件写入路径误差**：首次用 Write 工具写入时使用了主仓库绝对路径（`/cecelia/packages/brain/migrations/`），未写入 worktree 路径。需手动 cp 到 worktree 后才被 git 追踪。

### 下次预防

- [ ] 写 migration 文件时直接用 worktree 相对路径，不用绝对路径
- [ ] Task Card `## 成功标准` 始终用有序列表（`1.`/`2.`），不用 `- [ ]` checkbox
- [ ] 迁移脚本涉及跨表 FK 时，先查目标表 FK 的 nullable 属性，再决定是否置 NULL
- [ ] 继承旧 migration 时，在注释中明确标注"已知问题：migration N 的 XXX 误迁，本 migration 不修复，见 PR M"
