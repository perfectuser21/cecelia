# Learning: OKR 业务代码迁移 PR8 — 17个 HARD 文件 SELECT 迁移

## 任务概述

将 Brain 中 17 个源文件的 SELECT 查询从旧表（goals/projects）迁移到新 OKR 表（key_results/objectives/visions/okr_projects/okr_scopes/okr_initiatives）。

---

### 根本原因

旧 goals/projects 表承载了所有层级数据（vision/objective/kr/project/scope/initiative），靠 `type` 字段区分。新 OKR 七表结构将每种类型拆分到独立表，读路径必须跟上，否则 SELECT 仍走旧表导致后续读写不一致。

---

### 下次预防

- [ ] 列出所有"只在旧表存在"的列（notion_id, repo_path, lead_agent, sequence_order, time_budget_days, execution_mode, domain, current_phase）——凡 SELECT 涉及这些列，保留旧表并加注释，不要强行迁移
- [ ] `goals.type='area_okr'` → `objectives`；`type='area_kr'/'global_kr'` → `key_results`；`type='vision'` → `visions`
- [ ] `projects.type='project'` → `okr_projects`；`type='scope'` → `okr_scopes`；`type='initiative'` → `okr_initiatives`
- [ ] 跨层级查询（initiative via scope via project）需要两次 JOIN：`okr_initiatives JOIN okr_scopes ON os.id=oi.scope_id WHERE os.project_id=$1`
- [ ] 列别名保持 API 兼容：`title AS name`、`vision_id AS parent_id`、`objective_id AS parent_id`
- [ ] Notion 同步文件（notion-full-sync.js）全部 SELECT/INSERT/UPDATE 均需保留旧表：notion_id 列只在旧表，且 INSERT/UPDATE 由触发器自动同步到新表
- [ ] UPDATE/INSERT 永远写旧表，触发器负责同步到新 OKR 表，迁移只改 SELECT
- [ ] `goals UNION ALL` 模式：需要横跨 objectives + key_results 时用 `SELECT ... FROM objectives UNION ALL SELECT ... FROM key_results`

---
