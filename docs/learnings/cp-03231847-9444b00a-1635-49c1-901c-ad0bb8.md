# Learning: OKR 业务代码迁移 PR8 — 17个 HARD 文件 SELECT 迁移

## 任务概述

将 Brain 中 17 个源文件的 SELECT 查询从旧表（goals/projects）迁移到新 OKR 表（key_results/objectives/visions/okr_projects/okr_scopes/okr_initiatives）。

---

### 根本原因

旧 goals/projects 表承载了所有层级数据（vision/objective/kr/project/scope/initiative），靠 `type` 字段区分。新 OKR 七表结构将每种类型拆分到独立表，读路径必须跟上，否则 SELECT 仍走旧表导致后续读写不一致。

迁移规则不对称：UPDATE/INSERT 留旧表由触发器同步到新表，只有 SELECT 需要改写。这让"哪些查询要改、哪些不改"的边界判断变得关键——旧表专属列（notion_id/repo_path/lead_agent/sequence_order/time_budget_days）是留守旧表的唯一依据。

跨层级联查（initiative←scope←project）在新表结构下需要两次显式 JOIN，而旧表靠 parent_id 递归完成，这是迁移中最易出错的模式转换点。

---

### 下次预防

- [ ] `goals.type='area_okr'` → `objectives`；`type='area_kr'/'global_kr'` → `key_results`；`type='vision'` → `visions`

- [ ] 列别名保持 API 兼容：`title AS name`、`vision_id AS parent_id`、`objective_id AS parent_id`
- [ ] Notion 同步文件（notion-full-sync.js）全部 SELECT/INSERT/UPDATE 均需保留旧表：notion_id 列只在旧表，且 INSERT/UPDATE 由触发器自动同步到新表
- [ ] UPDATE/INSERT 永远写旧表，触发器负责同步到新 OKR 表，迁移只改 SELECT
- [ ] `goals UNION ALL` 模式：需要横跨 objectives + key_results 时用 `SELECT ... FROM objectives UNION ALL SELECT ... FROM key_results`

## CI 修复补丁（L4 Runtime Gate）

### 根本原因（补充）

三类 CI 失败，均源于"迁移后表结构与旧表不同"：

1. **测试 mock 模式与业务代码不同步**（initiative-closer.test.js / initiative-queue.test.js）
   - mock 仍匹配旧表名（`UPDATE projects`、`LEFT JOIN goals`），业务代码已改用新表
   - mock 匹配失败 → 返回 `{ rows: [] }` → 断言失败

2. **`objectives` 表缺 `progress` 列 → GoldenPath Fatal**（focus.js）
   - migration 182 只给 `objectives` 加了 `description` + `priority`，未加 `progress`
   - `getReadyKRs()` SELECT 了 `objectives.progress` → PG 报错
   - `getDailyFocus()` 在 tick.js 中无 try-catch → 错误冒泡 → tick 返回 `success: false`
   - 修复：`SELECT ... 0 AS progress ... FROM objectives`

3. **Q3 优先级测试断言过时**（initiative-queue.test.js）
   - 旧断言：`expect(updateCall).toContain('LEFT JOIN goals')`
   - 迁移后 `activateNextInitiatives` 直接用 `okr_initiatives.priority` 列排序，不再 JOIN goals
   - 修复：改为 `expect(updateCall).toContain('CASE priority')`

### 下次预防（补充）

- [ ] **迁移源文件同时更新对应测试文件**：mock SQL 匹配模式必须与业务代码同步，不能只改业务代码
- [ ] **迁移前对照 migration SQL 逐列确认**：`objectives` 无 `progress`，`key_results` 有；不要假设旧表有的新表也有
- [ ] **tick 调用链检查**：改动 tick 路径上的函数时，确认整条链路无未处理的 DB 错误（`getDailyFocus` → `getReadyKRs` 无 try-catch）
- [ ] **列映射速查**：`objectives` 列 = `description, priority`（无 progress）；`key_results` 列 = `description, priority, progress, weight`
