# Learning: Pipeline Security Bug Fix（4个 P0/P1 安全漏洞）

**分支**: cp-03292331-b18a39d9-5a76-4fc0-942c-677b06
**日期**: 2026-03-30

## 任务概述

修复 /dev 工作流中4个安全漏洞：seal 内容校验、fallback 降级、flock 硬错误、PR 合并前二次确认。

---

### 根本原因

**Bug 1（P0）devloop-check.sh seal 只检查存在性**：
条件 1.5/2.5 仅对 `verdict==FAIL` 做拦截，当 seal 文件存在但 verdict 字段为空（`jq` 输出空字符串）时，代码既不命中 FAIL 分支也不命中 PASS 分支，直接 pass-through 放行。攻击者可以写一个只有空 JSON `{}` 的 seal 文件绕过 Gate。

**Bug 2（P0）stop-dev.sh fallback 路径降级**：
当 devloop-check.sh 未加载时，代码进入旧内联逻辑（200+ 行）。该路径没有 seal 文件验证、divergence_count 门禁等新安全机制，等于彻底绕过了所有 v15.x 新增的安全检查。

**Bug 3（P1）lock-utils.sh flock 静默返回 0**：
`flock` 不可用时（macOS 无 coreutils），`acquire_dev_mode_lock` 直接 return 0，调用方误以为锁获取成功，实际上是无锁状态，多会话并发时可能导致 .dev-mode 文件损坏。

**Bug 4（P1）PR 合并前无二次确认**：
`gh pr merge` 执行前仅依赖之前检查的 CI 状态（可能是几分钟前的快照），若 CI 在此期间失败或出现新冲突，合并仍会执行。

---

### 下次预防

- [ ] **seal 内容校验三步法**：每次写 seal 检查逻辑时，必须覆盖三种情况：(1) 字段缺失/空值、(2) 无效值（非 PASS/FAIL）、(3) 明确 FAIL 值。只检查 FAIL 不够。
- [ ] **fallback 路径安全原则**：安全机制的 fallback 不能是降级执行，必须是 fail-closed（exit 2 + 错误信息）。有 fallback = 绕过 = 安全漏洞。
- [ ] **flock 依赖声明**：使用 flock 的模块必须在函数头注释声明依赖，不可用时 return 1 而非静默 skip。
- [ ] **merge 操作前必须二次确认**：任何 `gh pr merge` 调用前都要重新查一次 mergeable + CI 状态，不能信任缓存快照。
- [ ] **DoD Test 写法**：检查"防御代码逻辑存在"比检查"注释文字存在"强——应检查 `[[ -z "$var" ]]` 这样的条件判断代码，而不是只检查中文注释字符串。
