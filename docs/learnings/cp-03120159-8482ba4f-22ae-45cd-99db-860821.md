# Learning: liveness DEAD 改 requeue

## 背景

executor.js probeTaskLiveness 在双次确认进程 DEAD 后，原来直接调用 `updateTaskStatus(failed)`，导致任务永久失败。进程死亡通常是 OOM 或系统抢占（非代码 bug），应该重新入队。

## 关键决策

- **改调 requeueTask('liveness_dead', errorDetails)**：复用已有的 requeue + FAILURE_THRESHOLD + quarantine 逻辑，不新增分支
- **evidence 传 errorDetails**：保留 liveness diagnostic_info（pid/reason/probe_ticks）写入 payload.watchdog_kill
- **action 字段区分**：requeue 成功 → `liveness_auto_requeue`；quarantined → `liveness_auto_fail`

## 测试 Mock 顺序

requeueTask 内部按顺序发 4 个 SQL 查询：
1. `SELECT payload, task_type, project_id, title FROM tasks WHERE id = $1 AND status = 'in_progress'`
2. `UPDATE tasks SET status = 'queued', started_at = NULL, payload = ...`
3. `SELECT id FROM learnings WHERE content_hash = $1 AND is_latest = true`
4. `INSERT INTO learnings (...)`

测试 mock 必须按此顺序准备 4 个 `mockResolvedValueOnce`，否则查询数量不匹配导致测试抖动。

## 注意事项

- requeueTask 使用 `watchdog_retry_count` 追踪重试次数（`QUARANTINE_AFTER_KILLS=2`）
- liveness 路径和 watchdog 路径共用同一个计数器：两者都通过 requeueTask 累计 failure_count，防止交替失败绕过隔离
