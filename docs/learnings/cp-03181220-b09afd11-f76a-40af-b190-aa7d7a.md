# Learning: Engine 状态机三层防御 P0+P1

Branch: cp-03181220-b09afd11-f76a-40af-b190-aa7d7a
Date: 2026-03-18

### 根本原因

AI 可以通过 Bash 工具（`echo "step_1_taskcard: done" >> .dev-mode.branch`）绕过 PreToolUse:Write 拦截，
直接向 `.dev-mode.*` 写入 `step_N: done`，而不触发 `verify-step.sh` 验证。
原本的防御只在 Write/Edit 工具层（`branch-protect.sh`），Bash 写操作不受约束，形成旁路。

### 下次预防

- [ ] 任何新的步骤完成标记机制，必须同时覆盖 Write/Edit 层和 Bash 层
- [ ] seal 机制（`step_N_seal: verified@timestamp`）应作为所有步骤验证的标准凭证格式，防止 AI 回滚或绕过
- [ ] Stop Hook 层（`devloop-check.sh`）需对每层防御做兜底检查：有 done 无 seal → blocked
- [ ] 新增防御规则时，测试必须使用隔离环境（`runHookIsolated`），避免真实 monorepo 环境中的 task card 使测试通过
- [ ] 三层防御模式：工具层拦截 → 验证层签证 → 终止层兜底，这是封堵状态机旁路的标准架构
