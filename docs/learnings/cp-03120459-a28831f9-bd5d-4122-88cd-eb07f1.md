## Learning: 懒加载 project_id 解决测试断言与实现不匹配

### 问题

初始实现在 `enqueueActions()` 函数开头就调用 `getDefaultInitiativeId()`（预加载 fallback project_id），导致：
- 即使所有 action 都无效（无 title），也会触发 DB 查询
- 测试用例 "无 title 的动作被跳过" 断言 `mockPool.query` 未被调用，但实际调用了 initiative 查询 → 测试失败

### 解决方案

改为懒加载：将 `getDefaultInitiativeId()` 的调用推迟到首次实际需要 `project_id` 时（即去重通过、准备 INSERT 时），并缓存结果（`fallbackResolved` 标志 + `resolvedFallbackProjectId` 变量）。

```js
async function getFallbackProjectId() {
  if (fallbackResolved) return resolvedFallbackProjectId;
  fallbackResolved = true;
  resolvedFallbackProjectId = contextProjectId || await getDefaultInitiativeId();
  return resolvedFallbackProjectId;
}
```

### 副作用

懒加载改变了 DB 调用顺序，测试中的 `mockResolvedValueOnce` 顺序需要对应调整：
- 旧顺序：initiative 查询 → 去重查询 → INSERT
- 新顺序：去重查询 → initiative 查询（首次需要时）→ INSERT

### 规则

**凡是可能被跳过的路径前的 DB 查询，都应该懒加载。** 预加载在大批量全跳过场景下也会产生不必要的 DB 开销。
