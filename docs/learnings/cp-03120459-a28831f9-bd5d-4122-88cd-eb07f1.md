### [2026-03-12] Learning action-pipeline：懒加载 project_id 与 mock 顺序陷阱

**失败统计**：CI 失败 2 次（Learning 格式错误 1 次 + 实现问题 1 次），本地测试失败 1 次

#### 根本原因

1. **预加载 vs 懒加载陷阱**：初始实现在 `enqueueActions()` 函数开头预加载 `getDefaultInitiativeId()`，导致即使所有 action 都无效（无 title），也会触发 DB 查询，破坏"无 title 时不调用 DB"的测试断言
2. **mock 顺序依赖实现细节**：`mockResolvedValueOnce` 的顺序必须与代码实际调用顺序完全一致——实现改变后测试 mock 顺序要同步更新
3. **Learning 格式错误**：首次提交使用了自定义标题和粗体，CI 的 `grep -qE` 无法匹配必须章节名

#### 修复方式

- 将 `getDefaultInitiativeId()` 改为懒加载（闭包缓存），仅在首次需要 project_id 时才查询
- 同步调整所有测试用例的 mock 调用顺序：去重查询 → initiative 查询 → INSERT
- Learning 格式改为标准 `####` 章节标题 + `- [ ]` checklist

#### 下次预防

- [ ] 新模块中任何 DB 查询都问自己：这个查询是否可能被跳过？是 → 懒加载
- [ ] 测试写 mock 时，按实际代码执行路径从上到下排列 `mockResolvedValueOnce`，不要按"逻辑分组"排列
- [ ] Learning 文件必须包含 `#### 根本原因`、`#### 下次预防` 章节和 `- [ ]` checklist，用 `####` 标题不能用粗体
