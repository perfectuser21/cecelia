# Learning: pipeline_rescue 任务风暴导致成功率虚低

## 根本原因

`pipeline-patrol.js` 的 dedup 检查只阻止 **活跃状态** 任务在 2h 内重复创建，但 `cancelDuplicateTasks`（healing.js）会取消多余任务。取消后 dedup 窗口重置，2h 后 patrol 重新为同一分支创建 rescue 任务，形成死循环：

```
创建(queued) → cancelDuplicateTasks 取消 → 2h 后 dedup 过期 → 重新创建 → 循环
```

单个分支产生了 26-27 个 pipeline_rescue 任务，24h 累计 320+ 个 canceled 任务。

`getTaskStats24h()` 把 canceled 任务也计入 total 分母，导致 630/1088 = 57% 的虚低成功率。

## 修复

1. `pipeline-patrol.js`：dedup 冷却时间 2h → 24h，且扩展 SQL 条件以覆盖 24h 内 canceled/completed 状态的旧任务，彻底断开循环
2. `self-drive.js`：`getTaskStats24h()` total 改为 `started_at IS NOT NULL`（只计入实际派发执行的任务），failed 含 quarantined

## 预防

- [ ] pipeline_rescue 任务的 dedup 冷却必须 ≥ cancelDuplicateTasks 扫描周期（实际需 24h）
- [ ] 成功率指标不能包含「从未派发的 canceled 任务」，分母应用 `started_at IS NOT NULL`
- [ ] 出现大量同 task_type canceled 任务时，优先排查 dedup 逻辑是否失效
