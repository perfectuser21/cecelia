# Learning: OKR PR10 — 旧表迁移收尾

## 根本原因

PR1-PR9 迁移新 OKR 表时，为保证数据一致性采用了"先查新表，找不到 fallback 旧表"的渐进策略。PR10 任务是在数据迁移完成（migration 179）、触发器双写（migration 181）就位后，移除这些 fallback，完成最终清理。

主要 fallback 模式：
1. `if (result.rowCount === 0) { UPDATE goals/projects... }` — 试新表失败则写旧表
2. `else { INSERT INTO goals... }` — 未知类型 fallback 到旧表

## 下次预防

- [ ] 新表迁移时，fallback 代码应带 `TODO: remove after migration 1xx` 注释，明确标注清理时机
- [ ] notion-full-sync.js 的旧表依赖是特例（无 notion_id 列），已在代码注释中说明，不需要迁移
- [ ] planner.js:1194 的 `repo_path` 约束仅存在于旧 `projects` 表，是有效的 hard constraint，不能随迁移删除
- [ ] 旧表依赖扫描：`grep -rn "FROM goals\|FROM projects\|INTO goals\|INTO projects" packages/brain/src/` 可快速发现剩余引用
