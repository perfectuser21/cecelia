# Learning: 圈复杂度降低 — gatherFailureContext 重构

### 根本原因

`gatherFailureContext` 复杂度 25 的真正来源不是 7 个 if/catch，而是 **约 17 个 `||` 操作符**（每个 `||` 都是一个分支）。上下文对象初始化 `{ a: x || 'default', ... }` 贡献了大半复杂度。

### 下次预防

- [ ] 函数初始化多个字段时，用 `??`（nullish coalescing）而非 `||` 可以减少语义混淆，但两者对复杂度贡献相同
- [ ] 降低复杂度的最有效方式：**将多步 DB 查询提取为独立子函数**，每个子函数只负责一个查询，早返回（early return）消除嵌套 if
- [ ] vitest OOM 失败（`ERR_WORKER_OUT_OF_MEMORY`）是现有基础设施问题，不影响特定模块测试验证 — 直接运行 `npx vitest run <specific-test-file>` 验证目标模块
