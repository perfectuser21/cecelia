# Learning: CI 审计发现的 5 个问题验证

**任务**：fix(ci): 修复 CI 审计发现的 5 个问题（coverage-delta/mts/CONFIG豁免/release豁免/changes失败防御）
**结果**：所有 5 个问题均已在 main 分支中修复，本次任务为验证确认

---

### 根本原因

Brain 生成本任务时，5 个 CI 问题已通过之前的 PR 被修复合并进 main。Brain 的任务调度基于旧版审计结果，导致重复调度同一修复任务。

具体修复来源：
1. **coverage-delta brain-unit 依赖问题** — ci-l3-code.yml coverage-delta 的 `needs` 不含 brain-unit，`if:` 条件已仅依赖 should_run_l3 + commit_type + brain
2. **.mts 测试文件支持** — TEST_PATTERNS 已含 `\.test\.mts$|\.spec\.mts$`
3. **[CONFIG] 豁免路径精确化** — 已从 `^packages/engine/` 收窄为 `^packages/engine/(hooks|skills|scripts|config|contracts|...)`
4. **release: 豁免缺失** — ci-l2-consistency.yml engine-l2 version check 的 if 已含 `!startsWith(..., 'release:')`
5. **changes job 失败防御** — ci-l1-process.yml l1-passed gate 已先验证 `changes.result == success` 再读 engine output

---

### 下次预防

- [ ] Brain 在调度 CI 审计修复任务前，应先检查对应修复 PR 是否已合并
- [ ] 审计报告应标记"已修复"状态，避免重复生成同类任务
- [ ] 任务执行代理收到修复任务时，应先快速验证问题是否仍然存在，若已修复则直接关闭任务
