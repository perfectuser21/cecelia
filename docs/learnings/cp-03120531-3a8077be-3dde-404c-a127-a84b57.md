# Learning: learning-quality-scorer.js 实现经验

## 根因分析

learning 写入时 quality_score 字段长期为 NULL（migration 056 在 2026-02-22 加了字段但从未填充），导致知识过滤器无法按质量筛选。

## 关键决策

### 1. source_type 存储在 metadata 而非新列

PRD 要求填充 `quality_score + source_type`，但 learnings 表没有 `source_type` 列（只有 `learning_type` 是另一套枚举）。解决方案：将 `source_type` 存入 `metadata` JSONB 的 `source_type` 键，无需 schema 变更，满足"不修改 DB schema"约束。

### 2. 空壳词仅在短文本中扣分

Shell words（test/ok/pass/done 等）在长文本（>= 120 字）中可能只是自然提及，不代表内容空洞。因此规则：`shellMatches > 0 && text.length < 120` 才扣 30 分。

### 3. 测试用例内容设计要精确计算得分

测试 `source_type = 'standard'` 时，需要精确控制内容的扣分路径才能落到 60-79 区间。比如"含根因+含改进+含 TODO 模板"才能得 65 分（standard）而非 rich。

初始用 `'The crash happened because of a race condition. We should add a mutex lock.'` 测 standard，实际得 100（rich）——因为长度 >= 50、有根因（because）、有改进（should）、无模板，导致零扣分。需换成含 TODO 模板的内容。

### 4. 多写入路径的集成策略

learnings 表有 8+ 个 INSERT 位置（learning.js、cortex.js、auto-learning.js、thalamus.js 等）。本 PR 只集成主路径（`recordLearning` 和 `extractConversationLearning`），其他路径在后续 PR 处理。避免一次改动过多，降低风险。

## 改进建议

- 后续 PR 可将 `scoreLearning` 集成到 `cortex.js` 的 learning INSERT 路径
- 可考虑将 `source_type` 正式添加为 learnings 表的列（新 migration），便于 SQL 查询过滤
- 阈值（20 字、120 字等）可以提取为常量配置，便于调整
