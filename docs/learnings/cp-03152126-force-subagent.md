# Learning: /dev Subagent 强制执行（2026-03-15）

## 陷阱 1：独立裁判的调用权在被裁判者手里（2026-03-15）

### 根本原因

步骤文件写了「简单任务可跳过 Explore subagent」和「流程顺畅可跳过 Learning」，把是否调用 Subagent 的决定权给了主 agent。主 agent 有确认偏差——自己写的代码自己觉得对，自然倾向于认为「这次任务简单，不需要 Verifier」。这等于运动员决定比赛要不要裁判。

### 下次预防

- [ ] 步骤文件里永远不写「可跳过」——任何豁免条款都是潜在的逃跑路线
- [ ] Subagent 的调用规则只有「必须」，差别是数量（1/2/3个），没有 0 个的情况
- [ ] Learning 永远是强制的，哪怕写「这次很顺，因为做了 XYZ 所以没踩坑」也算

## 陷阱 2：「默认走 subagent」不等于「必须走 subagent」（2026-03-15）

### 根本原因

原文写「默认走 subagent 模式，只有明确是简单任务才跳过」。这句话逻辑上仍然保留了跳过的可能性，只是把门槛设高了一点。但只要有跳过的可能，主 agent 就会在某次判断"这就是简单任务"然后跳过。今天 PR #966 就是这样发生的。

### 下次预防

- [ ] 规则必须是绝对的「所有任务都必须」，不能是「默认必须，除非...」
- [ ] 写步骤文件时，检查是否存在任何「除非」「如果...则跳过」的条款，有则删除

## 架构结论

**独立性的前提是调用权与被裁判对象分离。**

| 设计 | 问题 |
|------|------|
| 主 agent 决定是否请 Verifier | ❌ 裁判权在被裁判者手里 |
| 步骤文件强制「每条 DoD 必须召唤 Verifier」 | ✅ 规则在系统里，不在 agent 的判断里 |
