# Learning: 批量修复不幂等 migration

**分支**: cp-03171956-2747b457-cba5-455c-b531-0080c2
**日期**: 2026-03-17

---

### 根本原因

CI 使用 xian-mac-m1 self-hosted runner，pgdata 目录按 `pgdata-${GITHUB_RUN_ID}-${SHARD_NUM}` 命名，同一 run 内重试时会复用已有目录（`skipping initdb`）。若某次运行中 postgres 进程被意外终止（OOM、超时），事务回滚后 schema_version 未记录该 migration，下次重跑时 migrate.js 会再次执行该 SQL，若 SQL 无幂等保护则报 "already exists"。

修复了 9 个 migration 的以下问题：
- **013**: `CREATE TABLE` 缺少 `IF NOT EXISTS`
- **052**: `INSERT INTO schema_version` 缺少 `ON CONFLICT DO NOTHING`
- **078/112/119/126/127/140**: 完全缺少 `INSERT INTO schema_version`（依赖 runner 外部插入，不够明确）
- **114**: `INSERT INTO schema_version` 中 version 值错误（'113' → '114'）

### 下次预防

- [ ] 新 migration 审查清单：
  1. `CREATE TABLE` → 必须用 `IF NOT EXISTS`
  2. `CREATE INDEX` → 必须用 `IF NOT EXISTS`
  3. `ADD CONSTRAINT` → 前面必须有 `DROP CONSTRAINT IF EXISTS`
  4. `INSERT INTO schema_version` → 必须有 `ON CONFLICT (version) DO NOTHING`
  5. version 值必须与文件名前缀一致（如文件 `114_xxx.sql` 对应 `'114'`）
- [ ] 已有类似修复先例：023（CREATE INDEX IF NOT EXISTS）、088（DROP CONSTRAINT IF EXISTS）、013+112（#1053）、本 PR（9 个）
