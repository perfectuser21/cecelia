### 根本原因

executor.js 中 `getDmesgInfo` 是通过 `platformGetDmesgInfo as` 别名从 platform-utils.js 导入，
再包装在局部函数 `getDmesgInfo()` 中。测试中 mock `platform-utils.js` 的 `getDmesgInfo` 时，
binding 通过 ESM live binding 传递给 executor.js，mockReturnValueOnce 可以正确影响 checkExitReason。

并发幂等保护依赖 SQL `WHERE status = 'in_progress'` 条件：第一次调用后任务已 queued，
第二次 SELECT 自然返回空，不会重复 requeue。此行为已有单元测试验证。
