# Learning: OKR体系统一PR2 — tick/task-router切换到新OKR表

Date: 2026-03-23

## 变更概要

将 Brain 核心调度（tick.js、task-router.js）从旧表（goals/projects/project_kr_links）切换到新 OKR 表（key_results/okr_projects/okr_scopes/okr_initiatives）。

### 根本原因

PR1 完成了数据迁移但未更新调度代码，导致 Brain tick 循环继续查询旧 goals 表，而 okr-closer.js 已经写新表，造成数据不一致。

### 关键发现

1. **新旧表 UUID 相同**：迁移时保留了原始 UUID，`tasks.goal_id` FK 无需修改，向后兼容
2. **key_results 状态值不同**：旧 goals 用 'ready'，新 key_results 用 'active'；fallback 排除用 'archived'（非旧的 'canceled'）
3. **okr_projects 列名不同**：`title`（非 `name`），查询时需要 `title AS name` 别名
4. **diagnoseKR initiatives 需两表 JOIN**：`okr_initiatives JOIN okr_scopes` 中转，不能直接查 `parent_id`
5. **key_results 无 priority/progress**：返回时需 `?? null` 兜底防止 undefined 传出

### DoD 测试设计教训

**[BEHAVIOR]/[GATE] 测试不能依赖本地基础设施（DB/Brain服务）**，CI 环境没有 PostgreSQL 也没有运行中的 Brain。应改为：
- `psql` 查询 → 改用 `node -e` 检查代码内容（ARTIFACT 风格）
- `curl localhost:5221` 健康检查 → 改用 `node --check` 语法校验

### 下次预防

- [ ] 写 DoD 时，`[BEHAVIOR]` 和 `[GATE]` 测试默认考虑 CI 环境限制——无本地 DB、无本地服务
- [ ] `psql` 命令只用于纯本地验证，不放 DoD Test 字段
- [ ] `curl localhost:PORT` 健康检查改为 `node --check` 语法验证 + L4 smoke check
- [ ] 切换 SQL 表时，先确认新表的状态值（status enum）与旧表的过滤条件是否对应
- [ ] JOIN 层级变化（`parent_id` → 中间表）需要在 Spec 阶段画出查询路径图再写代码
