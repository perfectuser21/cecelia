# Learning: rescue 风暴根因 — .dev-mode 文件被 git 追踪导致 worktree 继承

**分支**: cp-04080115-aabe0ea2-35c6-48fb-b1fe-7b23c7
**日期**: 2026-04-08

## 根本原因

`/dev` Stage 1 中 `git add .dev-mode.${BRANCH} && git commit` 使用了显式 `git add`，绕过了 `.gitignore` 中的 `.dev-mode*` 规则。这导致 .dev-mode 文件被提交到 feature 分支，并通过 PR 合并到 main。此后所有从 main 创建的新 worktree 都继承了这些 stale 文件。

pipeline-patrol 扫描所有 worktree，对每个 stale .dev-mode 文件都误判为 stuck pipeline 并创建 rescue 任务。同一分支的 stale 文件在 10 个 worktree 中重复出现，每个 tick 可能产生 10 个 rescue 任务。

`writeCleanupDone` 只写入单个 worktree 路径，导致其他 worktree 中同名文件未被标记，继续触发 rescue 循环。

## 下次预防

- [ ] /dev Stage 1 不得 `git add .dev-mode*`（.gitignore 中已有规则，无需显式绕过追踪）
- [ ] `writeCleanupDone` 已修复：扫描所有 worktree 路径，对所有出现的同名文件一次性标记
- [ ] 发现 rescue 任务激增（>10/小时同一分支）时，优先检查是否有 stale .dev-mode 在 git 追踪中
- [ ] PR 合并前检查 diff 中是否包含 .dev-mode* 文件（应该从不出现）
