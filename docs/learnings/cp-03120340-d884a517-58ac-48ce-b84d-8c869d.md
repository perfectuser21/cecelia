# Learning: tick step 6.5 梯度 requeue — DoD grep 与实际 SQL 须对齐

## 背景

实现 billing_pause 过期后梯度释放 quota_exhausted 任务（PR #894）。
把原来的 `UPDATE ... WHERE status='quota_exhausted'` 改为带 LIMIT 的子查询。

## 踩坑：DoD grep 命令与实际 SQL 不一致

**问题**：DoD 写的测试命令是：
```bash
grep -c 'ORDER BY priority ASC'
```
但实际 SQL 用的是 CASE 表达式：
```sql
ORDER BY CASE priority WHEN 'P0' THEN 0 WHEN 'P1' THEN 1 ELSE 9 END ASC
```
导致 CI DoD Verification Gate 失败（grep 返回 0，exit code 1）。

**修复**：DoD 测试命令改为匹配实际写法：
```bash
grep -c 'ORDER BY CASE priority'
```

**规律**：DoD Test 命令写完后，必须在本地实际执行一遍验证，确认 exit code 为 0，再 push。
不要假设 grep 字符串和代码实现一致。

## PostgreSQL UPDATE...ORDER BY 必须用子查询

PostgreSQL 不支持 `UPDATE ... ORDER BY`，必须改写成：
```sql
UPDATE tasks SET status = 'queued'
WHERE id IN (
  SELECT id FROM tasks
  WHERE status = 'quota_exhausted'
  ORDER BY ... LIMIT $1
)
```

MySQL 支持直接 `UPDATE...ORDER BY LIMIT`，PostgreSQL 不行。

## 梯度流控常量命名规范

`tick.js` 已有同类常量区：
- `UNBLOCK_BATCH_LIMIT = 5`（blocked 任务每 tick 释放上限）
- `QUARANTINE_RELEASE_LIMIT = 2`（quarantine 任务每 tick 释放上限）
- `MAX_REQUEUE_PER_TICK = 2`（新增，quota_exhausted 每 tick 释放上限）

三者命名风格略有不一致（LIMIT vs MAX...PER_TICK），未来统一命名时可参考。
