# Learning: Scanner 误判内嵌能力为孤岛

**分支**: cp-03230914-6b03e188-c3ec-4635-bcb0-864a41
**日期**: 2026-03-23
**类型**: 架构认知修正

---

### 根本原因

capability-scanner 仅通过三条证据链判断能力活跃度：
1. `run_events` / `tasks` 表中的技能使用记录
2. `cecelia_events` 中的特定 source 事件
3. `key_tables` 中是否有数据

但 Brain 有一类"架构性能力"（`dev-workflow`、`self-healing`、`self-healing-immunity` 等），
它们**不产生任何可观测的 DB 证据**，而是作为 Brain 进程的固有组成部分始终存在：
- `dev-workflow`：Brain 随时可派发 /dev 任务，无需预先调用证明存在
- `self-healing` / `self-healing-immunity`：免疫系统未触发 = 系统健康，不等于能力不存在

扫描器用"无证据 = 孤岛"的逻辑，误判了"存在即运行但无 DB 足迹"的能力类型。

---

### 次要原因（预存在测试问题）

1. **learning.js catch 降级**：OpenAI 429 quota 错误导致 vectorSearch 失败，catch 直接返回
   `getRecentLearnings`（无 `relevance_score`），破坏了测试的排序断言
2. **llm-caller.test.js openai 路径**：`OPENAI_API_KEY` 存在时 fetch 被实际调用但无 mock，
   导致 `undefined.ok` TypeError

---

### 下次预防

- [ ] 扩充 Brain 内嵌能力白名单时，同步在 `BRAIN_ALWAYS_ACTIVE` 注释中说明"无 DB 足迹原因"
- [ ] 新增能力到 capabilities 表时，若无 skill/event/table 证据，优先检查是否应归入 BRAIN_ALWAYS_ACTIVE 或 INFRA_DEPLOYED_CAPABILITIES
- [ ] learning.js 的任何外部依赖（OpenAI/向量搜索）失败路径，catch 必须保证返回相同结构（含 relevance_score）
- [ ] 测试依赖环境变量（OPENAI_API_KEY）时，必须在测试开头 mock 所有可能的 fetch 路径，不依赖"key 不存在"的隐式行为
