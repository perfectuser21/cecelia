# Phase 7.4 — /dev 7 棒接力链 + Stop Hook 循环机制 E2E regression test

## 背景

Phase 7.1 / 7.2 / 7.3 三连发把 Stop Hook 循环机制从「形同虚设」修到「端到端能跑」：

- **Phase 7.1 (v18.1.0)**：新增 `scripts/claude-launch.sh`，强制 `--session-id` + export `$CLAUDE_SESSION_ID`，让交互 claude 也能拿到 session_id。配合 `worktree-manage.sh::_resolve_claude_session_id` 的 env 优先分支，`.dev-lock` 的 `owner_session` 字段能被正确写入。
- **Phase 7.2 (v18.1.1)**：`hooks/stop.sh` 的 `_STOP_HOOK_WT_LIST` 空数组在 bash 3.2 + `set -u` 下炸 `unbound variable` bug（Phase 7.1 之前早退所以没暴露），用 `${arr[@]+${arr[@]}}` guard 修了。
- **Phase 7.3 (v18.3.1)**：全仓扫 11 处同类潜伏炸弹，统一加 guard。

这三个 Phase 每次都加了**单点**回归测试（launcher 透传 / session-resolve env 优先 / stop.sh 空数组 guard / multi-worktree 路由 / hardening-sweep 20 条断言），但**没有**把这条从 launcher 一路到 devloop-check 的「接力链」作为一个整体 E2E 覆盖起来。任何一环未来被某个无关 PR 破坏，CI 都不会发现——只能等下次交互 /dev 翻车、Stop Hook 提前 exit 0 放行，看到 assistant 中途退出才知道。

### 根本原因

单元测试覆盖了每块积木，但接力链的「关节」没有集成测试。举例：

1. 某人把 `claude-launch.sh` 的 `export CLAUDE_SESSION_ID="$SID"` 删了——launcher.test.ts 的「有 env 时透传」依然过（test 自己 set 了 env），但真实调用里 env 不会进子进程。
2. 某人改 `worktree-manage.sh` 的 `_resolve_claude_session_id`，让它先走 PPID fallback——session-resolve.test.ts 的「env 优先」那条会 fail，但如果有人「顺手」把 env 优先逻辑改错，PPID 命中场景 test 并不覆盖。
3. 某人改 `stop.sh` 的 `for _wt in "${_STOP_HOOK_WT_LIST[@]+...}"` 逻辑——multi-worktree-routing.test.ts 测的是具体 session 路由行为，没测「当 owner_session 不匹配任何 lock 时 exit 0 不路由」这条关键安全兜底（Phase 7 之前误路由的根因）。

单元 test 只保护「已知的已知」，E2E test 保护「链条的完整性」。

## 方案

新增 `packages/engine/tests/integration/dev-flow-e2e.test.ts`，用 vitest + `execSync` 驱动真实 bash 脚本，不起 claude CLI（mock 关键环节）。覆盖 6 个场景：

1. **S1 — claude-launch 链路**：mock 一个假 `claude` 脚本（dump env + args 到 stdout）放入 PATH，跑 `CLAUDE_SESSION_ID=fixed bash scripts/claude-launch.sh --help`，断言 mock 同时收到 `CLAUDE_SESSION_ID=fixed` env 和 `--session-id fixed` 参数。
2. **S2 — worktree-manage 按 env 写 owner_session**：临时 git repo + `CLAUDE_SESSION_ID=test-env-uuid` 跑 `worktree-manage.sh create ...`，断言生成的 `.dev-lock.*` 文件里 `owner_session: test-env-uuid`（走 `_resolve_claude_session_id` 的 env 优先分支）。
3. **S3 — stop.sh 按 owner_session 精确路由**：真实 git repo + 2 worktree 各有不同 owner_session 的 `.dev-lock`，用 `CLAUDE_HOOK_STDIN_JSON_OVERRIDE` env 注入 session_id，断言匹配的 session 路由到 stop-dev.sh，不匹配的 session exit 0 不路由。
4. **S4 — stop.sh 空数组 guard（Phase 7.2 回归）**：空目录 + stdin JSON，断言 stop.sh 不含 "unbound variable" 且 exit 0。
5. **S5 — devloop-check CI 失败 → blocked**：source `devloop-check.sh`，PATH 前置 mock gh（返回 open PR + `{status:completed, conclusion:failure}`），断言输出 JSON `status="blocked"` + `reason` 含 "CI 失败"。
6. **S6 — devloop-check cleanup_done → done**：`.dev-mode` 含 `cleanup_done: true`，断言 `status="done"`。

### 下次预防

新增的 E2E test 在整条接力链关键关节被破坏时都会炸：

- Launcher 不 export env → S1 fail
- worktree-manage 不读 env → S2 fail
- stop.sh 不按 owner_session 匹配 → S3 fail
- stop.sh 空数组 guard 被删 → S4 fail
- devloop-check CI 失败分支被改错 → S5 fail
- devloop-check cleanup_done 出口被挪位置 → S6 fail

### 新写关键接力链/脚本时的 checklist

- [ ] 新增 / 修改 `hooks/stop*.sh` / `scripts/claude-launch.sh` / `worktree-manage.sh` / `lib/devloop-check.sh` 任一个 → **必须更新** `packages/engine/tests/integration/dev-flow-e2e.test.ts` 对应场景
- [ ] 新增「链式依赖」（一个脚本依赖另一个脚本的输出/副作用）→ 在 integration/ 下加端到端 test，不要只加单元 test
- [ ] mock 外部 CLI（gh / git / claude）时用 PATH 前置机制，不要 `vi.mock` 或 `sinon`（bash 脚本不走 JS module resolution）
- [ ] 涉及 jq 输出断言用 `toMatch(/"status":\s*"blocked"/)` 容忍空白，不要 `toContain('"status":"blocked"')` 死匹配
- [ ] 临时 git repo + `-b main` + `config user.email/name` + `commit --allow-empty -qm init` 是 CI-safe 的起点
- [ ] 修改接力链任何一环时运行 `npx vitest run tests/integration/dev-flow-e2e.test.ts` 先验证 6 个场景都过
