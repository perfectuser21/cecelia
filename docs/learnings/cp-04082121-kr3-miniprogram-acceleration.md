### 根本原因

KR3 卡在 25% 的核心原因不是代码量不足，而是**微信小程序上线链路有人工操作无法绕过的环节**：云函数部署、体验版提交、审核提交均需要人在微信开发者工具中手动操作。Cecelia/Codex 可以产出所有代码，但无法替代这些操作。

次要原因是**范围悄然扩大**：原始 KR3 定义"基础功能可用"，实际却在执行中扩展为 AI 聊天 + 会员付费 + 内容 CMS + Copywriter 创作流 4 个模块，每个都有 70-80% 完成度但没一个达到"可上线"。

### 下次预防

- [ ] 新增"人工操作检查清单"：凡涉及第三方平台（微信/App Store/Google Play）的上线，需在 KR 定义阶段就明确"谁、用什么工具、哪天操作"
- [ ] KR 定义中加"MVP 范围边界"条款：明确哪些功能是 v1 必须，哪些是 v2，避免 scope 蔓延
- [ ] 分析任务（SelfDrive 类型）产出文档后，Brain 应自动将分析结论写入 design-docs 表，方便后续 SelfDrive 循环引用
- [ ] 小程序 KR 进度拆分为 4 个子检查点：代码完成(25%) / 云函数部署(60%) / 提交审核(80%) / 上线(100%)，而不是粗糙的百分比
