# Learning: rumination 链路 NotebookLM 前缀噪音污染

## 根本原因

NotebookLM CLI 在切换 notebook 时（notebook_id 与当前活跃会话不一致），
会在真实答案前注入固定前缀：
```
Different notebook specified, starting new conversation...
Continuing conversation b6c941fc...
Answer:
[真实内容]
```

此前缀长度 > 50 字符，通过了 `length > 50` 质量检查，被误写入 memory_stream。
同时 cecelia_events 中 rumination_output 无记录（jsonb INSERT 类型推断问题），
导致 dedup 失效，相同前缀文本反复写入。

## 下次预防

- [ ] NotebookLM 响应处理：接收后先调用 `cleanNotebookResponse()` 剥离前缀，再做长度检查
- [ ] 凡向 jsonb 列 INSERT 参数，在 SQL 中显式加 `::jsonb` cast 或传对象类型
- [ ] memory_stream 写入前对内容做关键字检查：含 "Different notebook specified" 则拒绝写入

## 修复内容

1. 新增 `cleanNotebookResponse()` 导出函数（`rumination.js`）
2. `digestLearnings()` 在 NotebookLM 响应后调用清洗，清洗后重新验证 > 50 字符
3. `cecelia_events` INSERT 改用 `$1::jsonb` 显式类型转换
