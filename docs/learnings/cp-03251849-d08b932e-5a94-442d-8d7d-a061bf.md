# Learning: rumination NotebookLM 前缀噪音清洗

**Branch**: cp-03251849-d08b932e-5a94-442d-8d7d-a061bf
**Date**: 2026-03-26

### 根本原因

NotebookLM CLI 切换 notebook 时，在真实 Answer 前注入前缀文本：
```
Different notebook specified, starting new conversation...
Continuing conversation b6c941fc...
Answer:
[真实内容]
```
旧代码只检查 `nbResult.text.trim().length > 50`，该前缀（~80字符）直接通过，
导致垃圾前缀被写入 `memory_stream` 作为"洞察"。同时 `cecelia_events` 的 `rumination_output`
INSERT 使用 `JSON.stringify()` 字符串传给 jsonb 参数可能导致类型推断失败，
使 dedup 机制完全失效，同一前缀文本被多次重复写入。

### 修复方案

- 新增 `cleanNotebookResponse(text)` 函数：检测前缀 → 提取 "Answer:" 后内容
- 在 `digestLearnings()` 中获得 NotebookLM 响应后立即清洗，重新验证 `cleaned.length > 50`
- 清洗后仍不足 → fallback 到 callLLM，确保生成真实洞察
- `cecelia_events` INSERT 改用 `$1::jsonb` 显式类型转换，修复 dedup 机制
