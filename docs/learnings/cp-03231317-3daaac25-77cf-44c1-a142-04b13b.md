# Learning: SelfDrive 24h 任务失败诊断

## 任务概述

分析最近 24h 任务 11% 失败率的根因，输出优先级排序的修复建议。

### 根本原因

失败集中在一个 4.5h 窗口（05:27-10:00 上海时间），72% 的失败与任务内容质量无关，属于基础设施/调度层问题：

1. **P0 - worktree 并发冲突**（36%）：多任务同时创建 worktree，无 flock 互斥锁，git 内部锁竞争导致 `worktree_creation_failed`
2. **P1 - liveness 阈值过严**（28%）：西安 Codex 任务 5 分钟心跳阈值太严格，网络延迟导致正常运行任务被 Watchdog kill
3. **P1 - Brain 重启孤儿任务**（24%）：重启后 in_progress 任务直接标 failed 而非 requeue
4. **P1 - 执行器预启动失败不重试**（8%）：error_during_execution(0 turns) 被归为 TASK_ERROR 而非 transient，不触发重试

### 下次预防

- [ ] P0 修复：`worktree-manage.sh` 加 `flock -w 30` 互斥锁，彻底消除并发冲突
- [ ] P1 修复：`monitor-loop.js` 按 location='cn' 使用 15min 阈值（而非固定 5min）
- [ ] P1 修复：Brain 启动时孤儿任务改为 `requeue` 而非直接 `failed`
- [ ] P1 修复：`quarantine.js` 识别 `error_during_execution + num_turns=0` 为 transient，触发重试
- [ ] 数据可见性：`task_run_metrics` 表（PR #1476）已启用，下次可直接查失败 `exit_status`
- [ ] 诊断提示：Brain 重启前后应记录 `brain_restart` 事件到 `cecelia_events`，便于关联分析

### 关键结论

转折点 12:09 后 41 个连续任务 100% 成功，说明系统已自愈。上午失败波是一次性系统扰动，不代表代码质量问题。修复上述 4 项后，预期成功率可从 89% 提升至 ≥97%。
