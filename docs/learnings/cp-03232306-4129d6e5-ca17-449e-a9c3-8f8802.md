# Learning: strategy_session KR 断链修复

**Branch**: cp-03232306-4129d6e5-ca17-449e-a9c3-8f8802
**Date**: 2026-03-24

### 根本原因

OKR 迁移（PR1-PR9）完成后，`key_results.objective_id` 变为关联 `objectives` 表的外键，但 `execution.js` 的 strategy_session 回调未同步更新。`INSERT INTO key_results` 仅传 `title/status/owner_role/metadata`，未传 `objective_id`，导致所有 strategy_session 产生的 KR 成为孤岛（OKR 树不可见）。

### 下次预防

- [ ] OKR 相关表 schema 迁移（`objectives`/`key_results` FK）时，需全局搜索所有 `INSERT INTO key_results` 语句，逐一确认 `objective_id` 传递
- [ ] strategy_session 等特殊 task_type 的执行回调逻辑（`execution.js` 中的 `if (ssTaskType === 'strategy_session')` 分支）在 OKR 迁移后必须单独审查
- [ ] DoD 模板：修改写入 OKR 表的路由时，[PRESERVE] 条目必须覆盖 FK 关联完整性（不仅是功能保持）

### 实现要点

三级 fallback 链避免孤岛：
1. `parsedOutput.objective_id`（策略会议 JSON 顶层携带）
2. `task.payload.objective_id`（任务创建时注入）
3. auto-create：以 `meeting_summary` 前 80 字或 task title 为标题，INSERT INTO objectives 得到 id
