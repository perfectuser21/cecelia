**分支**: cp-03292000-a39703b4-766a-4978-8e23-7d1950
**日期**: 2026-03-30

### 根本原因

1. **DoD Test 字段格式**：Planner subagent 生成的 Test 字段使用了 `  - Test:` 格式（带 dash），但 verify-step.sh 要求 `  Test:`（无 dash）。导致 bash-guard.sh 拦截 `.dev-mode` 写入。

2. **`accessSync` 无显式断言**：`node -e "require('fs').accessSync('...')"` 虽然在文件不存在时会抛出异常并退出非零，但 check-dod-mapping.cjs 的静态分析认为该命令"无 `process.exit/throw/if` 等退出码判定"，导致 DoD 映射检查失败。正确写法：`try{...}catch(e){process.exit(1)}`。

3. **spec_review 中 CI 文件名错误**：首次 spec_review 建议的 DoD7 测试引用了 `engine-ci.yml/brain-ci.yml/workspace-ci.yml`，但实际仓库的 CI 文件名为 `ci-l1-process.yml/ci-l2-consistency.yml` 等。需先通过 `ls .github/workflows/` 确认实际文件名。

4. **Learning 文件须在首次 push 前提交**：首次推送后 CI 的 L1 Learning Format Gate 要求 Learning 文件已在本次 PR 的提交中，否则 Stage 4 判定为"缺失 Learning"导致 CI FAIL。

### 下次预防

- [ ] Planner subagent prompt 中明确写明 Test 字段格式规则：`  Test: manual:...`（无 dash）
- [ ] DoD 文件存在验证统一使用 `try{require('fs').accessSync('...')}catch(e){process.exit(1)}`，避免无断言假测试
- [ ] spec_review 或 Generator 中引用现有文件路径前，先执行 `ls` 确认文件名
- [ ] Learning 文件必须在首次 push **之前**写好并加入 commit（与 pr-review.yml 同一 commit，或单独 commit 在 push 前）
