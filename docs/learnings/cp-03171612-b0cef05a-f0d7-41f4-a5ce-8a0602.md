# Learning: fix(ci): CI 审计 5 个问题修复

## 任务信息
- 分支: cp-03171612-b0cef05a-f0d7-41f4-a5ce-8a0602
- 日期: 2026-03-17
- 类型: CI 配置修复

## 根本原因

### 问题1 (P0): coverage-delta 被 skip 绕过
`coverage-delta` job 的 `needs` 包含 `brain-unit`，导致 `brain-unit` 被 skip 时
`coverage-delta` 的 `if` 条件中 `brain-unit.result == 'success'` 永远为 false，
整个 coverage-delta 也被跳过。正确做法：coverage-delta 只依赖直接条件判断，
不依赖上游 job 的 result。

### 问题2 (P1): .mts 扩展名漏判
`TEST_PATTERNS` 只列了 `.ts/.js/.cjs/.mjs`，缺少 `.mts`（TypeScript ES Module）。
随着 ES Module 的普及，`.mts` 测试文件已出现，不能漏。

### 问题3 (P1): [CONFIG] 豁免路径太宽
`^packages/engine/` 匹配了整个 engine 目录，包括 `packages/engine/docs/`。
改 docs 也能豁免 test coverage 检查，绕过了 feat PR 必须带测试的规则。
正确做法：只匹配实际的 CI/配置子目录。

### 问题4 (P2): release: 前缀未豁免 version check
`release:` 类型的 PR 是版本发布操作，已由自动化处理版本 bump，
不应要求手动 bump。与 chore/docs/test 保持一致。

### 问题5 (P2): changes 失败时 engine 检查静默跳过
`l1-passed` gate 没有检查 `changes.result`。当 changes job 失败时，
`changes.outputs.engine` 输出为空（不等于 'true'），engine 检查条件为 false，
被静默跳过——l1-passed 误报通过。

## 下次预防

- [ ] CI gate job 必须先检查所有 `needs` 中的 changes/detect job 是否成功
- [ ] job 的 `if` 条件不应依赖其他 job 的 `.result`（用直接条件替代）
- [ ] TEST_PATTERNS 新增扩展名时同步检查 .mts/.cts 等变体
- [ ] [CONFIG] 等豁免标签的路径匹配应明确列出子目录，避免宽泛的前缀匹配
- [ ] 豁免列表（chore/docs/test/release）每次新增 PR 类型时同步更新所有相关 if 条件
