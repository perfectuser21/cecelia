## Content Pipeline 错误可观测性修复（2026-03-30）

### 根本原因

`tasks.error_message` 列在 migration #142 已添加，但 `content-pipeline-orchestrator.js` 中所有失败路径的 UPDATE SQL 均未包含该字段，导致 pipeline 失败时原因不可见。主要遗漏点：
1. `_startOnePipeline`：content_type 无效时只更新 status，不写 error_message
2. `orchestrateContentPipelines` catch 块：只打印日志，不写 DB
3. `_handleCopyReviewComplete` / `_handleImageReviewComplete`：重试超限时只更新 status
4. `_executeStageTask`：executor 返回 failed+error 时走了无 error_message 的通用分支
5. `executeQueuedContentTasks` catch 块：同样只更新 status
6. `GET /api/brain/pipelines` 路由 SELECT 缺少 error_message 字段

### 下次预防

- [ ] 新增 DB 列后，立即全局搜索相关 UPDATE 语句，确保所有写入路径都包含新列
- [ ] 失败路径的 UPDATE SQL 模板：统一使用 `SET status = $2, completed_at = NOW(), error_message = $3` 形式，而不是硬编码 `status = 'failed'`（便于参数化）
- [ ] 路由 SELECT 语句改动后，对应的测试应显式断言响应体包含新字段（not just length）
- [ ] TDD 先写红测试可有效发现"列存在但未被使用"的问题
