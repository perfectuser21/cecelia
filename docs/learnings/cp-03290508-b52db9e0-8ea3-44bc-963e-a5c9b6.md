**任务**: 升级 /dev Sprint Contract 机制：双向协商 DoD + Evaluator Calibration

---

### 根本原因

修改 `packages/workflows/skills/` 下的文件时，branch-protect.sh 使用就近检测：
从文件路径向上找 `.prd.md` 或 `.prd-{branch}.md`，在到达项目根目录之前已找到 `packages/workflows/.prd.md`（旧任务文件），因此使用了该目录而非根目录。

- 根目录的 `.prd-{branch}.md` 不会被检测到
- 必须在 **`packages/workflows/`** 也放 `.prd-{branch}.md` 和 `.dod-{branch}.md`

同理，修改 `packages/engine/` 下的文件时，根目录的 `.prd-{branch}.md` 生效（engine 目录无旧 .prd.md），但需要根目录也有 `.dod-{branch}.md`。

---

### 下次预防

- [ ] 修改 `packages/workflows/skills/` 时，PRD/DoD 必须放两处：
  1. 项目根目录：`.prd-{branch}.md` + `.dod-{branch}.md`
  2. `packages/workflows/`：`.prd-{branch}.md` + `.dod-{branch}.md`
- [ ] 修改 `packages/engine/skills/` 时，PRD/DoD 放项目根目录即可（engine 目录无旧 `.prd.md`）
- [ ] `.task-cp-{branch}.md` 的命名前缀是 `cp-` 不是 `task-cp-`——hook 查找 `.task-{BRANCH}.md`，BRANCH 已含 `cp-` 前缀，因此文件名为 `.task-cp-{branch-without-cp-prefix}...` 不对，实际文件名就是 `.task-{BRANCH}.md` = `.task-cp-xxxxx.md` ✓

### 已验证的工作流顺序

1. 创建 `.task-cp-{BRANCH}.md`（Stage 1 Task Card）
2. 创建 `.prd-{BRANCH}.md`（根目录）
3. 创建 `.dod-{BRANCH}.md`（根目录）
4. 创建 `packages/workflows/.prd-{BRANCH}.md`（workflows 就近检测）
5. 创建 `packages/workflows/.dod-{BRANCH}.md`（workflows 就近检测）
6. 在 `.dev-mode` 加 `tasks_created: true`
7. 运行 `verify-step.sh step2` 后再标记 `step_2_code: done`
