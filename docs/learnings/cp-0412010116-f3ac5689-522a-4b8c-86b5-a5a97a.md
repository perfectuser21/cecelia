# Learning: 重构 preparePrompt 圈复杂度（77→24→6）

## 根本原因

Brain 复杂度扫描器（complexity-scanner.js）使用大括号配对算法找函数体边界，**不区分内层箭头函数作用域**。`preparePrompt` 函数体内定义的路由表包含大量 `(t) => ...` 内联 lambda，lambda 内部的 `||` 运算符全部被计入父函数复杂度。

前一次重构（PR #2273）把 Harness 系列逻辑提取为 5 个命名子函数（77→24），但路由表的内联 lambda 仍留在函数体内，残余 15 个 `||` 导致复杂度仍为 24。

## 下次预防

- [ ] 在有路由表（`const routes = { ... }`）的函数中，若 lambda 含 `||`/`&&`，考虑直接提为命名函数，避免被扫描器误计为父函数复杂度
- [ ] 复杂度扫描任务创建时，先用 `node -e` 模拟扫描器的 `countBranches` 逻辑确认实际值，再规划重构幅度
- [ ] 路由表放模块级常量（`const _TASK_ROUTES = {...}`）不仅降低 dispatcher 复杂度，还有"只创建一次"的性能优势

## 解决方案

1. 提取 6 个内联 lambda 为命名函数：`_prepareInitiativePlanPrompt`、`_prepareInitiativeVerifyPrompt`、`_prepareArchitectureDesignPrompt`、`_prepareDecompReviewPrompt`、`_preparePrdReviewPrompt`、`_prepareInitiativeReviewPrompt`
2. 将路由表提升为模块级常量 `_TASK_ROUTES`（脱离 `preparePrompt` 函数体）
3. 用 Set 常量替换 `||` 条件链：`_DECOMP_TYPES`、`_HARNESS_GENERATE_TYPES`
4. 提取 `_isSprintOrHarnessDevMode()` 辅助函数

结果：`preparePrompt` cyclomatic 从 24 → 6（threshold 10）。
