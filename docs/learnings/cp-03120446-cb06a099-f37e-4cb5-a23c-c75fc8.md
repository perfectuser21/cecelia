# Learning: migration 152 — DEFINITION.md 两处 schema_version 必须同步（2026-03-12）

## 背景

PR #901 为 `learnings` 表新增 `quality_score` + `source_type` 字段（migration 152）。selfcheck.js 的 EXPECTED_SCHEMA_VERSION 从 '151' 更新到 '152'，但遗漏了 DEFINITION.md 两处同步，导致 L2 Consistency Gate 失败。

## 根本原因

`facts-check.mjs` 校验 `selfcheck.js` 中的 `EXPECTED_SCHEMA_VERSION` 与 `DEFINITION.md` 中的版本号是否一致。

`DEFINITION.md` 有**两处**记录 schema_version：
1. 第 386 行表格：`| **schema_version** | 迁移版本追踪 | Schema 版本: 151 |`
2. 第 676 行描述：`5. **Schema 版本** — DB 版本 >= '151'（>= 检查，向前兼容）`

只更新 selfcheck.js 而不更新 DEFINITION.md → L2 facts-check 失败：
```
✗ schema_version: code=152 (selfcheck.js:18) ≠ doc=151 (DEFINITION.md)
```

## CI-safe 单测路径层级陷阱

从 `packages/brain/src/__tests__/` 向上解析路径时，`../../../` 跳三层到 `packages/`，不是 `packages/brain/`：
- `../` → `packages/brain/src/`
- `../../` → `packages/brain/`
- `../../migrations/` → `packages/brain/migrations/` ✓
- `../../../migrations/` → `packages/migrations/` ✗（不存在）

## 下次预防

- [ ] 每次修改 EXPECTED_SCHEMA_VERSION 时，先 `grep -n "schema_version\|<旧版本>" DEFINITION.md` 找到所有位置再一起改
- [ ] migration PR 的改动范围：migration SQL + selfcheck.js + DEFINITION.md（两处）+ 单测
- [ ] 新建 CI-safe 测试时，仔细数从 `__tests__/` 到目标目录的相对层级（`../../migrations/`，不是 `../../../`）
