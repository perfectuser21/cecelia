## Schema sync 241→243（2026-04-23）

### 根本原因

`packages/brain/src/selfcheck.js` 的 `EXPECTED_SCHEMA_VERSION` 常量和 `packages/brain/migrations/` 目录最高号长期不同步：#2515 合并 migration 242 + 243 时 `[CONFIG]` 前缀 PR 经过了"可跳过 facts-check"的路径，常量没被 bump，留下 241 vs 243 漂移。几天后我推 #2538（想在 `packages/brain/src/*/` 下加 README）时 local-precheck 的 facts-check 才把这个坑暴露。

更深一层：facts-check.mjs 有**两段检查**——Section 1 对 DEFINITION.md 的 regex 匹配 + Section 2 代码完整性（`selfcheck_version_sync`）。PR #2515 bypass 时只过了静态 lint，没真跑 facts-check。而日常 push 路径里这两段都会跑，所以漂移被保留到下一个真推 Brain 的 PR 才爆雷。

### 下次预防

- [ ] `[CONFIG]` / admin-merge 路径不应跳过 facts-check —— facts-check 是**事实对齐**检查（migration 号、schema version、task_types），和"配置变更 vs 代码变更"无关。建议 engine-devline 把 facts-check 挪到一个**不可 bypass** 的 CI job 里（当前在 local-precheck 和部分 CI 流里，有 bypass 口子）
- [ ] migration 加号和 `EXPECTED_SCHEMA_VERSION` 常量 bump 之间应该有 **pre-commit 强耦合**：加新 migration 文件时自动更新常量（或 pre-commit hook 拒绝 commit 直到两者对齐）
- [ ] DoD 验收命令要覆盖整个 facts-check（不能只验 Section 2 的 `selfcheck_version_sync`）——这次评审第一轮里 spec reviewer 就误判了，以为 DEFINITION.md 改动是越界。以后写 DoD `[BEHAVIOR]` 时，`facts-check 通过` 要明确标注"含 Section 1 的 DEFINITION.md 同步"
- [ ] 硬编码断言反模式：`selfcheck.test.js:150` 和 `learnings-vectorize.test.js:434` 都在 import 了常量后又 `expect(EXPECTED_SCHEMA_VERSION).toBe('241')` — 这种"断言常量等于自己"的测试每次 bump 都要跟着改，没实际价值。将来重构 selfcheck 时应该改成语义检查（例如 `>= MIN_SCHEMA_VERSION` 或 `matches migrations max`）
