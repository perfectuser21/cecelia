# Learning: Scanner stage≥3 能力误判为孤岛

**Branch**: cp-03231315-7a262d20-76c4-4d58-9450-6d9514
**Date**: 2026-03-23
**Task**: [SelfDrive] 修正 Scanner 对内嵌能力的识别规则

---

### 根本原因

`determineStatus()` 函数在 `!hasSkillActivity && !hasTableData` 时直接返回 `island`，
未考虑能力的 `current_stage`。

**语义错误**：`island` = 能力未部署/不存在；`dormant` = 能力存在但未被触发。
stage=3（已部署运行）的能力在无活动记录时（例如新安装后扫描早于首次任务）被误判为孤岛。

PR #1479 通过手动加白名单修复了具体实例（`dev-workflow`、`self-healing`、`self-healing-immunity`），
但缺乏防止未来同类误判的系统性规则。

### 修复方案

`determineStatus()` 新增第五个参数 `stage`，`evaluateCapability()` 传入 `cap.current_stage`：

```js
if (!hasSkillActivity && !hasTableData) {
  return (stage >= 3) ? 'dormant' : 'island';
}
```

**规则**：stage ≥ 3 = 已完全部署，无活动 = 休眠；island 保留给 stage < 3 的规划中能力。

### 下次预防

- [ ] 新增 capability 时，若 `current_stage=3` 且无 related_skills/key_tables，Scanner 自动标记为 `dormant` 而非 `island`
- [ ] 手动加白名单前先检查是否可以通过 stage 规则覆盖（避免白名单膨胀）
- [ ] 每次 Scanner 行为修复后，确认新的通用规则是否能替代既有白名单条目
