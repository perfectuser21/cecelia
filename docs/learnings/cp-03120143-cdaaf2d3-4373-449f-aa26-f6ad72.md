# Learning: KR 进度诚实化 — /audit 端点注册顺序 + DEFINITION.md 双处同步

## 背景

PR #885 实现 `GET /api/brain/tasks/goals/audit` 端点和 migration 150，将虚标 KR progress 修正为基于 initiative 实际完成率的真实值。

## 关键陷阱

### 1. Express 路由注册顺序：静态路径必须在 /:id 之前

`task-goals.js` 原有 `GET /:id` 路由。新增 `GET /audit` 时必须注册在 `/:id` **之前**，否则 Express 会把字符串 `'audit'` 当成 `:id` 参数匹配，导致 404。

**做法**：在文件中把 `/audit` 路由代码放在 `GET /` 之前，并加注释 `// 必须在 /:id 之前注册`。

### 2. DEFINITION.md schema_version 有两处需同步

修改 `selfcheck.js` 的 `EXPECTED_SCHEMA_VERSION` 时，DEFINITION.md 里有**两处**硬编码：
- `packages/brain/src/selfcheck.js:18` → 代码值
- `DEFINITION.md line ~386`：`Schema 版本: NNN`
- `DEFINITION.md line ~676`：`DB 版本 >= 'NNN'`

`facts-check.mjs` 只校验 `code` 与 `DEFINITION.md` 两者是否一致，漏改任何一处都会导致 L2 Consistency Gate 失败。

**做法**：改 selfcheck.js 后，立即用 `grep -n "schema_version\|<旧版本>" DEFINITION.md` 找两处并全部更新。

### 3. migration 中 WITH + UPDATE + INSERT 的组合模式

要同时更新数据并记录审计日志，可用 PostgreSQL 的 `WITH` CTE：

```sql
WITH updated AS (
  UPDATE goals ... RETURNING *
)
INSERT INTO memory_stream ...
SELECT ... FROM updated;
```

注意：`RETURNING` 子句返回的是 UPDATE 前的列值（除非 CTE 内用 `RETURNING goals.id, kr_actual.new_progress` 等），要确认字段对应关系。

## 结论

- Express 静态路由必须在参数化路由前注册
- schema_version 改动必须同步 selfcheck.js + DEFINITION.md（两处）
- migration 中 CTE+UPDATE+INSERT 是写入审计日志的惯用模式
