- 分支: cp-03221802-844576b3-ebc1-475a-a61b-d3b4ee
- 日期: 2026-03-22

### 根本原因
`analyzeDeep` 函数将 8 个异步数据源获取、LLM 调用、响应处理、错误处理全部内联在一个函数体中，每个 try/catch + if 分支都累加圈复杂度，导致最终复杂度达到 55。

### 重构模式
将函数按"职责"而非"代码顺序"拆分：
1. 每个数据源一个私有辅助函数（`_enrichWithXxx`）
2. LLM 响应处理提取为 `_processLLMResponse`
3. 错误处理提取为 `_handleAnalysisError`
4. 主函数 `analyzeDeep` 只做协调（圈复杂度从 55 降至 4）

### 下次预防

- [ ] 函数超过 80 行时考虑是否需要提取辅助函数
- [ ] 多个 try/catch 块并列时，应各自提取为独立函数
- [ ] 每个"数据获取源"都应该有自己的命名函数，而不是内联 try/catch
- [ ] 提取的私有辅助函数以 `_` 前缀命名，明确标识为模块内部实现细节
- [ ] DoD 的 `Test:` 命令中检查 export 白名单时，需先确认文件中所有已有的 export（避免漏掉历史 export 导致误判）
