# Learning: auth 类失败误触发熔断器，导致 dispatch 系统停摆

## 根因链路

```
account2 OAuth token 过期
  → cecelia-run 执行 heartbeat 任务报 "Failed to authenticate. OAuth token has expired"
  → classifyFailure() 未匹配 AUTH_PATTERNS（旧规则只有 "authentication failed"，不匹配 "Failed to authenticate"）
  → 被分类为 task_error（默认）
  → cbFailure('cecelia-run') 被调用
  → 连续 3 次 → circuit breaker OPEN（30 分钟）
  → 所有 dispatch 阻塞（dispatch success rate 跌至 7.9%）
  → 30 分钟后 HALF_OPEN → 成功任务恢复 → CLOSED
  → 下一轮 heartbeat 任务 → 再次失败 → 再次 OPEN（循环）
```

## 修复

1. **quarantine.js AUTH_PATTERNS** 新增两条规则：
   - `/authentication\s+failed|auth\s+error|failed\s+to\s+authenticate/i` — 覆盖 "Failed to authenticate"（顺序颠倒的写法）
   - `/OAuth.*token.*expired|token.*has.*expired|token.*expired/i` — 覆盖 OAuth token 过期错误

2. **routes.js cbFailure 旁路** 将 `auth` class 加入旁路条件：
   ```js
   isTransientApiError = classification.class === 'rate_limit'
     || classification.class === 'network'
     || classification.class === 'auth';  // 新增：凭据问题不计入熔断
   ```

## 经验总结

### AUTH_PATTERNS 匹配陷阱

错误信息往往不是"标准"格式。`authentication failed` 和 `Failed to authenticate` 是同一语义但词序相反，必须都覆盖。添加新模式时要检查真实错误日志，不能只凭直觉写正则。

### 熔断器设计原则

**单账户凭据过期不应导致全局 dispatch 停摆**。凭据失败（auth）与系统故障（code error）是本质不同的两类问题：
- `auth` → 凭据层面的问题，`cecelia-run` 本身是健康的
- `network/rate_limit` → 外部 API 瞬时问题，同上
- `code_error/task_error` → 代码/任务本身问题，才应计入熔断

### DoD Test 命令规范

Test 命令必须使用**相对路径**（相对于 repo root），不能写绝对路径（如 `/Users/administrator/perfect21/...`）。CI 在 runner 上执行时绝对路径不存在，导致 grep 返回非零退出码，DoD 检查失败。

正确格式：
```
Test: manual:bash -c "grep -c 'pattern' packages/brain/src/file.js"
```

### circuit-breaker-transient-bypass 测试模板

这个测试文件已经有完整的 mock 框架，添加新的旁路类型只需在 describe 块末尾追加 `it()` 即可，不需要修改 mock 设置。

## 未解决问题

- account2 OAuth token 仍然过期，需用户手动 re-auth（`claude auth login` in `.claude-account2` 配置目录）
- 隔离区中的 heartbeat zenithjoy 任务将在 24h TTL 后自动释放，但届时仍会失败（auth 错误），不过这次不再触发 circuit breaker
