**PR**: feat(engine): DoD Test 字段强度升级 — 拦截弱测试命令和 CI 反模式

---

### 根本原因

`check-dod-mapping.cjs` 的 `detectFakeTest` 函数缺少对弱测试命令的检测模式（ls/cat/true/exit 0），`isInlineCommand` 正则也未覆盖这些命令，导致 `manual:ls src/` 等弱测试会落入 evidence 文件路径而非弱测试检测路径，产生不明确的错误信息。同时，`curl localhost` 这类在 CI 环境必然失败的命令也未被拦截。

---

### 关键教训

1. **curl-to-localhost 正则必须用行首锚定（^）**
   - 初始用 `/\bcurl\b.*\b(localhost|127\.0\.0\.1)\b/` 会误拦截 `curl-to-localhost`（标记字符串本身）以及 node -e 内含 curl localhost 的字符串
   - 正确方案：`/^\s*curl\s+(?:-[a-zA-Z\-]+\s+)*(?:localhost|127\.0\.0\.1)/` 只匹配顶层 curl 命令

2. **bash-guard 对 Task Card Test 字段的过度匹配**
   - bash-guard 会扫描 DoD Test 字段内容，若包含 `|wc|` 等弱测试模式字符串会误拦截（即使是 node -e 字符串里的子字符串）
   - 解决：用唯一标记（`P2-weak-inline`、`curl-to-localhost`）作为 Test 验证目标，而非直接验证检测模式字符串

3. **code_review_gate seal 文件路径必须是 `.dev-gate-crg.<branch>`**
   - subagent 写入 `.dev-gate-code.<branch>` 但 bash-guard 要求 `.dev-gate-crg.<branch>`
   - 如果 subagent 写到错误路径，需手动创建正确路径的 seal 文件

4. **BEHAVIOR DoD Test 路径解析规则**
   - `check-dod-mapping.cjs` 用 projectRoot（worktree 根）解析 `tests/` 相对路径
   - 若测试文件在 `packages/engine/tests/ci/`，应用 `manual:node -e "..."` 格式调用导出函数，不能直接用 `tests/ci/` 路径
