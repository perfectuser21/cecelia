# Learning: zenithjoy OKR 派发冻结自救 — Plan B 种子脚本

**Branch**: cp-03191049-83a61e0c-435b-46f8-9afc-bcec0c
**Date**: 2026-03-19
**Task**: [Plan B] 创建 zenithjoy 9 个 Initiative — 自救执行

---

### 根本原因

zenithjoy 部门 3 个 KR（KR1 发布自动化/KR2 数据采集/KR3 内容生成）长期处于 `decomposing` 状态，多次 /decomp 拆解任务因 worktree 创建失败（`worktree_creation_failed: unknown error`）而中止，导致 0 个 Initiative 存在，Brain 派发系统无法输出任何 Task。

### 触发条件

- OKR Tick 将 KR 置为 `decomposing` 状态并创建拆解任务
- 拆解任务（executor → /decomp skill）在 worktree 初始化阶段连续失败（exit_code=1）
- 失败任务不自动回滚 KR 状态，KR 持续处于 `decomposing`
- `decomposing` 状态的 KR 不再触发新的拆解任务（幂等保护）
- 结果：系统进入死锁——KR 在分解中，但分解永远无法完成

### 解决方案（Plan B）

绕过 /decomp 工作流，直接通过幂等种子脚本向 `projects` 表插入 9 个 Initiative 记录：

```
KR1(发布自动化) → 3 个 Initiative（短视频/图文/其他平台）
KR2(数据采集)   → 3 个 Initiative（播放数据/粉丝数据/数据完整性）
KR3(内容生成)   → 3 个 Initiative（选题文案/素材组装/发布集成）
```

脚本特性：幂等（按 name+kr_id 去重），可重复执行不会产生重复记录。

### 下次预防

- [ ] OKR Tick 应在 KR `decomposing` 超过 N 次失败后自动回滚到 `ready` 状态，允许重新触发拆解
- [ ] /decomp worktree 创建失败应有明确的错误分类和自动重试机制
- [ ] Brain 应维护"拆解失败计数器"，超阈值时发送飞书告警
- [ ] 考虑为每个部门维护"最低 Initiative 保障"，低于阈值时自动触发补充

### DoD 测试格式教训

DoD 中的 `[BEHAVIOR]` 测试不能连接生产数据库（CI 环境无 DB），应改为：
- 检查脚本文件内容（`fs.readFileSync`）验证定义完整性
- 或使用 `tests/` 格式指向 mock 测试

直接写 `manual:node -e "const {Pool}=require('pg')..."` 在 CI 中会因无 DB 连接而失败。
