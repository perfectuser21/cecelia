# Learning: P0/P1 架构修复 — g.priority死循环+Rumination自循环+RCA auto-fix限制

**Branch**: cp-03240531-a2f3e8db-c3a9-458e-82f3-d317a3
**Date**: 2026-03-24

### 根本原因

1. **g.priority 列不存在**：`decision.js` 中 `goalsQuery` 的 UNION 派生表 `g` 来自 `key_results UNION objectives`，两者均无 `priority` 列，但 ORDER BY 引用了 `g.priority`，导致每次 tick 都报 DB 错误。

2. **Rumination 自循环**：丘脑 L0 的 `RUMINATION_RESULT` handler 直接调用 `create_task`，无任何去重逻辑。相同洞察每次反刍都重新创建 research 任务，任务队列无限膨胀。

3. **RCA auto-fix 死循环**：`auto-fix.js` 的 `dispatchToDevSkill` 无上限限制。失败的 auto-fix 任务本身也会触发 RCA，再次创建 auto-fix 任务，形成指数级增长。

4. **测试 mock 与新 guard 代码不兼容**：新增 `pool.query()` guard 后，现有测试的 pool mock 返回 `undefined`，解构赋值直接抛出 TypeError。需用 try-catch 降级而非修改已有 mock。

### 下次预防

- [ ] UNION 派生表的 ORDER BY 必须只引用 SELECT 列表中实际存在的列；写 SQL 时先验证 UNION 结果的列集合
- [ ] 凡 Brain 中 create_task 调用路径，必须评估是否需要去重（同名+非完结状态查询）
- [ ] auto-fix/RCA 等自动触发链必须设置 per-signature 上限，防止反馈死循环
- [ ] 新增 `pool.query()` 调用时，若函数已有 vitest mock 覆盖，需考虑 mock 降级策略（try-catch）或同步更新测试 mock
