# Learning: tick.js burst limiter 实现

## 功能概述

为 tick.js Step 7 派发循环加入 `MAX_NEW_DISPATCHES_PER_TICK = 2` burst limiter，防止队列积压后资源恢复时多个 agent 同时启动触发雪崩。

### 根本原因

系统在队列积压后资源恢复时，tick Step 7（7a + 7b 两段派发循环）会在单次 tick 内尝试填满所有可用 slot。agent 启动阶段资源消耗约 2-3x，多个 agent 同时启动时峰值叠加，触发资源雪崩（OOM / predictive gate 连锁触发）。虽然已有 `getRampedDispatchMax` 渐进限速和 predictive gate，但这两者只在"稳定爬升"和"预测压力"时生效，无法覆盖"已有 slot 且压力当前 OK 但启动峰值高"的场景。

### 下次预防

- [ ] 新增任何影响单次 tick 派发数量的逻辑时，先评估是否需要和 `MAX_NEW_DISPATCHES_PER_TICK` 协同
- [ ] 7a/7b 两个循环共用同一个 `newDispatchCount` 计数器，若将来拆分循环需注意计数器作用域
- [ ] 如需动态调整 burst limit（如根据 alertness 等级降低），在 `MAX_NEW_DISPATCHES_PER_TICK` 基础上乘以系数，不要直接修改常量

## 技术细节

- `MAX_NEW_DISPATCHES_PER_TICK = 2` 放在顶部流控常量区（与 `UNBLOCK_BATCH_LIMIT`/`QUARANTINE_RELEASE_LIMIT` 同区）
- `newDispatchCount` 在 7a 循环前初始化，7a/7b 两段循环共用
- burst limit 检查放在每次 `dispatchNextTask()` 调用**之前**，优先于 predictive gate 检查（顺序：burst → predictive gate → dispatch）
- `burstLimited = newDispatchCount >= MAX_NEW_DISPATCHES_PER_TICK` 在循环后计算，加入 summary 日志和返回对象 `dispatch.burst_limited`
- 测试用 `tick-burst-limiter.test.js` 通过文件内容 grep 验证代码路径（无需 mock DB），6 个测试全通
