# Learning: quota_exhausted 独立任务状态（migration 144）

## 问题

配额耗尽时任务被错误标记为 `failed`，计入 `failure_count`，触发 FAILURE_THRESHOLD 隔离机制，污染真实失败率。业务失败（代码 bug）和资源限制（配额耗尽）被混为一谈。

## 解决方案

引入 `quota_exhausted` 作为独立任务状态，与 `blocked`（限流/billing_cap）并列，不计入 `failure_count`，不触发隔离。

### 根本原因

系统缺少对"资源不足"这一类暂时性外部限制的专用状态，所有非成功结果都流向 `failed` 路径，导致失败计数语义混乱。

### 关键设计决策

- **migration 144**：`quota_exhausted_at TIMESTAMPTZ` + `chk_quota_exhausted_at` CHECK 约束（与 blocked 状态的 `chk_blocked_at_not_null` 模式一致）
- **routes.js**：在 `status === 'AI Quota Exhausted'` 时映射到 `quota_exhausted`，不进入 `else if (newStatus === 'failed')` 分支，`handleTaskFailure` 自然不被调用
- **DB UPDATE**：新增 `$11 isQuotaExhausted` 参数，原子写入 `quota_exhausted_at = NOW()`
- **单测模式**：SQL 匹配不能用 `'UPDATE tasks SET'`（模板字面量含换行），应用 `'UPDATE tasks'` 并排除 `decision_log`

### 下次预防

- [ ] routes.js 的 SQL 模板字面量包含换行，测试中匹配时不能假设 `SET` 紧跟 `tasks`
- [ ] 新增状态时，CHECK 约束参照 `137_blocked_status.sql` 的 `chk_` 命名模式
- [ ] 记住：`handleTaskFailure` 调用在 `else if (newStatus === 'failed')` 内，新状态天然不触发，无需修改 quarantine.js
