# Learning: fix(ci) L3 覆盖率门禁扩展 fix/refactor + L4 task-lifecycle E2E

**Branch**: cp-03230724-11ec26cc-cd80-495c-8e12-21adba
**Date**: 2026-03-23

### 根本原因

1. `check-changed-coverage.cjs` 只有 `isFeatPR` 判断，fix/refactor PR 的 Gate 3 在无覆盖率数据时静默跳过，等同于未检查。
2. L4 缺少任务生命周期全链路 E2E（创建→dispatch→完成），出现任务状态机 regression 时无法被 CI 捕获。
3. CI 配置改动的 PR 必须加 `[CONFIG]` 前缀标签（L1 规定），Engine devgate 脚本改动需同步 `feature-registry.yml`（L2 规定）。

### 下次预防

- [ ] 改动 `packages/engine/scripts/devgate/` 下的文件时，**同时**更新 `feature-registry.yml` changelog 条目
- [ ] `.github/workflows/` 变更的 PR 标题**必须**包含 `[CONFIG]` 或 `[INFRA]` 前缀
- [ ] 新增 CI 覆盖检查时验证：非 feat PR 是否也会被正确处理（不应仅 `isFeatPR` 分支）
- [ ] L4 E2E 脚本需要独立 Brain 端口（如 15222），避免与 goldenpath-check.sh 的端口冲突
