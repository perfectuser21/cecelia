# Learning: fix(ci): L3 覆盖率门禁扩展 fix/refactor + L4 task-lifecycle E2E

**分支**: cp-03230724-11ec26cc-cd80-495c-8e12-21adba
**日期**: 2026-03-23

### 根本原因

1. **L3 覆盖率门禁只对 feat PR 有效**：`check-changed-coverage.cjs` 的 Gate 1/2 通过 `isFeatPR()` 判断，fix/refactor PR 会跳过这两个门禁。Gate 3（变更行覆盖率）虽然对所有 PR 运行，但对 fix/refactor PR 在无覆盖率数据时会直接 skip 而非 fail。

2. **L4 缺少任务生命周期 E2E**：`goldenpath-check.sh` 只验证服务启动和基础 API，不测试任务从 queued → in_progress → completed 的完整状态流转。

3. **PATCH 响应 vs DB 实际状态不一致**：Brain PATCH 任务状态端点可能因 `emitEvent is not defined` 返回 `success:false`，但 DB 已实际更新。E2E 测试必须通过 GET 验证实际状态，不能依赖 PATCH 响应。

### 下次预防

- [ ] DoD 必须包含 `[BEHAVIOR]` 条目（CI DoD Verification Gate 强制要求）；纯 `[ARTIFACT]` 的 DoD 会被 CI 拒绝
- [ ] CI Config Audit 要求 PR title 含 `[CONFIG]` 标签（修改 .github/workflows/ 下的文件时）；在 PR 创建时就加好标签，避免历史 CI run 基于无标签的 title 失败
- [ ] 写 E2E 脚本时，区分"API 响应"和"DB 实际状态"——对有已知 bug 的端点，用 GET 再查一次验证真实状态
- [ ] `isFixOrRefactorPR` 函数对 Gate 3 的强制执行：当 fix/refactor PR 有覆盖率数据时，Gate 3 强制检查；无数据时打印警告（非 fail）
