# Learning: 内容飞轮 I3 — 发布后数据回收

## 根本原因

内容发布后缺少效果反馈环节：content_publish 任务完成后没有自动触发数据采集机制，各平台阅读/点赞/评论/分享等指标无法回流到 pipeline 系统，内容飞轮在"发布→反馈"这一环断裂。
设计选择：新增独立模块 `post-publish-data-collector.js` 与现有 publish-monitor.js 分离，通过 Brain 任务队列派发 platform_scraper 任务（而非直接调用 scraper 脚本），保持架构一致性。
集成方式：在 tick.js 的 fire-and-forget 区块追加 `schedulePostPublishCollection` 调用，不影响 tick 主流程，并在 routes.js 注册 `/pipelines/:id/stats` 路由供 API 消费。

## 下次预防

- [ ] 新功能模块应保持独立性，避免修改现有稳定模块（如 publish-monitor.js）
- [ ] migration 编号需要检查最新编号（本次为 207）避免冲突
- [ ] routes.js 的 DoD Test 检查的是 `packages/brain/src/routes.js`，而不是 server.js，新路由需要同时在两处注册以保持一致性
- [ ] branch-protect hook 要求存在 per-branch PRD 文件（`.prd-{branch}.md`），孤儿 pipeline 恢复时需要先创建
- [ ] branch-protect hook 要求 .dev-mode 文件有 `tasks_created: true` 才能写 packages/ 文件
