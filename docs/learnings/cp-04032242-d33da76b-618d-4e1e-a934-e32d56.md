# Learning: executor.preparePrompt 重构（复杂度 107 → 10）

## 根本原因

`preparePrompt` 函数在不断添加新 taskType 的过程中，采用 if-else 链逐个分支累积，导致单函数圈复杂度达到 107。每次加新 taskType 都直接在函数体内嵌 return，无任何抽象。

## 解法

将每种 taskType 的 prompt 逻辑提取为独立命名函数（`_prepareXxxPrompt`），主函数只保留：
1. decomposition 特殊分支（3 子场景）
2. sprint/harness 多 taskType 共享分支
3. routes dispatch table（其余 16 种 taskType）
4. 兜底 `_prepareDefaultPrompt`

主函数复杂度降至 ≤ 8，每个子函数复杂度均 ≤ 6。

## 下次预防

- [ ] 每当向 `preparePrompt` 添加新 taskType 时，直接在 routes 对象中加一行，不允许在主函数内直接嵌逻辑
- [ ] CI 复杂度扫描已配置阈值 10，会自动拦截退化
- [ ] 测试模式：executor 重依赖多模块时用静态源码分析（readFileSync + toContain），避免 uuid/pg 等 mock 地狱
