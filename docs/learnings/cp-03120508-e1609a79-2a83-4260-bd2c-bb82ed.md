# Learning: quarantine 路径 BILLING_CAP 全局熔断缺口修复

## 核心发现

billing_pause 全局熔断有**两条触发路径**，但只有一条完整：

| 路径 | 触发时机 | 是否调用 setBillingPause |
|------|---------|------------------------|
| routes.js quota_exhausted 回调 | task 执行完毕后回调 | ✅ 已调用 |
| quarantine.js BILLING_CAP 分类 | task 失败时 handleTaskFailure | ❌ 漏调用（修复前）|

**结果**：通过 quarantine 路径捕获 BILLING_CAP 时，只有当前任务被 blockTask 阻塞，全局 `_billingPause` 未设置，dispatcher 继续派发剩余任务直至全部撞墙（O(n) 开销）。

## 修复方式

在 quarantine.js BILLING_CAP 处理块，blockTask 之后动态 import setBillingPause：

```js
const { setBillingPause } = await import('./executor.js');
setBillingPause(resetTime.toISOString(), 'billing_cap');
```

与现有 blockTask 的动态 import 模式一致，不引入循环依赖（executor.js 不 import quarantine.js）。

## 测试陷阱

`&& echo found` 在 DoD Test 命令中会被 check-dod-mapping.cjs 识别为"假测试"并报错。
验证文件存在应改用 `grep -c '关键词' 文件路径`，CI 可以执行并返回非零数字。

## 设计规律

**双路径触发同一保护机制时，必须两路都触发**。billing_pause 是全局状态，任何检测到 BILLING_CAP 的代码路径都应及时设置，否则会有时间窗口暴露。
