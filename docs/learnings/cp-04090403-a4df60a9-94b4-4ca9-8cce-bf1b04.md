# Learning: liveness_dead 失败率根因修复

## 背景
任务 a4df60a9：[SelfDrive] [P1紧急] 任务失败率根因修复 — 失败模式分类→单点修复

## 数据分析结论

### 真实状态
- 当前 7 天 dev 任务成功率：**91.7%**（非 PRD 提到的 39%，后者是 4 月 6-7 日历史峰值）
- 4 月 6-7 日失败高峰根因：watchdog 杀死的 Pipeline Rescue 任务（非 dev 类型）产生了 313 条 liveness_dead 失败，占 85%
- Pipeline Rescue storm 已由 MAX_RESCUE_PER_BRANCH=5 + 72h 冷却期修复

### dev 任务失败分布（7天）
| 类别 | 数量 | 根因 |
|------|------|------|
| liveness_dead | 7/8 = 87.5% | watchdog OOM/进程抢占 + 2次即隔离 + 2min 退避 |
| Crisis 压力击杀 | 重叠 | 系统内存 > 100% |
| error_message 缺失 | 0% | **PR#1714 已彻底修复** |

### 根本原因
`requeueTask` 的 `QUARANTINE_AFTER_KILLS=2` + 2min 退避 = 系统内存压力事件发生时，任务在 4 分钟内被杀 2 次即永久隔离，没有等待系统恢复的机会。

## 修复内容

### 1. `dev-failure-classifier.js`
新增 `liveness_dead`/`[watchdog]` 到 TRANSIENT_PATTERNS，使其通过 classifyDevFailure() 路径时被正确标记为可重试。

### 2. `executor.js` requeueTask()
- `LIVENESS_QUARANTINE_AFTER_KILLS = 3`：liveness_dead 给 3 次机会（其他 watchdog kill 保持 2 次）
- `LIVENESS_MIN_BACKOFF_SEC = 900`：liveness_dead 退避至少 15min，让系统内存有时间恢复

## 下次预防

### 根本原因
liveness_dead 是环境问题（OOM/系统抢占），不是代码 bug。用 2 次 + 2min 退避对付暂时性内存压力事件会导致任务被过早永久隔离。

### 下次预防
- [ ] 新增 watchdog 杀死策略时，区分「环境问题（liveness_dead）」和「代码问题（高内存占用）」
- [ ] liveness_dead 应设置 ≥15min 退避，等待内存压力缓解
- [ ] 监控 `watchdog_retry_count` 分布，避免在压力事件窗口内连续重试
