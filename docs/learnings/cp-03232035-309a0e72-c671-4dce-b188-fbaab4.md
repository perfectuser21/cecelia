# Learning: OKR 写操作迁移 PR9

### 根本原因

PR1-PR8 已将所有 SELECT 读操作迁移到新 OKR 表，但 INSERT/UPDATE 写操作仍写旧 goals/projects 表，新表数据依赖 migration 181/182 触发器同步。本 PR 完成最终写路径切换，移除对触发器同步的依赖。

迁移过程中遇到的关键挑战：
1. **FK 约束阻塞**：tasks.project_id/goal_id、project_kr_links.*、project_repos.project_id 引用旧表，直接写新表 UUID 会触发约束失败 → Migration 184 DROP 这些约束
2. **缺少列**：okr_projects 没有 sequence_order/current_phase/time_budget_days → Migration 184 ADD COLUMN
3. **迁移注释触发 DoD regex**：注释如 `// 迁移：INSERT INTO goals →` 含有被 DoD 检查正则匹配的模式，导致静态检查误报 → 注释被清理或改写

### 下次预防

- [ ] 写迁移注释时避免直接包含被 DoD 检查的 SQL 关键词，改用"写入新表"等中性表述
- [ ] 迁移前检查所有 FK 约束依赖链，迁移 migration 中一并 DROP
- [ ] 新 OKR 表缺少的操作列（sequence_order 等）应在首次使用前的 migration 中添加
- [ ] DoD 必须包含至少 1 个 `[BEHAVIOR]` 运行时验证条目，不能全是 `[ARTIFACT]`
- [ ] Learning 文件须在第一次 push 前写好加入 commit
