## Cortex Insight → Task 自动闭合机制（2026-03-12）

## 根本原因

Cortex insight（category=cortex_insight）写入 learnings 表后 `applied=false` 永远不变，
没有任何机制把 insight 转化为可执行的 dev task，导致洞察积累但零行动（观察者惰性）。
同时 `recordLearnings` 只做 INSERT，不返回 id，无法在写入后立即关联后续操作。

## 下次预防

- [ ] 新增 insight 类记录时，评估是否需要配套"闭合机制"（action trigger）

## 关键设计决策

**为何不加 migration 列**：tasks 表已有 `payload JSONB`，将 `insight_learning_id` 存入 payload
即可实现关联和去重查询，避免 migration 带来的 selfcheck 版本更新和 CI 验证开销。

**循环依赖检查**：`cortex.js → actions.js` 单向，`actions.js` 不 import `cortex.js`，
无循环依赖风险，直接 static import 安全。

**信号检测粒度**：关键词覆盖中英文通用修复语义（bug/fix/修复/代码/重构等），
故意不做 LLM 二次分析——避免在 cortex 内部再调 LLM 引发递归和资源竞争。

## 测试覆盖

- `hasCodeFixSignal`: 6 个单元测试（有/无信号、中英文、空串）
- `maybeCreateInsightTask`: 4 个集成测试（正常创建+applied标记、无信号跳过、去重、异常静默）
- 全部 10 个测试通过
