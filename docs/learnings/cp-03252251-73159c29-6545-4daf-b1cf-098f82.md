# Learning: rumination probe 24h 窗口误判

## 根本原因

`probeRumination` 使用严格的 24h 窗口检查 `synthesis_archive.created_at`，
而系统存在两路写入机制：
1. `runRumination` — 处理 learnings 时随时写入
2. `runDailySynthesis`（合成调度器）— UTC 18:00 固定触发

当 `runRumination` 在 UTC 01:47 写入后，合成调度器在 18:00 检测到
`hasTodaySynthesis=true` 而跳过。下一天的 probe 在 UTC 01:48（恰好超过 24h）
运行时，24h 窗口已不包含最近的写入记录，导致 `24h_count=0 last_run=never` 误报。
每天约 16 小时（UTC 01:47 ~ 18:00）probe 持续误报，触发 self-drive 反复建任务。

### 下次预防

- [ ] probe 时间窗口设计时，必须考虑写入路径的最大间隔（此处约 40h）
- [ ] 双路写入同一表时，需检查 `ON CONFLICT/hasTodaySynthesis` 等防重复机制是否会阻止定期触发
- [ ] probe 的 `ok: false` 条件应区分"系统故障"（有积压但无产出）和"合理静默"（无积压）
- [ ] 添加 idle 状态判断可避免将正常静默误判为故障
