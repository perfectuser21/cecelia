## 概要

为 Brain tick loop 补全了 content-pipeline 子任务路由注册，并修复了父任务阻塞子任务派发的 bug。

### 根本原因

1. **LOCATION_MAP 缺失**：task-router.js 的 LOCATION_MAP 没有 content-research/generate/review/export 四种子任务类型。虽然 getTaskLocation() 有 fallback（DEFAULT_LOCATION='us'），但 getValidTaskTypes() 通过 Object.keys(LOCATION_MAP) 返回，导致这四种类型不在合法任务类型列表中。

2. **项目级序列化阻塞**：selectNextDispatchableTask 的 SQL 检查"同一 project 是否有 in_progress 任务"。content-pipeline 父任务在 orchestrate 时变为 in_progress，而子任务继承相同 project_id，导致子任务永远无法被派发。

### 修复方案

1. 在 LOCATION_MAP 末尾新增 5 条（含 content-pipeline 本身）
2. SQL 的 NOT EXISTS 子查询中加 `AND t2.task_type != 'content-pipeline'`，让父任务不阻塞子任务

### 下次预防

- [ ] 新增 task_type 时，必须同时检查：LOCATION_MAP、VALID_TASK_TYPES、SKILL_WHITELIST 三处是否齐全
- [ ] 创建父子任务关系（共享 project_id）时，需要检查 selectNextDispatchableTask 是否会因父任务 in_progress 阻塞子任务
- [ ] brain-register skill 应该在注册流程中提示检查 LOCATION_MAP
