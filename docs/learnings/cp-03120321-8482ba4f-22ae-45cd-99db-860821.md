## Learning: liveness DEAD 判定后改为 requeue

**PR**: #893
**分支**: cp-03120321-8482ba4f-22ae-45cd-99db-860821
**日期**: 2026-03-12

### 核心变更

`executor.js` `probeTaskLiveness` 在双次确认进程 DEAD 后，原来直接调用 `updateTaskStatus(failed)`，
永久失败任务。进程死亡通常是 OOM 或系统抢占，任务本身无问题。

改为调用 `requeueTask('liveness_dead', errorDetails)`，利用其内置 `QUARANTINE_AFTER_KILLS=2` 机制：
- 第 1 次 DEAD → requeueTask → status=queued（backoff 2min）
- 第 2 次 DEAD → requeueTask 内部超阈值 → status=quarantined

### requeueTask mock 链（测试要点）

`requeueTask(taskId, reason, evidence)` 内部执行 3 次 DB 查询：
1. `SELECT payload, task_type, project_id, title FROM tasks WHERE id=$1 AND status='in_progress'`
2. `UPDATE tasks SET status='queued', started_at=NULL, payload=...`
3. `SELECT id FROM learnings WHERE content_hash=$1 AND is_latest=true LIMIT 1`（dedup）

测试中需按此顺序 mock 这 3 次查询，否则 UPDATE 返回 undefined 导致 rowCount 报错。
