# Learning: 重组 TaskTypeConfigPage — ABCD 类别框架

**Branch**: cp-03222041-abcd-task-type-page
**Task**: 5f8b10ef-9378-4090-8ec3-80c24a60e2ce

### 根本原因

原页面按设备地点分组（A类开发执行/西安Codex机群/香港VPS等），没有体现任务的本质约束：
- 哪些任务是"双锁"（机器+模型都不能换）
- 哪些是"单锁机器"（需要本机上下文但模型可换）
- 哪些是"模型锁定不锁机器"（Codex 但地点灵活）
- 哪些完全不需要大模型

### 解决方案

引入 ABCD 四类框架，按约束维度而非地点分组：
- A类：锁机器（美国M4）+ 锁模型（Claude Code），目前只有 `dev`
- B类：锁机器（美国M4），模型不锁，需本机代码/worktree/Brain DB 上下文
- C类：Codex，不锁机器（美国M4 或 西安M4 均可），动态任务可编辑切换
- D类：香港VPS，纯脚本/轻量，无大模型

页面顶部用卡片展示四类框架约束说明，让用户一眼理解分类逻辑。

### 下次预防

- [ ] 新增任务类型时，先判断它属于哪个约束类别（ABCD），再决定路由
- [ ] A类一般不会增加（dev 的约束是最强的）
- [ ] C类动态任务如需增加，在 task_type_configs DB 表中添加，不需要改前端代码
- [ ] 注意区分"当前跑在哪"和"约束是锁在哪"——C类当前在西安M4但可切换
