# Learning: 重构 advanceContentPipeline（复杂度 33→8）

**分支**: cp-03220542-40920991-9049-4eb5-9c9b-527a45
**日期**: 2026-03-22

### 根本原因

`advanceContentPipeline` 函数将 4 个阶段的调度逻辑（research/generate/review/export）全部内联在一个函数中，导致：
- 4 个顺序 `if (task_type === ...)` 链每增加一分支 +1 复杂度
- review 分支内再嵌套 if-else（pass/fail） + 再嵌套重试判断 → 深度嵌套
- 内联三元运算符 `?.` 和 `||` 链每个都贡献复杂度点
- 最终圈复杂度累积到 33

### 修复方案

1. 提取 `_loadPipelineContext` 合并所有守卫检查和 DB 加载（把 4 处早返回合并为 1 个 null 返回）
2. 每个阶段提取独立处理函数：`_handleResearchComplete` / `_handleGenerateComplete` / `_handleReviewComplete` / `_handleExportComplete`
3. 引入 `STAGE_HANDLER_MAP` dispatch table 替代顺序 if 链
4. `advanceContentPipeline` 精简至 8 行（加载上下文 → dispatch → return）
