## 背景

PR #2745 散点 12 → 集中 3 处。PR #2746 探测失败 fail-closed。但 stop-dev.sh case 仍是 `not-dev|done) exit 0` 共用——`not-dev` 和 `done` 共享同一行 exit 0。Alex 反复字面要求："**一旦确认进入开发模式，只有一个 exit 0 = PR 真完成**"。

## 根本原因

字面 1 个 exit 0 ≠ 语义 1 个出口。case 共用是语义混叠：
- `not-dev` 路径走 exit 0：普通对话场景下放行
- `done` 路径走 exit 0：PR 真完成

虽然在开发模式中（.dev-mode + cp-* 分支）classify_session 不会产出 `not-dev`（不会触发这条 case），但**代码上读不出这个保证**——任何后续修改都可能让 `not-dev` 在开发模式中触发，悄悄回到 PR1 开就停的故障模式。

## 本次解法

v20.1.0 严格三态拓扑：
- `exit 0` = done（PR 真完成）— 字面唯一，且仅服务此语义
- `exit 99` = not-applicable（bypass / 主分支 / 无 .dev-mode）— 由 stop.sh 路由层识别 fall-through
- `exit 2` = blocked（中间状态 + 探测异常 fail-closed）

出口码与 status 一一对应。从 `exit 0` 那一行代码读到的语义就是"PR 真完成"，没有歧义。

CI 守护扩展校验（从 3 项扩到 7 项）：
- stop-dev.sh `exit 0 = 1` ✓
- stop-dev.sh `exit 99 = 1` ✓ 新增
- stop-dev.sh `exit 2 = 1` ✓ 新增
- hooks/stop-dev.sh 镜像同上 ✓
- devloop-check.sh `return 0 = 2` ✓
- devloop-check.sh `return 99 = 1` ✓ 新增

永久阻止三态崩塌回二态共用。
