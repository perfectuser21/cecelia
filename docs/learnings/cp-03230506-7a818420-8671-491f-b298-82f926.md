# Learning: OKR routes 层切新表——task-goals/task-projects 迁移

**Branch**: cp-03230506-7a818420-8671-491f-b298-82f926
**Date**: 2026-03-23

## 任务概述

将 `task-goals.js` 和 `task-projects.js` 从旧表（`goals`/`projects`）迁移到新 OKR 表（`objectives`/`key_results`/`okr_projects`），这是 OKR 体系统一 PR3。

### 根本原因

PR1 已完成数据迁移，新表中已有数据，但 API 路由层仍读旧表，导致前端拿到的数据和新 OKR 体系不一致。

### 关键设计决策

1. **UNION ALL 参数共享**：PostgreSQL 在 UNION ALL 中所有子查询共享同一套参数绑定（`$1`/`$2`），`objectives` 和 `key_results` 子查询中的 `status = $1` 都绑定到同一个 params[0]，无需重复传参。

2. **双表 fallback 策略**：GET /:id 和 PATCH /:id 均采用"先查/改 objectives，0 行再查/改 key_results"的策略，避免 SELECT-before-UPDATE 的额外一次查询（PATCH 快乐路径只需 1 次 DB call）。

3. **parent_id 语义映射**：旧 `goals.parent_id` 在新表中映射为 `key_results.objective_id`；当 GET / 带 `parent_id` 过滤时，只查 key_results（objectives 无父级）。

4. **okr_projects 无 type/workspace_id 字段**：移除了旧 projects 特有的过滤条件；kr_id 直接作为 FK 存储，无需 project_kr_links 子查询。

### 下次预防

- [ ] 迁移路由前先确认新表字段列表，避免在 WHERE 子句引用不存在的列
- [ ] UNION ALL 查询中两个 SELECT 的列数和类型必须严格对齐，否则 PostgreSQL 会报错
- [ ] Task Card 的 DoD 条目必须包含至少一个 `[BEHAVIOR]` 标签，否则 DoD Verification Gate 会失败（Stage 4 检查）
- [ ] 测试 mock 双表 fallback 时，需要为 404 用例 mock 两次 pool.query（`mockResolvedValueOnce` 链式调用），并断言 `toHaveBeenCalledTimes(2)`
