## Scanner 误判 Brain 内嵌/基础设施能力为孤岛（2026-03-21）

### 根本原因

capability-scanner.js 的检测逻辑只识别两类 active 证据：
1. cecelia_events 中的 source 事件（BRAIN_EMBEDDED_SOURCES）
2. run_events / tasks 中的 skill/table 使用记录

但有两类能力不走这两条路：
- `postgresql-database-service`：Brain 进程直接通过 db.js pool 使用 PostgreSQL，不会向 cecelia_events 写入 source 事件，也没有 related_skills。Brain 运行即 PostgreSQL 运行，但扫描器无感知。
- `notion-integration`：notion_sync_log 表有 13 条历史记录（10 条成功），但 capabilities 表中 key_tables 字段为空，扫描器无法发现这个表。

结果：孤岛 14 个，其中 2 个是误判，体检报告失真。

### 下次预防

- [ ] 新增 Brain 进程内嵌能力时，先确认其检测路径属于哪类（BRAIN_ALWAYS_ACTIVE / BRAIN_EMBEDDED_SOURCES / related_skills / key_tables），避免漏配。
- [ ] 新增 migration 时同步更新 capabilities 表的 key_tables 字段（在迁移脚本末尾加 UPDATE capabilities SET key_tables=... WHERE id=...）。
- [ ] BRAIN_ALWAYS_ACTIVE 的判断标准：只要 Brain 进程在运行该能力就在运行（DB pool、tick 内嵌模块等），此类能力不需要 cecelia_events 证据。
