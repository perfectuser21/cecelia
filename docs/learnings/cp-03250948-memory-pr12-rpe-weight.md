# Learning: 记忆系统 PR12 — RPE 闭环实现

## 任务概述

在 task-weight.js 中加入 RPE（奖赏预测误差）信号，让历史执行反馈影响任务调度优先级。

### 根本原因

PR5 实现的 computeRPE 和 rpe_signal 事件记录是孤岛——数据写入了但没有被读取用于任何决策。
系统记录了"实际奖赏 vs 期望奖赏"的差值，但这个差值从未被下游模块消费，也就无法形成闭环。
RPE 闭环缺失使得系统无法从历史经验（任务超预期/低于预期）中学习并调整调度行为，奖赏信号纯粹是观测数据而非决策输入。

### 设计决策

1. **同步/异步双版本**：保持 `calculateTaskWeight`（同步）不变，新增 `calculateTaskWeightAsync`（异步，支持 DB 查询 RPE）。这样现有调用方不受影响，新调用方可按需接入 RPE。

2. **安全降级原则**：无 RPE 数据（空数据库、DB 异常、无 task_type）→ `rpe_bonus = 0`，行为完全等同于同步版本。

3. **RPE 缩放因子**：`rpe_bonus = avg_rpe * 10`，上限 ±10。P0 和 P1 差距为 40 分，RPE 最大影响 ±10，不会颠覆 priority 的主导地位。

4. **查询窗口**：最近 20 条 `rpe_signal` 事件均值，不用时间范围（避免冷启动问题）。

### 下次预防

- [ ] 新信号系统（RPE/奖赏/惩罚）实现后，同一 PR 必须同时实现"消费端"（闭环），否则会产生孤岛数据
- [ ] 异步扩展同步函数时，采用"同步保持兼容 + 异步新增"模式，避免破坏现有调用方
- [ ] DB 查询必须有 try/catch 安全降级，永远不能让权重计算因 DB 异常而崩溃
