# Learning: DoD Test 字段升级 — 拦截 CI 不兼容命令

## 任务概要

为 `check-dod-mapping.cjs` 新增 `detectCiIncompatibleCommand()` 函数，检测并拦截在 L1 CI 中必然失败的 `manual:` 命令（curl localhost、psql、npm test），给出友好的替代建议。同时更新 `DOD-TEMPLATE.md` 模板，移除旧式 evidence ID 格式，改用 CI 兼容的可执行命令示例。

### 根本原因

`check-dod-mapping.cjs` 在 `GITHUB_ACTIONS=true` 时会实际执行 `manual:` 内联命令。L1 CI 环境无 PostgreSQL 服务、无 Brain API 服务、无完整 `node_modules`，导致 `curl localhost:5221/...`、`psql cecelia -c "..."`、`npm test` 等命令静默失败，且错误信息不清晰，难以排查。

已有的 `detectFakeTest()` 只拦截语义弱测试（echo/ls/cat/grep|wc-l），但不拦截"语义强但 CI 环境缺失服务"的命令。这是一个覆盖盲区。

### 解决方案

在 `validateTestRef()` 的内联命令路径中，在 `detectFakeTest()` 检查之后、`validateAssertionStrength()` 之后，增加 `detectCiIncompatibleCommand()` 函数调用。

阻断模式：
- `curl (localhost|127.0.0.1|0.0.0.0|$*URL|$*HOST)` → 提示用 `tests/` 自动化测试
- `psql ` → 提示用 `tests/` 集成测试
- `npm (test|run test)` / `npx vitest` → 提示用 `Test: tests/path/to/test.ts`

### 并行 PR 冲突处理

本 PR 开发过程中，PR #1682（`validateAssertionStrength` 函数）并发合并到 main，导致 rebase 时 `check-dod-mapping.cjs` 产生冲突。解决策略：两个函数互补（强度检查 + CI 兼容检查），均保留，在调用链中依次执行。
