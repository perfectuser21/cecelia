# Learning: dev pipeline 补三个硬锁单元

**分支**: cp-03220402-6dc7c198-b393-4337-bf30-7e2b1d
**日期**: 2026-03-22

## 任务背景

在 `packages/engine/skills/dev/steps/02-code.md` 补充三个强制单元，并扩展 `verify-step.sh` 对新条目类型的支持。

### 根本原因

1. **[PRESERVE] 缺失**：Stage 2 没有要求记录改动前的现有行为。开发者改动代码时，不知道"以前是怎样的"，无法判断是否引入了回归。行为快照条目类型 `[PRESERVE]` 的引入，让改动前的现有行为成为明文合约。

2. **TDD 先行缺失**：没有强制要求先写失败测试，开发者可能直接写实现，导致测试成为"验证自己写的逻辑"而非"独立的行为规格"。红灯→绿灯流程确保测试的独立有效性。

3. **`|| true` 导致 CI 镜像失效**：`check-learning.sh` 和 `check-dod-mapping.cjs` 两个关键检查因 `|| true` 在本地不报错，但 CI 仍然会失败。本地镜像失去阻断作用，浪费 CI 资源。

### 下次预防

- [ ] 每次修改 `02-code.md` 时，检查所有检查命令后是否有 `|| true`——一旦发现，立即改为 `|| { echo "..."; exit 1; }`
- [ ] 新增 DoD 条目类型（如 `[PRESERVE]`）时，同步更新 `verify-step.sh` 的条目解析 `elif` 分支，以及所有相关汇总输出文本
- [ ] `regression-contract.yaml` 版本格式是 `version: X.Y.Z`（无引号），sed 命令需去掉引号
- [ ] `check-version-sync.sh` 从根目录运行读的是根 `package.json`（Brain 版本），Engine CI 实际从 `packages/engine/` 运行——本地验证时需 `cd packages/engine` 再运行
- [ ] per-branch PRD 文件（`.prd-cp-*.md`）和 `tasks_created: true` 需在写代码前创建，branch-protect.sh 会阻止没有这两者的改动
