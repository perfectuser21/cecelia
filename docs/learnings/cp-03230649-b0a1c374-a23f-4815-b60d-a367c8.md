# Learning: 补充 INFRA 能力 DB 元数据，减少 capability-scanner 白名单依赖

**分支**: cp-03230649-b0a1c374-a23f-4815-b60d-a367c8
**日期**: 2026-03-23

## 根本原因

PR #1463 为 capability-scanner 添加了 `INFRA_DEPLOYED_CAPABILITIES` 白名单（9个能力），但这些能力在 `capabilities` 表中缺少 `related_skills`/`key_tables` 元数据：

1. `branch-protection-hooks/ci-devgate-quality/brain-deployment` 有间接证据（dev 任务执行即证明这些能力在运行），但 DB 中 `related_skills=NULL`
2. `cecelia-dashboard/zenithjoy-dashboard` 读取 tasks 表，但 DB 中 `key_tables=NULL`
3. 剩余 4 个（cloudflare/nas/tailscale/vpn）真的没有 DB 可查证据，白名单是唯一正确方案

这导致扫描器完全依赖 JS 硬编码集合，新增 INFRA 能力时必须同时改代码和 DB，缺一不可。

## 修复方案

Migration 182 补充 5 个能力的 DB 元数据：
- `branch-protection-hooks / ci-devgate-quality / brain-deployment` → `related_skills=['dev']`
- `cecelia-dashboard / zenithjoy-dashboard` → `key_tables=['tasks']`

这使这 5 个能力即使在白名单被移除时，仍可通过 `taskUsageMap['dev']`（任务表有 dev 任务）或 `tableCountCache['tasks']`（tasks 表有数据）自主检测为 active。

## 下次预防

- [ ] 测试白名单内的能力 ID 时，`evaluateCapability` 会在白名单处提前返回，无法走 DB 元数据路径。需用不在任何白名单中的占位 ID（如 `test-dev-skill-detection`）验证 DB 检测路径逻辑
- [ ] INFRA 能力有两层防护：白名单（代码层）+ DB 元数据（数据层），两层互为冗余
- [ ] DoD [BEHAVIOR] Test 字段用 `tests/brain/xxx` 格式时，需确认 `tests/brain/xxx.test.js` 文件实际存在
- [ ] migration 号创建前先 `ls migrations/ | sort | tail -3` 确认最高号，避免冲突
