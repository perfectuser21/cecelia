# Learning: 补充 INFRA 能力元数据，减少 capability-scanner 白名单依赖

**分支**: cp-03230649-b0a1c374-a23f-4815-b60d-a367c8
**日期**: 2026-03-23

## 根本原因

PR #1463 为 capability-scanner 添加了 `INFRA_DEPLOYED_CAPABILITIES` JS 白名单（9个基础设施能力），
但这些能力在 `capabilities` DB 表中仍然缺少 `related_skills`/`key_tables` 元数据，
导致系统完全依赖硬编码 JS Set 来识别它们。如果白名单遗漏某个能力，DB 中没有任何备用证据。

## 修正方案

**migration 182**：为 5 个有间接证据链的 INFRA 能力补充 DB 元数据：
- `branch-protection-hooks` → `related_skills=['dev']`（dev 任务执行 = hooks 运行）
- `ci-devgate-quality` → `related_skills=['dev']`（dev PR 流经 CI）
- `brain-deployment` → `related_skills=['dev']`（dev 合并触发部署）
- `cecelia-dashboard` → `key_tables=['tasks']`（Dashboard 读取 tasks 表）
- `zenithjoy-dashboard` → `key_tables=['tasks']`（同上）

剩余 4 个纯基础设施能力（cloudflare/nas/tailscale/vpn）无 DB 可查证据，保留白名单。

**selfcheck.js**：EXPECTED_SCHEMA_VERSION 181 → 182

**测试**：新增 3 个 migration-182 回归测试（16 total，全部通过）

## 下次预防

- [ ] 新增 INFRA 能力时，优先为其补充 related_skills/key_tables（哪怕是间接证据）
- [ ] 不要只依赖 JS 白名单：DB 元数据是 scanner 的第一道防线，白名单是后备
- [ ] INFRA 能力的 key_tables/related_skills 可以是"逻辑关联"而非"直接写入"
