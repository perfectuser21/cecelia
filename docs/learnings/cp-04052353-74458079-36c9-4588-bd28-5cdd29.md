# Learning: 全平台数据采集端点完整化 — 补齐缺失链路

## 任务背景

补齐 PR #1945 之后仍缺失的数据采集链路：content_analytics 数据仍为空（0 rows）。

## 根本原因

PR #1945 添加了写回端点和 platform_scrape.sh 支持，但没有：
1. 自动定时触发平台采集的机制（无 scheduler）
2. LLM 算力消耗的历史快照（account_usage_cache 只存当前值）
3. 算力消耗查询端点（周报无法包含 LLM 成本数据）

导致链路断裂：数据永远不会写入 content_analytics，周报和选题引擎无法获取有效数据。

### 根本原因

采集脚本存在但无触发器。platform_scrape.sh 需要在 CN 机器上运行，但 Brain tick 没有定期创建 platform_scraper 任务的逻辑。同理，account_usage_cache 的实时数据没有被持久化为历史快照。

### 下次预防

- [ ] 每次新增数据写入端点时，同步检查是否有对应的"触发器"（定时任务/调度器）
- [ ] 数据仓库表创建后，立即验证是否有实际数据写入（而不仅仅验证表结构存在）
- [ ] `content_analytics` 这类依赖外部爬虫的表，需要在 Brain tick 中添加定时调度逻辑
