# Learning: 数据闭环 KR 卡点 — payload 字段名不匹配 + migration 版本号冲突

## 根本原因

### 问题 1：pipeline_id vs parent_pipeline_id 字段名不一致
`post-publish-data-collector.js` 的 SQL 过滤条件要求 `payload->>'pipeline_id' IS NOT NULL`，但实际 `content_publish` 任务的 payload 使用的是 `parent_pipeline_id`（不是 `pipeline_id`）。导致所有发布任务（6 条）全部被过滤，0 个 `platform_scraper` 任务被派发，数据闭环 KR 进度始终为 0%。

### 问题 2：Migration 207 版本号冲突导致 pipeline_publish_stats 表未创建
Migration 207 最初写入的是 `brain_health_checks` 表，后来文件被重命名为 `207_pipeline_publish_stats.sql` 并替换内容，但 `schema_version` 中版本 207 已标记为"已应用"，导致新的 pipeline_publish_stats SQL 从未被执行。

## 修复

1. 修改 SQL 查询：`COALESCE(payload->>'pipeline_id', payload->>'parent_pipeline_id')` 并移除对 pipeline_id NOT NULL 的强制要求
2. 新建 `213_pipeline_publish_stats.sql` migration 并直接执行

## 下次预防

- [ ] 修改 content_publish 任务创建逻辑，统一使用 `pipeline_id` 字段（或同时写两个字段），避免此类不一致
- [ ] Migration 文件一旦写入版本号并应用，禁止重命名或修改内容，新需求必须新建 migration
- [ ] 定期检查 `schema_version` 记录与文件描述的对应关系
