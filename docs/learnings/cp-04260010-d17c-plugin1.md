# Learning: D1.7c-plugin1 — 4 scheduled job 拆 plugin

## 上下文

Brain v2 Phase D 阶段 1.7c。在 D1.7b 把 `executeTick` 抽到 `tick-runner.js` 后，进一步把 8 个 inline scheduled job 段落拆成独立 plugin。
plugin1 负责 4 个：dept-heartbeat / kr-progress-sync / heartbeat / goal-eval。
plugin2（PR #2626，已合）并行处理另外 4 个。

## 根本原因

D1.7b 把 1626 行 executeTick 整体搬家时**没有动 30+ scheduled job 调度调用**，导致 tick-runner.js 仍然是单巨函数（1843 行），包含 8 个语义独立的"巡检/触发"段落，每段都是「时间窗口 + 节流计时器 + try/catch + push action」的样板代码。耦合在一起难单测、难复用、难加新巡检。

D1.7c 拆 plugin 的时机来自 D 阶段重构计划：让每个巡检/触发独立可测试，tick-runner.js 退化为编排器。

## 实现要点

1. **plugin 接口标准化**：每个 plugin export `tick(now, tickState)` 或 `tick(now, tickState, opts)`。返回 `{ran, actions}` 让 caller 决定是否 push 进 actionsTaken。
2. **节流计时器收口在 plugin 内部**：plugin 自己读 `tickState.lastXxxTime` 和环境变量 INTERVAL_MS，elapsed < INTERVAL 时返回 ran:false。caller 不再判断时间窗口。
3. **行为对齐**：每段 inline 原本有微妙的时间戳更新规则
   - kr-progress-sync / goal-eval：**先 mark 后 try**（inner error 也不立即重试）
   - heartbeat：**仅成功后更新**（inner error 时下次 tick 立即重试）
   两种语义被显式单测断言锁定，避免改写漂移。
4. **dept-heartbeat 守护搬家**：原 `isConsciousnessEnabled()` 守护在 tick-runner.js 内联；plugin 化后必须把守护**搬到 plugin tick 内部**，否则 caller 直接调用会绕过守护。
5. **dynamic import**：plugin tick() 内部用 `await import('./xxx.js')` 而不是顶层 import，避免引入循环依赖（heartbeat-inspector 等模块本身已被 tick-runner.js 导入链覆盖）。tick 周期 2min，import cache 后 cost 可忽略。
6. **harness-pipeline.test.ts 同步**：原断言 `tick-runner.js: triggerDeptHeartbeats 被 isConsciousnessEnabled() 门控` 必须同步到新位置（守护现在在 dept-heartbeat.js plugin tick 内）。

## 下次预防

- [ ] 拆 plugin 时**一定要先列出原 inline 段的隐式行为**（时间戳更新条件、错误处理边界、守护位置），写到 PRD 风险段，再写测试锁定。否则一句 `tickState.lastXxxTime = Date.now()` 的位置漂移就会引发"应该立即重试但延迟 30 分钟"的隐藏 bug。
- [ ] **守护逻辑搬家时必须扫所有源码自省测试**：源码字符串测试会查 `tick-runner.js` 含某 keyword，搬走后断言会失败。本次 `harness-pipeline.test.ts` 提前抓到，但 grep 全仓搜 `tick-runner.js` 字面 + 关键 keyword 是必要步骤。
- [ ] 并行 plugin1/plugin2 改同文件：双方约定**改不同段落** + 谁先 push 谁直接合。后合的 rebase main 解 imports section conflict（其他段落零冲突）。本次 plugin2 先合，plugin1 rebase 顺利。
- [ ] plugin tick 接口**禁止扩展为业务参数**：保持 `(now, tickState, opts?)` 三参，opts 仅用于测试依赖注入（如 pool）。否则不同 plugin 签名漂移会破坏 tick-runner.js 编排器的统一调用。
- [ ] **DoD `manual:node -e` 中 `require()` 调 ES module 包**：Node 22+ 才稳定支持 `require(esm)`。CI 用 Node 20 LTS 时要切到 dynamic import：`(async()=>{const x=await import(...);...})()`。本次本地 Node 25 跑通，但 CI 上需注意。
