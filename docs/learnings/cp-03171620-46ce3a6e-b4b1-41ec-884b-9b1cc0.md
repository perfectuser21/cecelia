## Task Card Test 命令 CI 兼容性（2026-03-17）

### 根本原因

Task Card 的 `Test:` 字段在本地通过了 `check-dod-mapping.cjs`，但在 CI 环境连续失败3次：
1. **第1次**：`.task-cp-03171620-...md` 未提交到 git，CI 落回到 `.dod.md`（另一任务的DoD），找到15个无Test的条目
2. **第2次**：`[BEHAVIOR]` 用了 `psql`，但 CI runner 无 PostgreSQL；`[GATE]` 用了 `/Users/administrator/...` 绝对路径，在 CI 不存在
3. **第3次**：`[GATE]` 改为 `node packages/engine/scripts/devgate/check-dod-mapping.cjs`，导致 CI 中 DoD 执行校验器调用自身 → 递归 → 30秒超时

### 下次预防

- [ ] Task Card 必须在第一次 push 时就 commit（不能有 "1 uncommitted change" 警告）
- [ ] `[BEHAVIOR]` Test 命令不能用 `psql`（CI 无 DB）；使用 `node -e "..."` 验证文件内容
- [ ] `[GATE]` Test 命令不能用绝对路径，不能用 `check-dod-mapping.cjs`（会递归超时）；用 `node -e "..."` 验证关键文件属性
- [ ] 本地写 Test 命令时先想"CI runner 有这个工具/路径吗？"——psql/绝对路径/本地服务都不行

## 诊断任务核心发现（可复用）

### 根本原因

worktree_creation_failed 分析的方法论：
- 当多个任务同时 quarantined 且错误为 `worktree_creation_failed: unknown error` 时，`unknown error` 意味着 cecelia-run 中的 `$wt_stderr_log` 临时文件在 error 捕获前已消失（竞态条件）
- 真正的 worktree 失败原因未被记录——这是一个日志盲区

### 下次预防

- [ ] 诊断 worktree_creation_failed 时，查看 `cecelia-run` 脚本 line 431 逻辑，理解 "unknown error" = stderr 临时文件丢失
- [ ] 批量 quarantine（相同秒内多个任务）= 系统事件（Brain 重启/批量派发），不是随机失败
- [ ] worktree 上限（MAX_WORKTREES=8）是积压的潜在根因，超过时任何新任务都无法 worktree 创建
