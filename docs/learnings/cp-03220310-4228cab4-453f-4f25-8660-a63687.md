# Learning: 重构 cecelia-routes 降低函数复杂度

## 根本原因

ComplexityScanner 误报 `mapStatus`（第38行）复杂度26，实为扫描器边界检测 bug（已由 PR #1353 修复）。
真正超阈值的函数：
- `inferStepProgress`：复杂度 15（多层嵌套条件 + 重复的 parseStepIndex 逻辑）
- `buildCheckpoints`：复杂度 13（map 内嵌多分支判断）

## 解决方案

提取辅助函数降低复杂度：
1. `parseStepIndex(stepStr, total)` — 统一处理 current_step 字段解析
2. `inferProgressForFailed/inferProgressForRunning` → 后被合并为更简洁的早返回形式
3. `getCheckpointStatus(status, stepNum, currentStepIdx)` — 提取 checkpoint 状态判断逻辑

结果：inferStepProgress 15→9，buildCheckpoints 13→5，全部测试通过。

## 下次预防

- [ ] 当 ComplexityScanner 报告某函数复杂度异常高（>20）时，先用扫描器逻辑手动验证，再创建任务
- [ ] 重构前确认：是否存在其他真实超阈值函数（同文件）
- [ ] 提取辅助函数时保持函数名语义清晰（体现职责，而非状态 A/B）
