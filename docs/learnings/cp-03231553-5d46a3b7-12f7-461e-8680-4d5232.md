# Learning: 重构高复杂度函数 analyzeDeep（CC=55 → ~8）

## 根本原因

`analyzeDeep` 函数承担了过多职责：
1. 构建 context 对象
2. 4 次独立 DB/语义检索（每次都有 try/catch）
3. LLM 调用 + 验证 + 去重熔断 + 副作用记录

每个 try/catch 块 + 条件分支都贡献 CC，216 行 7 层 try/catch 导致 CC=55。

## 解决方案

Extract Method 模式：按"职责边界"而非"代码长度"拆分。
- 每个 DB/检索操作 → 独立 `_enrich*` 函数（self-contained，内含 try/catch）
- LLM 响应处理 → `_processLLMResponse`（完整 try/catch 链）
- 主函数变为纯 orchestration

## 下次预防

- [ ] 函数超过 50 行时检查是否有多个 try/catch 块 → 考虑 extract method
- [ ] 圈复杂度阈值 10，超过时优先用"按职责提取"而非"按长度切割"
- [ ] branch-protect hook 需要 per-branch PRD + `tasks_created: true` 才允许写代码
- [ ] spec_review subagent 审查时需给完整 DoD 内容（不能用 `...` 简写）

## 关键经验

重构时 **行为等价性检查清单**：
1. 函数签名不变（`async function analyzeDeep(event, thalamusDecision = null)`）
2. 执行顺序不变（breaker → history → status → learnings → analyses → selfModel → rca_params → crossTask → LLM）
3. try/catch 语义不变（每个子函数内部 catch，不向上抛出，主流程继续）
4. 所有副作用保留（logCortexDecision / recordLearnings / storeAbsorptionPolicy / recordLLMError）
