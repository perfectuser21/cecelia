# Learning: Content Export NAS 上传 + export_path 回写

**分支**：cp-03301726-aadb4330-9d49-4edf-b7e5-d3132f
**日期**：2026-03-31

---

### 根本原因

content-export 阶段执行完后，文件只保存在本地 `/perfect21/zenithjoy/content-output/{keyword}/`，没有：
1. 上传到 NAS（`/volume1/workspace/vault/zenithjoy-creator/content/`）
2. 将 NAS 路径写回 pipeline payload 的 `export_path` 字段
3. `/api/brain/pipelines/:id/output` 返回 `export_path`
4. 前端 PipelineOutputPage 展示导出路径

### 修复方案

三处代码变更，形成完整的数据流链路：

1. **executors.js `executeExport()`**：manifest 生成后，rsync 到 NAS，返回值加 `export_path`
2. **orchestrator.js `_executeStageTask()`**：新增 `execResult.export_path` 分支，将 export_path 写入 task.payload（jsonb merge）
3. **orchestrator.js `_handleExportComplete()`**：从 `task.payload.export_path` 读出，再 jsonb merge 到 pipeline.payload
4. **routes/content-pipeline.js `/output`**：从 `pipeline.payload.export_path` 读取并返回
5. **zenithjoy PipelineOutputPage.tsx**（独立仓库）：添加导出路径面板渲染

### 下次预防

- [ ] NAS 相关常量（NAS_USER/NAS_IP/NAS_BASE）建议改为环境变量覆盖模式，提升可配置性
- [ ] `payload = payload || $2::jsonb` 在 payload 为 null 时可能问题，建议统一改为 `COALESCE(payload, '{}') || $2::jsonb`
- [ ] 跨仓库功能（cecelia 后端 + zenithjoy 前端）要在同一次任务中一起完成，避免 DoD 里的前端测试路径在 CI 上不可达

### 关键经验

**前端在独立仓库时的 DoD 处理**：

当功能需要修改不同仓库的文件时（如 cecelia brain + zenithjoy dashboard），cecelia PR 的 DoD 不能引用 zenithjoy 的绝对路径（CI ubuntu 上无此路径），应改为：
- 验证 Brain API 返回所需字段（路由代码在 cecelia diff 中）
- zenithjoy 前端修改在 zenithjoy 仓库单独 commit
