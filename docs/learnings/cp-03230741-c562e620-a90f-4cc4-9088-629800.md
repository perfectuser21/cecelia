# Learning: CI 覆盖率门禁扩展 + L4 task-lifecycle E2E

**Branch**: cp-03230741-c562e620-a90f-4cc4-9088-629800
**Task**: CI 覆盖率门禁扩展 + L4 task-lifecycle E2E
**Date**: 2026-03-23

## 变更摘要

1. `ci-l3-code.yml` 的 `coverage-delta` job 条件从 `commit_type == 'feat'` 扩展至 `contains(fromJson('["feat","fix","refactor"]'), ...)`
2. 新增 `packages/quality/src/task-lifecycle.test.ts`：纯状态机 E2E 测试，CI-safe，无需真实 Brain

### 根本原因

- **覆盖率门禁漏洞**：bug 修复 PR 绕过了覆盖率检查，导致修复引入的变更行可能没有测试覆盖
- **GoldenPath 覆盖不足**：L4 GoldenPath 只验证 Brain 启动 + 空 tick，不覆盖任务生命周期状态机

### 下次预防

- [ ] CI job 的 `if` 条件扩展多值时，统一用 `contains(fromJson([...]))` 模式，不用多个 `||`
- [ ] task-lifecycle 测试设计原则：纯状态机类（无 DB/API 依赖），覆盖正向路径 + 非法转换 + 终态保护
- [ ] `vitest.config.ts` 的 exclude 列表是 macOS/Linux 兼容性的关键防线，添加测试时检查 `stat -c %a` 用法
- [ ] PR title 含 `.github/workflows/**` 变更时必须加 `[CONFIG]` 标签，不要等 L1 失败后再改

## 关键决策

**为什么选 `contains(fromJson([...]))` 而不是多个 `||`**：GitHub Actions `if` 条件中，`contains()` + `fromJson()` 组合更易扩展，未来新增类型只需改数组，不需要修改比较逻辑。

**为什么新建 `packages/quality/src/` 而不是 `packages/quality/tests/`**：task-lifecycle 是单元测试级别的状态机验证，`src/` 目录符合"生产代码层测试"的语义，与 `tests/` 的集成测试区分清晰。

**macOS 不兼容的测试排除**：`pr-gate.test.ts` 和 `branch-protect.test.ts` 使用 Linux-only 的 `stat -c %a`，本次通过在 `vitest.config.ts` 中 exclude 解决（本地开发用 macOS，CI 用 Linux 跑集成测试）。
