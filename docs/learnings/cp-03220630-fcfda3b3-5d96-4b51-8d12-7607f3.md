# Learning: 重构 chat-action-dispatcher 高复杂度函数

Branch: cp-03220630-fcfda3b3-5d96-4b51-8d12-7607f3
Date: 2026-03-22

### 根本原因

`executeAction` 和 `executeViaIntentPipeline` 函数圈复杂度各约 13，超过阈值 10。
原因是业务逻辑（DB 查询、数据处理）和分发控制（switch/if-chain）混在一个函数体内，导致嵌套层级深、分支多。

### 解决方案

沿用 PR #1386 的分发器模式：
1. **`executeAction`**：将 4 个 case 逻辑提取为 `execCreateTask`、`execCreateLearning`、`execQueryStatus`、`execQueryGoals` 命名子函数；用 `ACTION_EXEC_HANDLERS` 分发表替换 switch 语句；`executeAction` 本身降至复杂度 ~4。
2. **`executeViaIntentPipeline`**：将意图处理提取为 `pipelineQueryStatus`、`pipelineCreateGoal`、`pipelineTaskAction`、`pipelineProjectAction`；用 `PIPELINE_HANDLERS` 分发表替换 if-chain；`executeViaIntentPipeline` 本身降至复杂度 ~5。

### 下次预防

- [ ] 新增动作类型时，直接在分发表（`ACTION_EXEC_HANDLERS`/`PIPELINE_HANDLERS`）加入口，不在 switch/if 里扩展
- [ ] 每个 handler 函数保持单一职责，若超过 20 行则考虑继续拆分
- [ ] 复杂度扫描阈值 10 是硬性门禁，写完新函数后本地粗估分支数
