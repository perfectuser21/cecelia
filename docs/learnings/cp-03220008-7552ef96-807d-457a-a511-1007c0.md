## 3 个核心洞察

### 洞察 1：系统存在多个"只靠人工记忆"的校验点，需自动化

25 条知识中，有大量规则属于"人要记住去做 X"类型：
- 修改 docker-compose 要同步检查 .env（条目 13、21）
- 新增 task_type 要检查 skill 文件存在（条目 25）
- 进入 worktree 要检查 .dod.md 归属（条目 23）

**洞察**：这些规则被反复写入 Learning，说明它们被反复违反。根本原因是**没有自动化校验机制**，只靠文档约束。解决方向：将这些规则转为 CI 检查脚本或 Hook，不依赖记忆。

### 洞察 2：Brain API 离线容错是系统性盲点

条目 14（fetchXxxSafe 包装）在多个脚本中未实施。CI 环境没有 Brain，任何直接调用 localhost:5221 的脚本都会在 CI 中 crash，而不是优雅降级。这是一个**跨文件的系统性问题**，需要一次性扫描修复，而非逐个 PR 修补。

### 洞察 3：DoD Test 命令合规性是持续性债务

条目 3、4、18、20 均涉及 DoD Test 命令的写法规范。每次 PR 都可能引入不合规的 Test 命令（grep、绝对路径、import 外部依赖等），而当前只在 CI 才能发现。**根本原因**：没有本地即时反馈机制。Stage 1 CI 镜像检查（check-dod-mapping.cjs）虽然已有，但覆盖规则不够全面。

---

### 根本原因

25 条积压知识均为 `learning_rule`，来源是历次 PR 产生的经验总结。这些经验**未被转化为自动化约束**，导致相同问题被反复踩。根本原因是"知识生产"（Learning 文件）和"知识落地"（代码/脚本/Hook）之间缺乏系统性转化机制。
