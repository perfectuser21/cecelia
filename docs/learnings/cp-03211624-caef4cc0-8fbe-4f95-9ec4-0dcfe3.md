# Learning: 激活 content_publish 多平台路由

**分支**: cp-03211624-caef4cc0-8fbe-4f95-9ec4-0dcfe3
**日期**: 2026-03-21

## 根本原因

`content_publish` 任务类型在 `task-router.js` 中完全缺席：
- 不在 `VALID_TASK_TYPES` → `detectRoutingFailure` 标记为 `invalid_task_type`
- 不在 `LOCATION_MAP` → `getTaskLocation('content_publish')` 返回默认 `'us'`
- 结果：所有发布任务被错误路由到 US 本机，而非西安 Codex Bridge

此问题导致前序任务（PR #1307 创建的 content_publish 任务队列）全部无法触达执行环境，
即使 executor.js 中已有正确的 platform→publisher skill 映射。

**相关历史失败**（exit code 137 = OOM）：三次任务因范围过大被 kill，本次缩小到仅 2 文件 49 行变更。

## 下次预防

- [ ] 新建任何 task_type 时，必须同时更新 task-router.js 三处：`VALID_TASK_TYPES` / `LOCATION_MAP` / `SKILL_WHITELIST`
- [ ] executor.js 的动态路由（如 content_publish 平台分发）不替代 task-router.js 的位置路由，两者独立
- [ ] OOM 失败（exit 137）→ 下一次执行必须拆分 scope，只改核心路由修复，不做大改造
- [ ] 新增统计 API 时，用 SQL `ROUND(..., 1)` 返回 numeric 类型，前端需要 `Number()` 转换
