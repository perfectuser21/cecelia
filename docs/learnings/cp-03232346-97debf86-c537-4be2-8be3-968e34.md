### 根本原因

测试 fixture 的两个问题同源于 OKR 双写触发器迁移（migration 181/182）：

1. **类型语义错误**：`createKR` 用 `type='area_okr'`，触发器把它同步到 `objectives`（OKR 层），而非 `key_results`（KR 层）。正确类型是 `'area_kr'`。

2. **隐式级联依赖**：`afterEach` 只清理旧表（`goals`/`projects`），依赖触发器级联删除新 OKR 表记录。但触发器用 `EXCEPTION WHEN OTHERS` 静默捕获异常——级联失败时新表记录残留，导致测试间数据污染（难以复现的 flaky test）。

3. **`key_results` 无 `domain` 列**：无法直接 INSERT `key_results`，必须通过 `goals` 的触发器同步，`domain` 字段由 `goals` 保留作为 SSOT。

### 下次预防

- [ ] 凡新增触发器双写迁移后，必须同步检查所有相关测试的 `afterEach`，确保显式清理新表（不依赖触发器级联）
- [ ] 新 OKR 表 `afterEach` 清理顺序：子表先于父表（`key_results` → `goals`，`okr_projects` → `projects`），避免 FK 约束冲突
- [ ] `type='area_*'` 值的触发器路由必须在代码评审时核查：`area_okr` → objectives，`area_kr` → key_results
