### 根本原因

Codex 审查 Skill（cto-review / code-quality / prd-audit）只通过 Brain API 回调传递 PASS/FAIL，
结果存在 Brain DB 中，无法离线查看或跨会话审计。需要在功能分支上留下持久化 Evidence 文件，
使审查过程可追溯。

---

### 下次预防

- [ ] 新增 Codex 类审查 Skill 时，默认模板中包含 Evidence 写入 + git push 章节
- [ ] `code_quality_review` 和 `prd_coverage_audit` task type 在 Brain DB 的 check 约束中已缺失（本次发现），需在后续 Brain PR 中补加迁移
- [ ] Evidence 文件写入逻辑中，`sha256sum` 在 macOS 需要用 `shasum -a 256`（Linux 可用 `sha256sum`），已用 `2>/dev/null || echo 'unknown'` 兜底

---

### 关键决策

1. **格式设计**：Evidence 文件记录 reviews[] 数组而非单次覆盖，支持增量追加（三审查依次完成）
2. **overall_decision 状态机**：PENDING → PASS（所有 3 条都 PASS/WARN）或 FAIL（任一 FAIL）
3. **不修改 devloop-check.sh**：Evidence 文件只是附加可审计层，现有 Brain 回调机制不变
4. **git push 是必须的**：Evidence 文件必须 push 到功能分支，才能在 PR 审查时可见
