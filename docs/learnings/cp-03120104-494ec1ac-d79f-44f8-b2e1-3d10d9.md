# Learning: Cortex Insight → Action 自动闭合机制

## 做了什么

实现了 cortex_insight 自动转化为 dev task 的闭合机制：
- 新增 `insight-to-task.js`：关键词检测（≥2 个命中）→ 自动创建 dev task
- 修改 `cortex.js` `recordLearnings`：INSERT 成功后 fire-and-forget 调用 `triggerInsightTask`
- migration 150：`tasks.source_learning_id UUID`（insight→task 溯源追踪）
- 去重 + `applied=true` 自动标记

## 踩的坑

1. **DEFINITION.md 有两处 schema_version 需要同步**（line ~386 和 ~676），只改一处会让 L2 facts-check 通过但 CI 仍 FAIL（因为另一处）。每次升 schema_version 必须双处同步。

2. **PRD 成功标准 ≥2 条**：L1 PRD Success Criteria Gate 要求至少 2 条可验证的成功标准，只写"任务完成"会被拦截。

3. **Learning 文件必须在第一次 push 前写好**：L1 Learning Format Gate 在 PR push 时检查 `docs/learnings/<branch>.md`，未提前写入会失败。本次是在 CI 失败后补写的，正确做法是 PR push 之前就写好。

## 架构决策

- **不使用 LLM 分析**：关键词检测避免热路径引入 LLM 延迟，≥2 关键词命中即触发，误判率低。
- **fire-and-forget 模式**：`triggerInsightTask` 通过 `.catch()` 包装，不影响 `recordLearnings` 主流程。
- **source_learning_id 独立列**：比 payload JSONB 更清晰，支持索引查询去重，是正确的数据建模选择。

## 下次记住

- schema_version 升级 = selfcheck.js + DEFINITION.md 两处（用 `grep -n "149" DEFINITION.md` 找两处）
- Learning 文件在 push 前写，不是 CI 失败后补
- PRD `## 成功标准` 至少写 2 条具体可验证的标准
