# Learning: 高复杂度函数提取子函数重构模式

### 根本原因

`buildLearningContext` 将评分、分组、格式化三种职责混入一个函数，导致圈复杂度达 21。核心问题是 `scored.map()` 内含 6 个决策点（task_type 判断、domain 判断、age 三元、category 判断等），与主流程耦合在同一作用域。

### 解决方案

提取 4 个单职责子函数：
- `getAgeBonus(createdAt)` — 年龄加分（2 个决策点）
- `scoreRow(row, taskType, domain, titleWords)` — 完整评分（6 个决策点）
- `groupByType(learnings)` — 按类型分组（2 个决策点）
- `formatSection(ltype, records)` — 格式化单节（1 个决策点）

主函数 `buildLearningContext` 降至 3 个决策点（try/catch + slow query warn + empty check）。

### 下次预防

- [ ] 函数超 30 行 且包含多个 `map/filter` 回调时，提前识别为"多职责"信号，在写代码时就拆分
- [ ] scoring 逻辑（多个 if + 累加）天然适合独立函数，即使只用一次也应提取
- [ ] 三元嵌套 `a ? x : b ? y : z` 出现即提取为命名函数
