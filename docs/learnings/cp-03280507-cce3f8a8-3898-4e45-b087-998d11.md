# Learning: 修复 KR 进度可见性 — 两套字段断连

**分支**: cp-03280507-cce3f8a8-3898-4e45-b087-998d11
**日期**: 2026-03-28

## 问题描述
KR 进度在仪表板全部显示 0%，但 kr-verifier 已正确采集到数据（管家闭环 = 30%）。

---

### 根本原因

`key_results` 表有两套字段：
- `progress`（integer）：由 `kr-verifier.js` 写入，代表真实百分比
- `current_value`（numeric）：由 OKR CRUD API 管理

`/api/brain/okr/current` 路由只读 `current_value / target_value`，完全忽略 `progress` 字段。
kr-verifier 计算正确但"写错了地方"——写入 `progress`，而 API 读另一个字段。

同时，`decision.js` 的 `compareGoalProgress()` 使用 `GROUP BY g.id`，但 SELECT 列表包含 `g.title, g.status, g.progress, g.created_at, g.priority`（均未在 GROUP BY 中），导致 PostgreSQL 报错，每次 tick 的 decision engine 静默失败。

---

### 下次预防

- [ ] 新建数据字段时，确认 API 层读取与写入层使用的是**同一个字段**
- [ ] 有多套字段表达同一概念时（如 `progress` 和 `current_value`），必须文档化哪个是 SSOT
- [ ] GROUP BY 聚合查询：PostgreSQL strict mode 要求所有非聚合 SELECT 列都在 GROUP BY 中，MySQL 允许省略导致编写习惯错误
- [ ] decision engine 错误被标记为 `non-fatal`（`catch → continue`），不会中断 tick，但会导致决策信号静默丢失；应定期检查 decisions 表的 `status=failed` 条目
