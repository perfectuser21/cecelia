# Learning: content-type YAML 固定 notebook_id + orchestrator 传参 + export 清空 notebook

## 任务摘要

将 solo-company-case 内容类型绑定到固定 NotebookLM notebook，orchestrator 创建 research 子任务时透传 notebook_id，export 完成后自动清空 notebook 源以供下次 pipeline 复用。

## 变更文件

- `packages/brain/src/content-types/solo-company-case.yaml`
- `packages/brain/src/content-pipeline-orchestrator.js`
- `packages/brain/src/content-pipeline-executors.js`

### 根本原因

`_startOnePipeline` 加载 `typeConfig` 仅用于校验存在性，但 `typeConfig` 中的 `notebook_id` 字段没有被传入 research 子任务的 payload，导致 research 阶段无法获取正确的 notebook。export 完成后也没有清理 notebook 内的源，造成下次 pipeline 运行时读到上次残留的旧内容。

### 下次预防

- [ ] content-type YAML 新增字段后，检查是否需要在 orchestrator 或 executor 中透传该字段
- [ ] 新建 content-type 时，立即在 YAML 中填写 notebook_id（不留空字符串）
- [ ] executor 涉及外部资源（NotebookLM、NAS 等）时，考虑「用完即清理」的生命周期设计
