# Stop Hook v23 PR-2 — 心跳模型核心切换（2026-05-07）

## 背景

Stop Hook v23 三 PR 重构序列的 PR-2/3。PR-1 (#2823) 已合，建好心跳模型物理基础（dev-heartbeat-guardian.sh 守护进程、abort-dev.sh 手动中止、log_hook_decision 决策日志、launcher dry-run 契约测试）。

PR-2 把 stop-dev.sh 的决策模型从 v22 的"考证档案"（cwd 路由 + session_id 双通道 + ghost rm + mtime expire，209 行）切到 v23 的"心跳"（扫 lights/ 目录看 mtime，76 行）。engine-worktree 启动 guardian + 写灯文件，engine-ship 调 ship-finalize 关 guardian + 写 done-marker。

19 case 测试矩阵（4 个文件：decision / routing / engine-worktree-guardian / engine-ship-guardian）覆盖灯亮/熄/多流/跨 session/JSON 损坏 + 路由 + skill 集成。

## 根本原因

v22 的 200+ 行复杂度，全部源于"用一个档案模拟活动状态"——需要持续维护档案的"活性"：

- ghost session_id 过滤（远端 sync 来的没传 session_id）
- mtime 30 分钟自动 expire（防永久 stuck）
- cwd→branch 路由 + session_id 精确匹配双通道（多 worktree 隔离）
- verify_dev_complete 7 阶段状态机（PR/CI/merge/deploy/health/Learning/cleanup）

每个机制都是为了让"档案"和"现实"对齐，每次错位都是 bug 来源。半年累积修过 50+ 次。

v23 改用 **OS 级 mtime + 进程心跳** 替代软件状态机：

- guardian 进程活 = 灯文件 mtime 持续刷新 = hook block
- guardian 进程死 = 灯文件 mtime 老化 = hook release
- 所有权键 = 文件名前缀 sid_short（session_id 前 8 位）
- 跨 session 隔离 = 文件名 glob 自然分组

"档案是死的，进程是活的"——心跳是事实，没有歧义。
