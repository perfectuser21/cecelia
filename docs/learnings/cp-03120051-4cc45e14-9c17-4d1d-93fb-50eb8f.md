# Learning: Cecelia 架构巡检 2026-03-12

## 任务背景

arch_review 定期巡检任务，巡检三层大脑（脑干/丘脑/皮层）、Tick Loop、Executor、Planner。

## 关键发现

### 1. tick.js（L0 脑干）直接调用 cortex.js（L2 皮层）

**发现**：tick.js 有 2 处直接 `await import('./cortex.js')` 调用：
- `processCortexTask()`（第 686 行）：处理 `task_type=cortex_rca` 任务
- `check48hReport()`（第 2552 行）：定时触发 48h 系统简报

**设计意图与问题**：这两处调用有意识地绕过了 L1 丘脑的事件路由，直接触发 L2 LLM。从功能上看是合理的（cortex_rca 任务就是要调用 Cortex），但打破了 "L0 脑干不调用 LLM" 的架构原则。

**经验**：当 Brain 需要内部批处理 LLM 任务时（如 RCA、系统报告），正确做法是将其封装在一个独立的中间层，而不是在 tick.js 直接调用。建议创建 `cortex-scheduler.js` 隔离这类逻辑。

### 2. tick.js 文件体积过大（2790 行）

**发现**：tick.js 混合了 5+ 种职责：Tick Loop 驱动、任务派发、Cortex 批处理、超时自动失败、Codex 免疫等。

**经验**：每次往 tick.js 加新功能时，应评估是否创建新的子模块。遵循单文件 > 500 行 → 拆分的规范。

### 3. cortex.js 反向导入 thalamus.js 是合理的

**发现**：`cortex.js` 导入 `thalamus.js` 的 `ACTION_WHITELIST`，用于约束皮层的额外 actions。

**经验**：L2 受 L1 白名单约束（上层受下层约束）是合理的依赖方向，不属于越权。不要把"反向导入"和"越权调用"混淆。

### 4. 全局文档与代码实现的漂移

**发现**：全局 CLAUDE.md 仍记录 "L1 丘脑（Haiku）"，但代码和 DEFINITION.md 已使用 MiniMax M2.1。

**经验**：模型迁移后，需要同步更新全局文档（CLAUDE.md），不只是 DEFINITION.md。建议把模型变更作为 DoD 要求之一。

## 架构健康结论

综合评分 81/100，核心架构健康，主要问题是 tick.js 职责过多导致的结构债务积累。
