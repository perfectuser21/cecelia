# Learning: KR 进度诚实化 — /goals/audit 端点 + migration 模式

## 背景

goals.progress 字段长期被手动置为 100，但 initiative 实际完成率远低于此，导致 Brain planner 误判 KR 已完成。

## 关键决策

### 1. Express 路由顺序：/audit 必须在 /:id 之前注册

`router.get('/audit', ...)` 必须放在 `router.get('/:id', ...)` **之前**，否则 `:id` 会匹配 `audit` 字符串，导致 404。

### 2. migration 用 DO 块而非批量 UPDATE

用 DO 块逐个处理每个 KR，可以在同一事务中既更新 progress 又写入 memory_stream。相比单条 UPDATE，能精确记录每个 KR 的修正前后值。

### 3. DEFINITION.md schema_version 有两处需要同步

- 第 386 行：`| **schema_version** | 迁移版本追踪 | Schema 版本: N |`
- 第 676 行：`DB 版本 >= 'N'（>= 检查，向前兼容）`

L2 facts-check.mjs 从 DEFINITION.md 读取 `schema_version` 并与 selfcheck.js 的 `EXPECTED_SCHEMA_VERSION` 比较，两处都要更新。

### 4. actual_progress null 处理

当 KR 没有任何 initiative 时（`COUNT(p.id) = 0`），`CASE WHEN` 返回 NULL。前端/消费者需要显式处理 `null` 与 `0` 的区别：null = 无法计算（无 initiative），0 = 有 initiative 但全未完成。

## 根本原因

`goals.progress` 是静态字段，由手动操作或旧 migration 直接写入，没有任何机制确保它与 initiative 实际完成率保持一致。历史上 migration 143 把 `progress>=100 AND status=in_progress` 批量置为 completed，但没有校验 initiative 完成率。

## 下次预防

- 每次新增 KR goal 时，同步检查是否有对应 initiative 并设置 `kr_id`
- 新增批量修改 `goals.progress` 的 migration 时，必须先查询 initiative 完成率而非直接置 100
