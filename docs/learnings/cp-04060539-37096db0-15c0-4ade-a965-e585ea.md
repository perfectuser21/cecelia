# Learning: content-pipeline notebook_id 缺失导致 0 进度

**Branch**: cp-04060539-37096db0-15c0-4ade-a965-e585ea  
**Date**: 2026-04-06

### 根本原因

`_parsePipelineParams` 在 pipeline 无 `content_type` 时返回 `null`，导致：
1. orchestrator 的 `_startOnePipeline` 跳过 `typeConfig` 查询
2. 创建的 `content-research` 子任务 payload 中没有 `notebook_id`
3. `executeResearch` 检查 `notebookId` 为 falsy → 立即返回 `{ success: false }` → 整个 pipeline 失败
4. 内容生成 KR 因此始终维持 0% 进度

触发路径：每日 recurring task `content-pipeline` 创建时未设置 `content_type`（任务在改为 topic-selection-scheduler 前就已禁用，但留下了孤儿任务实例）。

同时发现 `executeResearch` 在 payload 无 `notebook_id` 时没有尝试从 `typeConfig` 查找（`typeConfig` 已加载但未被利用作为 fallback）。

### 修复内容

1. **`_parsePipelineParams`**（`content-pipeline-orchestrator.js`）:
   - `content_type: pipeline.payload?.content_type || null` → `|| 'solo-company-case'`
   - 确保无 content_type 的 pipeline 也能获取正确的 typeConfig 和 notebook_id

2. **`executeResearch`**（`content-pipeline-executors.js`）:
   - 先加载 `typeConfig`，再赋值 `notebookId`
   - `notebookId = task.payload?.notebook_id || typeConfig?.notebook_id`
   - belt-and-suspenders：即使 orchestrator 端有遗漏，executor 也能自愈

### 下次预防

- [ ] content-pipeline 任务创建时，必须在 payload 包含 `content_type` 或 `notebook_id`；orchestrator 和 executor 均应有防御性 fallback
- [ ] 新增 `content-pipeline-notebook-fallback.test.ts` 确保两个 fallback 逻辑有测试覆盖
- [ ] 禁用的 recurring task 创建的孤儿任务需要在禁用时同步 cancel，避免留下无 content_type 的孤儿 pipeline
