# Learning: 修复错误黑洞

### 根本原因

1. **exit_code=1 不触发重试**：`classifyDevFailure()` 接收 `result` 对象时忽略了 `result.failure_class` 字段，导致 `cecelia-run` 已分类的 `code_error` 被重新判断为 `UNKNOWN`，`retryable: false`，retry_count=0 的任务直接沉底。

2. **retry_count 不同步 DB**：dev 重试路径只更新 `payload.retry_count`，不更新 DB `tasks.retry_count` 字段，导致 `max_retries` 检查统计失效。

3. **进程日志静默丢失**：`result=null` 时 `error_message` 仅含 exit_code 和 stderr，Claude 进程的详细日志在 `/tmp/cecelia-{taskId}.log` 未被读取，形成调试黑洞。

### 下次预防

- [ ] `classifyDevFailure` 应优先信任 `result.failure_class`（已由 cecelia-run 分类），避免二次误判
- [ ] dev 重试 UPDATE SQL 必须同时更新 `tasks.retry_count` 字段
- [ ] result=null 场景下主动读取进程日志文件并附加到 error_message
- [ ] 新增兜底：`exitCode !== 0 && result === null` → 视为 code_error 可重试，防止 agent crash 导致的静默沉底
