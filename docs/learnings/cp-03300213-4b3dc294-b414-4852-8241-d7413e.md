### 根本原因

1. **DoD 测试路径错误**：Task Card 中的 `[GATE]` 条目写了 `tests/apps/api/features/content/pages/ContentTypeConfigPage.test.tsx`，但实际测试文件放在 `apps/api/features/content/pages/ContentTypeConfigPage.test.tsx`（无 `tests/` 前缀）。check-dod-mapping.cjs 只接受 `tests/`、`manual:`、`contract:` 开头的格式，`apps/` 路径被识别为"缺少 Test 字段"。

2. **Learning 文件缺失**：Stage 2 Code 阶段被 verify-step.sh 阻断（因 desire-system.test.js 预存在的 schema version 断言错误），导致 Learning 文件未按规范在第一次 push 前写好并加入 commit。

3. **Instruction Book 遗漏**：feat PR 必须同步更新 docs/instruction-book/features/，但未在 Stage 2 Code 阶段纳入改动范围，导致 CI L1 Instruction Book Gate 失败。

### 下次预防

- [ ] Task Card 中 `tests/` 前缀路径规则：`packages/` 下的测试用 `tests/packages/...`；`apps/` 下的测试直接用 `apps/...`，不加 `tests/` 前缀
- [ ] Learning 文件必须在第一次 push 前创建并 git add，不能等 CI 失败后再补
- [ ] verify-step.sh 被阻断时，先检查是否是预存在的测试失败（与本 PR 无关），若是则单独 commit 修复该测试后再运行 verify-step
- [ ] desire-system.test.js 的 EXPECTED_SCHEMA_VERSION 断言应在每次 schema 迁移后立即更新，防止积累技术债

### 实现亮点

- ContentTypeConfigPage 采用每行独立 `TypeRow` 组件，避免全局状态污染，每个 type 独立加载/保存
- Pipeline 展开使用懒加载（首次展开才 fetch stages），避免列表首屏请求过多
- notebook_id 从 DB config 自动填入表单，做到"零配置"用户体验
