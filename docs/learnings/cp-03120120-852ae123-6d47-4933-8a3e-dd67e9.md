### [2026-03-12] 验证 PR #874 billing pause 熔断端到端路径

**失败统计**：CI 失败 0 次，本地测试失败 2 次

**本地测试失败记录**：
- 失败 #1：`PATCH /tasks/:id` 设置 `queued→in_progress` 返回 409
  - 根本原因：PATCH endpoint 的 allowedTransitions 只允许 `pending→in_progress`，不支持 `queued→in_progress`
  - 修复方式：移除 PATCH 步骤，利用 billing pause 设置在 DB 事务外（调用 setBillingPause 不依赖 WHERE status='in_progress'）
  - 预防：下次调用 PATCH /tasks/:id 前先确认当前 status 是否在 allowedTransitions 里

- 失败 #2：`POST /tick` 返回 `{ skipped: true, reason: "already_running" }`
  - 根本原因：Brain 后台 tick loop 正在运行，手动触发 tick 被节流保护跳过
  - 修复方式：等待 tick_running=false 后再触发，同时添加 already_running 降级逻辑（改为验证 billing_pause API 状态）
  - 预防：任何依赖手动触发 tick 的验证，都要先检查 GET /api/brain/tick/status 的 tick_running 字段

**关键发现**：
- billing pause 是**内存级全局状态**（`_billingPause` 变量），由 `executor.js` 管理
- `setBillingPause` 调用在 execution-callback 的 DB 事务 COMMIT 之后，即使任务不在 in_progress 状态（DB 更新影响 0 行），billing pause 依然会被设置
- 这是合理设计：billing pause 是系统级熔断，不应依赖单个任务的状态

**影响程度**: Low（流程顺畅，两次本地失败均有清晰根因）

**预防措施**：
- 调用 PATCH /tasks/:id 前验证状态机转换是否合法（pending→in_progress，不是 queued→in_progress）
- 验证脚本需要考虑 tick already_running 场景，添加等待 + 降级逻辑
