# Learning: fix(executor): syncOrphanTasksOnStartup requeue

## 根本原因

Brain 重启时，`syncOrphanTasksOnStartup` 将所有无进程的 in_progress 任务一律 failed，未区分"Brain 主动中断"（process_disappeared/unknown）与"资源耗尽杀死"（oom_killed/killed_signal）。正常开发任务因 Brain 重启永久丢失。

## 解决方案

新增 `REQUEUE_REASONS = ['process_disappeared', 'unknown']` 常量，分两路处理：
- process_disappeared/unknown → status=queued（Brain 重启，可安全重试）
- oom_killed/oom_likely/killed_signal/timeout → status=failed + auto-learning（真实失败）

## 踩坑记录

### 1. Task Card Test 命令格式：ARTIFACT vs BEHAVIOR 规则不同

**问题**：DoD 中 `[BEHAVIOR]` 条目用了 `grep -c` 命令，CI 报 "BEHAVIOR 条目禁止使用弱测试"。

**规则**：
- `[ARTIFACT]` → `bash -c "grep ..."` 可以（静态文件检查）
- `[BEHAVIOR]` → 必须引用 vitest 测试或 curl 运行时验证
- 纯 `grep -c`（不加 bash）→ 任何类型都不接受（需要包含 node/npm/bash 等）

**下次预防**：
- [ ] ARTIFACT Test 用 `manual:bash -c "grep ..."`（不能裸 grep）
- [ ] BEHAVIOR Test 用 `manual:bash -c "cd pkg && npx vitest run ... -t 'pattern'"` 或测试文件路径

### 2. platform-utils.js 在测试中需要 mock

**问题**：`checkExitReason()` 调用 `platformGetDmesgInfo()`（来自 platform-utils.js），未 mock 时在测试环境行为不确定。需要 `vi.hoisted` + `vi.mock('../platform-utils.js', ...)` 控制 dmesg 返回值才能测试不同 reason 路径。

**下次预防**：
- [ ] 测试 reason 路由逻辑时，先 mock platform-utils.js 的 getDmesgInfo

### 3. worktree 中操作要用 worktree 绝对路径

**问题**：Edit 工具使用主仓库路径 (`/cecelia/packages/...`) 时被 branch-protect 拦截，因为主仓库当前在 main 分支。必须使用 worktree 绝对路径 (`/cecelia/.claude/worktrees/<id>/packages/...`)。

**下次预防**：
- [ ] 在 worktree 中工作时，所有文件操作用 worktree 绝对路径
