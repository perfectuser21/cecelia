# Learning: fix(ci): 修复 CI 审计发现的 5 个问题

## 根本原因

### P0：coverage-delta 依赖 brain-unit.result
`needs` 数组中包含 brain-unit，且 `if` 条件要求 `brain-unit.result == 'success'`。
当 brain-unit 因任何原因 skip 时（如 docs commit type），coverage-delta 也会 skip，feat+brain PR 可绕过覆盖率检查。

### P1：TEST_PATTERNS 缺 .mts 扩展
模式字符串未包含 `.test.mts` / `.spec.mts`，使用 TypeScript ESM（.mts）写的测试文件无法被识别。

### P1：[CONFIG] 豁免路径过宽
`^packages/engine/` 匹配了 packages/engine 下所有目录，包括 docs/。修改文档也能豁免 test coverage 检查。

### P2：engine-l2 版本检查缺 release: 豁免
chore/docs/test 已豁免版本检查，但 release: 类型 PR 未在豁免列表，导致版本已在 release 流程中 bump 后仍被再次要求。

### P2：L1 gate 未防御 changes 失败
l1-passed 直接读取 `needs.changes.outputs.engine`，若 changes job 本身失败，outputs 为空，条件 `!= 'true'` 匹配失败，engine-l1 / dev-health 检查被静默绕过（误报通过）。

## 下次预防

- [ ] 新增依赖 job 时，检查 `if` 条件是否会因被依赖 job skip 而连带 skip（这是 CI 漏洞常见来源）
- [ ] Gate job（l1-passed/l2-passed/l3-passed）开头必须先检查 `changes.result != 'failure'`
- [ ] TEST_PATTERNS 变更时，检查是否覆盖所有 TS/JS 变体（.ts/.mts/.cts/.js/.mjs/.cjs）
- [ ] 豁免路径使用 `(subdir1|subdir2)` 精确匹配，不用 `/` 结尾的宽泛前缀
- [ ] engine-l2 version check 豁免列表与 engine-l1 RCI checks 豁免列表保持一致（chore/docs/test/release）
