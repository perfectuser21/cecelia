# Learning: CI CTO 安全漏洞修复

## 背景

CTO 全面诊断发现 CI 体系存在 6 个可被利用的漏洞，允许代码绕过关键检查合并到 main。

## 根本原因

CI 检查存在"防守性 exit 0"模式：开发者为了减少噪音，在文件不存在时选择静默跳过而非失败。这导致安全假设被破坏——攻击者可以删除关键文件（DoD、required-dev-paths.yml）或使用特定标签（docs:、[CONFIG]）绕过整个检查层。

具体根因：
1. **docs/style 无条件跳过 L3**：未检查是否有实际代码文件变更
2. **[CONFIG] 无条件豁免测试覆盖**：未验证是否真的改了 CI/引擎文件
3. **DoD 文件缺失 exit 0**：应该是 CI 硬要求，不应静默通过
4. **基线无上限**：技术债可以无限累积，没有偿还压力
5. **required-dev-paths.yml 缺失 exit 0**：同 Fix 3，关键配置文件不应允许缺失
6. **skipped 状态不一致**：虽然 `!= "success"` 已能捕获 skipped，但代码意图不清晰

## 修复策略

- **硬失败原则**：关键检查的前置条件不满足 → 必须 exit 1，不允许 exit 0 静默跳过
- **白名单验证**：豁免标签（如 [CONFIG]、docs:）必须配合路径验证，确保实际符合豁免条件
- **上限保护**：允许技术债的地方必须有上限（MAX_BASELINE=5）
- **意图明确**：即使逻辑等效，也要写清楚 "skipped 视为失败" 的注释

## 下次预防

- [ ] 每次新增 "跳过检查" 逻辑时，必须思考：能否被滥用绕过 CI？
- [ ] 所有 exit 0 的 if 分支，检查是否有安全假设被破坏
- [ ] 豁免标签（[CONFIG]、docs:、style:）应该有路径白名单验证，而不是仅匹配标题
- [ ] 定期（每季度）做 CTO 级 CI 诊断，检查新增的绕过路径

## 影响

- Fix 1/2/3/4/5 均为 P0，可直接被利用
- Fix 6 为 P1，意图澄清
- 修复后此 PR 本身（纯 CI 文件变更，无新代码文件）仍可正常通过
