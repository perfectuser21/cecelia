### [2026-03-12] 验证 billing pause 熔断端到端路径（PR #874）

**失败统计**：CI 失败 0 次，本地测试失败 3 次

**本地测试失败记录**：
- 失败 #1：bash `((PASS_COUNT++))` 在初始值为 0 时返回退出码 1，触发 `set -e` 退出 → 改用 `PASS_COUNT=$((PASS_COUNT + 1))`
- 失败 #2：`curl -sf` 任务创建失败（exit 22）：重复任务标题触发 dedup 约束 → 标题加时间戳 `$(date +%s)` 保证唯一性
- 失败 #3：手动触发 tick（POST /api/brain/tick）返回 `already_running`，导致 dispatch reason 为空 → 改为轮询 tick/status 等待 tick 完成后再触发，并增加 dispatch_stats 的 DB 级验证备用路径

**关键发现**：
1. **execution-callback 要求 WHERE status='in_progress'**：测试任务不能直接从 queued 触发 quota_exhausted 回调，需要先通过 DB 直接 UPDATE 设为 in_progress（PATCH API 只允许 pending→in_progress 转换）
2. **tick 长时间运行问题**：Brain 当前有 7 个并发任务，每个 tick 需要等所有任务槽检查完毕，手动触发 tick 会遇到 `already_running` 状态，需要轮询等待
3. **dispatch_stats 存储在 working_memory**：key=`dispatch_stats`，通过 `jsonb_array_length(value_json->'events')` 可以计数验证 billing_pause 事件

**影响程度**: Low（所有功能验证通过，只是脚本调试问题）

**预防措施**：
- bash 中避免使用 `((var++))` with `set -e`，改用 `var=$((var + 1))`
- 验证脚本中所有 curl 都要用唯一标识符（时间戳/UUID）避免 dedup 冲突
- 涉及 tick 操作的验证脚本，务必先 poll tick/status 等待 tick_running=false
