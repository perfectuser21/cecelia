# Learning：v22 遗产清理 + FD 201 leak 修复

## 背景
Stop Hook v23 序列收尾 PR。原计划"删 dev-active + verify_dev_complete"因依赖关系不能直接删：
- `dev-mode-tool-guard.sh`（PreToolUse hook）仍用 dev-active 存在性判 /dev 流程
- `verify_dev_complete` 函数被 9 测试 + 4 docs 引用

跑 PR-3 worktree create 时**实测命中 FD 201 leak bug**：上次 PR-2.5 的 guardian 孤儿持 worktree-create.lock 导致 cmd_create 失败。

## 根本原因

### FD 201 leak
v23 PR-2 加 guardian fork 时未关 worktree-create.lock 的 FD 201。bash 的 `nohup ... &` 默认继承父进程所有 FD。cmd_create 退出 → 父进程 FD 201 关 → 但 guardian 子进程仍开着 → flock 锁仍持有。下次 cmd_create flock -w 5 失败。

修：`nohup bash X </dev/null >/dev/null 2>&1 201>&- &` —— 显式 `201>&-` 关闭子进程的 FD 201。

### 删除困难
v22 状态机文件被多个独立模块共享是历史遗留：
- stop-dev.sh（v23 已迁出）
- dev-mode-tool-guard.sh（仍依赖）
- 9 个测试文件（仍引用）
- 4 个 docs（仍提到）

强行清理会炸 PreToolUse 拦截 + CI 测试。结论：**deprecate 优于删除**，留迁移作独立 PR。
