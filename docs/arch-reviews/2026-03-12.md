## 二、三层大脑边界检测

### 2.1 正常边界（预期）

```
L0 脑干（tick.js）
  → 调用 L1 丘脑（thalamus.processEvent）    ✅ 正常路径
  → L1 丘脑 level=2 → 升级到 L2 皮层（cortex.analyzeDeep）  ✅ 正常路径
```

### 2.2 发现异常：L0 直接调用 L2 ⚠️

**问题 1：processCortexTask()（tick.js 第 686 行）**

```javascript
// tick.js L0 脑干中：
const { performRCA } = await import('./cortex.js');  // 直接调用 L2
```

`processCortexTask()` 是 tick.js 内嵌的 Cortex 任务处理函数，直接 dynamic import cortex.js 的 `performRCA`，**绕过了 L1 丘脑的事件路由**。这是一种特殊任务类型（`task_type=cortex_rca`），由 L0 直接触发 L2 处理。

**问题 2：48h 系统报告生成（tick.js 第 2552 行）**

```javascript
// tick.js 定时报告触发：
const { generateSystemReport } = await import('./cortex.js');  // 直接调用 L2
```

定时 48h 报告也直接从 tick.js 调用 cortex.generateSystemReport()，同样绕过 L1。

**影响评估**：
- 这两处调用都是**有意识的绕过**（代码注释清晰标注了调用目的）
- 并非错误，但打破了 "L0 不调 LLM" 的设计原则的精神
- `processCortexTask` 是将 cortex_rca 类任务作为"内部批处理任务"执行，有合理性
- 建议：将 processCortexTask() 和 48h 报告逻辑迁移到独立的 `brain-services/cortex-scheduler.js`，保持 tick.js 仅做调度分发

### 2.3 cortex.js 反向导入 thalamus.js

```javascript
// cortex.js 第 22 行：
import { ACTION_WHITELIST, validateDecision, recordLLMError, recordTokenUsage } from './thalamus.js';
```

L2 皮层导入 L1 丘脑的 ACTION_WHITELIST 和验证函数。这是**共享工具函数**模式，不属于越权调用。L2 使用 L1 的白名单来约束自己的额外 actions，是合理的依赖方向（上层受下层约束）。✅

**建议拆分方向（非强制，供参考）**：
- `tick-scheduler.js` — 纯调度分发（dispatchNextTask）
- `tick-maintenance.js` — 维护任务（autoFail, releaseBlocked, Codex 免疫）
- `cortex-scheduler.js` — Cortex 批处理 + 48h 报告
