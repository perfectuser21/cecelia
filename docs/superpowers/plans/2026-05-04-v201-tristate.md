# Learning — Stop Hook v20.1.0 严格三态出口

分支：cp-0504172144-v201-tristate
日期：2026-05-04
Brain Task：47130342-f7ad-4ccb-8045-96e5f492724e
前置 PR：#2745（c1f1e65ed）+ #2746（194bf334a）

## 背景

PR #2745 散点 12 → 集中 3 处。PR #2746 探测失败 fail-closed。但 stop-dev.sh case 仍是 `not-dev|done) exit 0` 共用——`not-dev` 和 `done` 共享同一行 exit 0，代码读不出"哪条路径才是 PR 真完成"。Alex 反复字面要求："**一旦确认进入开发模式，只有一个 exit 0 = PR 真完成**"。

## 根本原因

字面 1 个 exit 0 ≠ 语义 1 个出口。case 共用是语义混叠：
- `not-dev` 路径走 exit 0：普通对话场景下放行
- `done` 路径走 exit 0：PR 真完成

虽然在开发模式中（.dev-mode + cp-* 分支）classify_session 不会产出 `not-dev`（不会触发这条 case），但**代码上读不出这个保证**——任何后续修改都可能让 `not-dev` 在开发模式中触发，悄悄回到 PR1 开就停的故障模式。

## 本次解法

v20.1.0 严格三态拓扑：
- `exit 0` = done（PR 真完成）— 字面唯一，且仅服务此语义
- `exit 99` = not-applicable（bypass / 主分支 / 无 .dev-mode）— 由 stop.sh 路由层识别 fall-through
- `exit 2` = blocked（中间状态 + 探测异常 fail-closed）

出口码与 status 一一对应。从 `exit 0` 那一行代码读到的语义就是"PR 真完成"，没有歧义。CI 守护扩展校验 `exit 99 = 1` / `return 99 = 1`，永久阻止三态崩塌回二态共用。

## 下次预防

- [ ] 任何 stop hook 协议级改动必须出口码与 status 字段一一对应（不允许多 status 共用同一 exit code）
- [ ] CI 守护必须**字面**校验每个 exit code 出现次数 = 1（防止散点回归）
- [ ] custom exit code（如 99）仅在内部路由用，对外（Claude Code）保持 0/2 二态协议
- [ ] 字面"1 个 exit 0"和语义"1 个出口"是不同要求，必须问清楚再实施
