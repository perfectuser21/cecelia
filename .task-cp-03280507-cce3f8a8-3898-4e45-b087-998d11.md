# Task Card: 修复 KR 进度可见性 — 接通 kr-verifier → OKR 显示链路

## 需求（What & Why）

**功能描述**：KR 进度在 Brain 中全部显示为 0%，但系统实际每天完成 94 个任务、kr-verifier 已采集到真实指标（管家闭环 progress=30%）。问题是两套字段断连：verifier 写 `key_results.progress`，但 API 读 `current_value/target_value`。

**背景**：已诊断出两个根本问题：
1. `/api/brain/okr/current` 路由用 `current_value/target_value` 计算 progress_pct，但 kr-verifier 更新的是 `progress` 字段（不更新 current_value）
2. `decision.js` 的 `compareGoalProgress()` SQL：`GROUP BY g.id` 缺少其他非聚合列，导致每次 decision engine tick 报错

**不做什么**：不重构 kr-verifier 机制，不修改 kr_verifiers 表查询逻辑，不批量更新任务的 goal_id

## 诊断数据
- 管家闭环 KR: `progress=30`（kr-verifier 已正确写入）, `current_value=0.00`（API 读此字段显示 0%）
- 所有 95 个已完成任务中，`goal_id=null`（后台队列任务脱离 OKR 框架）
- decision engine 每次 tick 报错：`column "g.title" must appear in the GROUP BY clause`

## 实现方案

**要改的文件**：
1. `packages/brain/src/routes/okr-hierarchy.js`（`/current` 路由 SQL）
2. `packages/brain/src/decision.js`（`compareGoalProgress()` SQL GROUP BY）

**Scope 锚定**：
- 修改 1：`/current` 路由中 SELECT 的 progress_pct CASE 表达式，改为 `COALESCE(progress, CASE WHEN target_value > 0 THEN ROUND(...) ELSE 0 END) AS progress_pct`
- 修改 2：`GROUP BY g.id` → `GROUP BY g.id, g.title, g.status, g.progress, g.created_at, g.priority`
