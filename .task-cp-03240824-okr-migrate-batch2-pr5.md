# Task Card: OKR 业务代码迁移 PR8 — SIMPLE 组（14个文件）

## 需求（What & Why）

**功能描述**：
将14个"SIMPLE"文件（每文件1-2处 goals/projects 旧表引用）中的 SELECT 查询迁移到新 OKR 七层表（visions / objectives / key_results / okr_projects / okr_scopes / okr_initiatives）。

**背景**：
- migration 177-182 已建好新 OKR 表体系
- migration 181 双向同步触发器保证旧表写入自动同步到新表，UUID 完全相同
- PR6（decomposition-checker/okr-tick/project-activator/project-compare）和 PR7（planner/self-drive/similarity/validate-okr）已处理 MEDIUM 复杂文件
- 本 PR 处理引用次数少（1-2次）的14个 SIMPLE 文件
- 目标：让所有业务代码读取新表，彻底不依赖 goals/projects，最终在 PR9 删除旧表

**14个文件及其迁移策略**：

1. `actions.js` — `SELECT type FROM goals WHERE id = $1`
   → goals.type 路由：vision→visions / area_okr→objectives / area_kr|global_kr→key_results
   → 迁移：先查 key_results，不存在则查 objectives，不存在则查 visions（或 COALESCE 三表）

2. `chat-action-dispatcher.js` — `SELECT title, status, progress FROM goals`
   → 显示 OKR 状态，goals 含多种类型，迁移：UNION ALL key_results + objectives ORDER BY created_at LIMIT 5

3. `code-review-trigger.js` — `SELECT type FROM projects WHERE id = $1`
   → projects.type 路由：project→okr_projects / scope→okr_scopes / initiative→okr_initiatives
   → 迁移：查 okr_initiatives（最常见），不存在则查 okr_projects；或用 UNION ALL

4. `daily-review-scheduler.js` — `SELECT DISTINCT repo_path FROM projects`
   → okr_projects + okr_initiatives 都有 metadata->>'repo_path'（若存在）
   → 迁移：UNION ALL 查 okr_projects/okr_scopes/okr_initiatives 的 metadata->>'repo_path'

5. `decision.js` — `FROM goals g LEFT JOIN tasks t ON t.goal_id = g.id`
   → tasks 表仍有 goal_id 字段（UUID 相同），直接改 FROM 为 key_results/objectives UNION ALL
   → 迁移：UNION ALL objectives+key_results，保留 LEFT JOIN tasks t ON t.goal_id = g.id（UUID相同）

6. `dept-heartbeat.js` — `SELECT id FROM goals WHERE metadata->>'dept' = $1`
   → 新表 key_results/objectives 有 metadata 字段
   → 迁移：UNION ALL key_results + objectives WHERE metadata->>'dept' = $1

7. `goal-evaluator.js` — `SELECT id,title,status,priority,progress FROM goals WHERE status = 'in_progress'`
   → key_results 有 progress/priority/status 字段
   → 迁移：FROM key_results WHERE status IN ('active', 'in_progress')

8. `intent.js` — `SELECT id, name FROM projects WHERE LOWER(name) LIKE $1`
   → okr_projects 无 name 字段，改为 title
   → 迁移：UNION ALL okr_projects + okr_scopes + okr_initiatives WHERE LOWER(title) LIKE $1

9. `orchestrator-chat.js` — `SELECT status, count(*) FROM goals GROUP BY status`
   → 统计 OKR 状态分布，迁移：UNION ALL key_results + objectives，GROUP BY status

10. `pr-callback-handler.js` — `JOIN projects p ON p.id = pkl.project_id`
    → project_kr_links 的 project_id UUID 与新表相同
    → 迁移：JOIN okr_projects p ON p.id = pkl.project_id（okr_projects 最常见）

11. `desire/perception.js` — `SELECT title,progress,status FROM goals WHERE status = 'in_progress'`
    → 迁移：FROM key_results WHERE status IN ('active', 'in_progress')

12. `routes/actions.js` — `SELECT id,title,status,priority,progress FROM goals WHERE status NOT IN ('completed','cancelled')`
    → 迁移：UNION ALL objectives + key_results WHERE status NOT IN ('completed','cancelled')

13. `routes/ops.js` — `FROM goal_evaluations ge JOIN goals g ON g.id = ge.goal_id`
    → goal_evaluations 表的 goal_id 指向 key_results/objectives（UUID相同）
    → 迁移：JOIN key_results g ON g.id = ge.goal_id（或 UNION ALL 两表）

14. `area-scheduler.js` — 两处：
    (a) `LEFT JOIN goals g ON t.goal_id = g.id`（任务调度统计）
    (b) `SELECT id FROM goals WHERE domain = $1 AND status IN ('ready','in_progress','decomposing')`
    → (a) 改：LEFT JOIN key_results g ON t.goal_id = g.id（key_results 有 area_id）
    → (b) 改：SELECT id FROM key_results WHERE area_id = (SELECT id FROM areas WHERE name = $1) AND status IN ('active','in_progress')

**迁移规则（全局）**：
- `goals.id == objectives/key_results.id`（UUID 相同）
- `projects.id == okr_projects/okr_scopes/okr_initiatives.id`（UUID 相同）
- `status: 'in_progress'` → `status IN ('active', 'in_progress')`（新表含 active 状态）
- `projects.name` → `title`（新表字段名改变）
- `goals.domain` → `key_results.area_id` / `objectives.area_id`（domain 改为关联 area_id）
- `project_kr_links` 保留不变（IDs 相同，仍可 JOIN 新表）

**不做什么**：
- 不修改 INSERT/UPDATE 操作（触发器自动双写）
- 不删除旧表（PR9 才做）
- 不修改 project_kr_links 表
- 不改测试文件中的 SQL（测试字符串除外）
- 不处理 MEDIUM/COMPLEX 文件（已在 PR6/PR7 中处理）
