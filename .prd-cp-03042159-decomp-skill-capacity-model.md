# PRD: 更新 decomp SKILL 反映 24/7×10 slot 产能模型

## 背景

Cecelia 当前运行在 24/7 模式，具备以下真实产能：
- VPS: 8 CPU / 15GB RAM → PHYSICAL_CAPACITY = 12 slots，AUTO_DISPATCH = 10 slots
- 3个 Claude Max 账号 + 1个 MiniMax 包月
- 每 PR 耗时 15-30 min（含CI）
- 月产能 ≥ 10,000 PR

现有 decomp SKILL.md 存在以下问题：
1. `Project → Initiative` 规则"3-8 个"严重低估了实际产能（应为 40-70 个/周）
2. Initiative 定义未明确"系统性"要求（应 ≥4 个 PR）
3. Initiative 完成后无 project_plan 循环机制（飞轮断裂）
4. 缺少 Phase 3: project_plan 模式

## 需求

### 1. 核心设计原则 — 新增"产能模型"小节

描述系统真实产能，指导拆解粒度：
- PHYSICAL_CAPACITY = 12 slots
- AUTO_DISPATCH = 10 slots
- 日产能：~336 PR/day（10 slot × 48 PR/slot × 70% 效率）
- 月产能：~10,000 PR/month
- 每 Initiative：4-8 PR × 20-25min avg = 1.5-3.5 小时
- 每 Project：每周可容纳 40-70 个 Initiative

### 2. Stage 2 Project→Initiative 规则更新

将"3-8 个"改为：
- **初始拆解**：10 个核心 Initiative（给 project_plan 启动用）
- **动态扩展**：project_plan 每次完成一个 Initiative 后再规划下一个
- 总量目标：每个 Project 40-70 个 Initiative（自动动态生成，无需一次性规划完）

### 3. Initiative 定义更新（模板 F）

新增：
- `min_tasks: 4`（最少 4 个 PR，才算系统性 Initiative）
- 系统性要求：必须是完整子系统或独立模块，不能是单一函数改动
- 正面例子："统一发布任务调度循环"（含 API + DB + 测试 + 监控）
- 反面例子："修复一个 bug"（应放入已有 Initiative 的 PR 中）

### 4. Phase 2 Step 3a — 添加 project_plan 触发

Initiative 标记 completed 后：
1. 检查 parent Project 是否完成（基于 Project 验收标准）
2. 若 Project 未完成 → 创建 task_type='project_plan' 任务（触发下一 Initiative 规划）
3. 若 Project 已完成 → 标记 Project completed，结束

### 5. 新增 Phase 3: project_plan 模式

触发：task_type = 'project_plan'
步骤：
1. 读取 Project + 所有已完成 Initiative
2. 评估 Project 是否达到验收标准
3. 若未完成 → 规划下一个 Initiative（写入 projects 表，type='initiative'）
4. 每次只创建 1 个新 Initiative（保持动态性）
5. Initiative 创建后，Brain 自动派发 initiative_plan 任务开始执行

### 6. HARD RULE 表格更新

新增第三行：
- Phase 3（project_plan）→ 写入 `projects` 表（type='initiative'）→ 绝不写 `tasks`
