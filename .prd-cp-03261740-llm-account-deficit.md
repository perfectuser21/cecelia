# PRD: 改进 LLM 账号选择算法（进度对齐 Deficit 排序）

## 背景

5个 Codex 账号和 3个 Claude Code 账号各自拥有 7天滚动配额窗口，窗口重置时间各不相同。
当前算法只看当前使用率（最低优先），无法有效利用即将到期的配额，造成浪费。

## 目标

改进账号选择排序逻辑：按照"进度对齐（deficit）"原则，优先使用落后于时间目标的账号，
使每个账号在窗口结束时尽量用满配额。

## 算法设计

### 核心公式

elapsed = window_duration - reset_after_seconds
target_pct = (elapsed / window_duration) * 100
deficit = target_pct - actual_used_pct

deficit 越大 = 落后越多 = 越应该优先使用

### Codex（executor.js selectBestAccountFromLocal）

- 5h primary_window.used_percent > 95% → gate filter（跳过）
- 读取 secondary_window.used_percent 和 secondary_window.reset_after_seconds
- 按 deficit DESC 排序（同 deficit 时 5h 用量 ASC）

### Claude Code（account-usage.js selectBestAccount）

- 5h 已有 isAccountEligibleForTier gate（five_hour_pct < 80，保持）
- 使用 DB 中已有的 seven_day_resets_at 字段计算 elapsed
- 按 deficit DESC 排序（sonnet tier 用 sevenDaySonnetPct，其他用 sevenDayPct）

## 成功标准

1. executor.js selectBestAccountFromLocal 使用 deficit 排序（5h>95% gate + 7d deficit DESC）
2. account-usage.js selectBestAccount 使用 deficit 排序（7d deficit DESC）
3. 单元测试 llm-account-deficit.test.js 覆盖 deficit 计算逻辑（≥5 用例）
4. 现有测试套件 npm test 全部通过
5. CI 全部通过
