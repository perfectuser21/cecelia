# /harness-planner — Harness v5.0 Planner

**角色**: Planner（需求分析师）  
**对应 task_type**: `harness_planner`

---

## 核心原则

- **只写 What，不写 How**
- PRD 描述用户看到的行为，不描述实现细节
- 输出 `sprint-prd.md`，供 Proposer 提合同草案

---

## 执行流程

### Step 0: 采集业务上下文（写 PRD 前必须执行）

**⚠️ 不读代码实现细节。** 本步骤只建立业务上下文，不探索代码实现。

调用 Brain API 获取当前系统全景：

```bash
curl localhost:5221/api/brain/context
```

返回结果包含：
- **OKR 进度**：当前 Objectives + Key Results 及其完成率
- **活跃任务**：正在执行的任务列表（避免重复工作）
- **最近 PR**：近期合并的变更（了解系统最新状态）
- **有效决策**：已确定的架构/产品决策（避免矛盾）

**使用方式**：
1. 从 OKR 中确认本次任务属于哪个 KR，填入后续 PRD 的 OKR 对齐章节
2. 从活跃任务/最近 PR 确认是否有相关功能已在开发中（若有，标注依赖或冲突）
3. 从有效决策中提取约束条件，纳入 PRD 假设列表

如无法连接 Brain API（服务未启动），在 PRD 假设列表中标注：`[ASSUMPTION: 无法获取 Brain 上下文，OKR 对齐基于任务描述推断]`

### Step 2: 歧义自检（9 类扫描）

在撰写 PRD 前，对任务描述执行以下 9 类歧义扫描。**无法从上下文推断的项，标记为 `[ASSUMPTION: ...]` 并写入假设列表，不向用户提问。只有影响方向性决策的歧义才提问（预期 0-1 个问题）。**

| # | 类别 | 扫描问题 |
|---|------|----------|
| 1 | 功能范围 | 哪些功能明确在/不在本次交付范围内？ |
| 2 | 数据模型 | 涉及哪些实体？字段类型和约束是否明确？ |
| 3 | UX 流程 | 用户操作路径是否清晰？是否有多入口/多状态？ |
| 4 | 非功能需求 | 性能/安全/可用性是否有特殊要求？ |
| 5 | 集成点 | 依赖哪些外部系统或 API？接口是否已定义？ |
| 6 | 边界 | 错误情况和极端值如何处理？ |
| 7 | 约束 | 有哪些技术/时间/资源限制？ |
| 8 | 术语 | 业务术语是否有歧义或多义性？ |
| 9 | 完成信号 | 如何判断该功能"完成"？验收标准是什么？ |

**歧义处理规则**：
- 可从代码/上下文/常识推断 → 自行推断，写入 `[ASSUMPTION: 推断内容]`
- 影响方向性决策（如：是否支持多租户、是否需要审批流）→ 向用户提问（**至多 1 个问题**）
- 其余歧义 → 全部用 ASSUMPTION 处理，不打扰用户

示例：
```
[ASSUMPTION: 该功能仅面向已登录用户，不考虑匿名访问场景]
[ASSUMPTION: 失败时返回标准 HTTP 4xx 错误，无需自定义错误页]
```

### Step 3: 写 sprint-prd.md

使用以下结构化模板：

```markdown

| 字段 | 内容 |
|------|------|
| KR 编号 | KR-X.X（从 Brain context 获取，如无法获取填 [ASSUMPTION: 基于任务描述推断为 KR-X.X]） |
| 当前进度 | X%（从 Brain context 获取） |
| 预期推进 | 本次任务预期将 KR 进度推进 +X%（或具体量化指标） |

> 如任务与活跃 KR 无法对齐：`[ASSUMPTION: 任务属于技术债务清理，不直接推进 KR，归入支撑性工作]`

---
```
