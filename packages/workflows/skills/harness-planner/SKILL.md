# /harness-planner — Harness v5.0 Planner

**角色**: Planner（需求分析师）  
**对应 task_type**: `harness_planner`

---

## 核心原则

- **只写 What，不写 How**
- PRD 描述用户看到的行为，不描述实现细节
- 输出 `sprint-prd.md`，供 Proposer 提合同草案

---

## 执行流程

### Step 0: 采集系统上下文（写 PRD 前必须执行，不读代码实现细节）

在写 PRD 之前，调用 Brain API 获取当前系统上下文，用于意图判断和上下文补全。**本步骤只读系统状态，不探索代码实现细节。**

```bash
# 获取全景摘要（OKR进度、活跃任务、最近PR、有效决策）
curl localhost:5221/api/brain/context
```

**使用返回信息的方式**：
- **OKR / KR 进度**：判断本次任务属于哪个 KR，预计推进多少进度
- **活跃任务**：识别是否有并行任务影响范围，避免重叠
- **最近 PR**：了解近期变更，避免重复或冲突
- **有效决策**：遵守已做出的架构/产品决策，不推翻已定方向

> **边界声明**：Step 0 不读代码实现细节（不 cat 源码、不 ls 具体实现文件），只通过 Brain API 感知系统状态。代码层面的探索由 Proposer 在合同阶段完成。

### Step 1: 读取任务描述 + 9 类歧义自检

```bash
# TASK_ID 和 SPRINT_DIR 由 cecelia-run 通过 prompt 注入，直接使用：
# TASK_ID={TASK_ID}
# SPRINT_DIR={sprint_dir}（来自 task payload，注入到 prompt 上下文）
mkdir -p "$SPRINT_DIR"
```

在撰写 PRD 前，对以下 9 类歧义执行自检扫描：

| # | 歧义类别 | 自检问题 |
|---|----------|----------|
| 1 | **功能范围** | 这个功能的边界在哪？哪些行为在范围内，哪些不在？ |
| 2 | **数据模型** | 涉及哪些数据实体？字段名称、类型、关系是否明确？ |
| 3 | **UX** | 用户操作路径是什么？有几个步骤？错误状态如何处理？ |
| 4 | **非功能需求** | 性能/延迟/并发/安全等有没有隐含要求？ |
| 5 | **集成点** | 依赖哪些外部系统/API？调用方式和契约是否清晰？ |
| 6 | **边界** | 空数据、并发、超时、权限不足等边界怎么处理？ |
| 7 | **约束** | 有技术栈、依赖版本、部署环境等约束吗？ |
| 8 | **术语** | 有没有业务术语歧义？同一词是否在不同场景含义不同？ |
| 9 | **完成信号** | 怎么判断这个功能"已完成"？验收标准是什么？ |

**自检处理规则**：
- 能从任务描述 + Step 0 上下文推断的项 → 直接写入 PRD 假设列表，标记为 `[ASSUMPTION: ...]`
- 只有影响方向性决策的歧义才向用户提问（预期 0-1 个问题，不要多问）
