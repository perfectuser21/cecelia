# Stage 3: Integrate — Push + PR

> push + 创建 PR。CI 监控和合并由 Stop Hook 自动控制。

---

## 3.0 Harness 模式

```bash
BRANCH_NAME=$(git rev-parse --abbrev-ref HEAD)
DEV_MODE_FILE=".dev-mode.${BRANCH_NAME}"
HARNESS_MODE=$(grep "^harness_mode:" "$DEV_MODE_FILE" 2>/dev/null | awk '{print $2}' || echo "false")
```

**harness_mode = true 时**：只需 push + 创建 PR，不监控 CI。
Stop Hook 0.5 通道检测到 step_2_code=done + PR 已创建 → 自动 exit 0。

## 3.3 CI 失败修复（使用 `superpowers:systematic-debugging`）

> 不需要主动轮询 CI。Stop Hook 会在 CI 失败时 exit 2 并告知原因。
> 收到 Stop Hook 的 CI 失败指令后，**使用系统化调试流程**（不是瞎试）：

### Phase 1: 根因调查（禁止跳过）

```
① 读完整错误（gh run view <run_id> --log-failed）
  → 不要只看最后一行，读完整 stack trace
② 检查最近改动（git diff HEAD~1）
  → 什么变了可能导致这个？
③ 本地复现
  → npm test / npx vitest run <failing-test>
  → 能本地复现 → Phase 3
  → 不能复现 → 检查 CI 环境差异（ubuntu vs macOS、node 版本、无 Brain）
```

### Phase 2: 修复 + 验证

```
④ 形成假设："我认为 X 是根因因为 Y"
⑤ 最小修复 — 一次只改一个变量
⑥ 本地验证通过 → push
```

### 3 次失败升级规则

```
CI 修复 push ≤ 2 次: 正常 — 回到 Phase 1 重新分析
CI 修复 push = 3 次: 停下来
  → 不要再 push 第 4 次修复
  → 这不是"还差一点"，是方向错了
  → 检查：是不是架构/方案本身有问题？
  → 使用 superpowers:dispatching-parallel-agents 派 subagent 独立分析
```
