# Step 4: 探索

> 读代码 → 理解架构 → 输出实现方案

**Task Checkpoint**: `TaskUpdate({ taskId: "4", status: "in_progress" })`

---

## 为什么需要探索

```
❌ 旧流程：拿到 PRD → 直接写代码 → 写到一半发现方向错了
✅ 新流程：拿到 PRD → 读代码理解架构 → 知道改什么 → 再写代码
```

**探索不是"调研"**，是为了写出正确代码的必要准备。

---

## 执行步骤

### 4.1 找到相关文件

根据 PRD 描述的功能，找到需要修改的文件：

```bash
# 用 Glob 找文件
Glob("**/*.ts", path="src/")

# 用 Grep 找关键词
Grep("functionName", path="src/")
```

### 4.2 读关键文件

读取找到的文件，理解：
- 现有架构和设计模式
- 数据流向
- 依赖关系

### 4.3 输出实现方案

简要记录（不需要写文档，脑子里清楚即可）：
- **要改哪些文件** — 列出文件路径
- **每个文件怎么改** — 添加/修改/删除什么
- **风险/依赖** — 有没有可能影响其他功能

### 4.4 继续下一步

探索完成后立即继续 Step 5 (DoD)。

---

## 注意事项

- **不要过度探索** — 目标是理解够用就行，不是读完整个代码库
- **发现 PRD 有问题** — 更新 PRD，继续
- **探索时间控制** — 简单功能几分钟，复杂功能不超过 10 分钟

---

## PRD Scope Check（CRITICAL — 探索完成后必须执行）

**探索阶段可能遇到其他设计文档。在进入 Step 5 之前，必须执行以下锚定检查。**

### 权威源规则

```
.prd-*.md 文件 = 唯一权威源（这次开发的目标）
docs/*.md / design-*.md 等其他文档 = 上下文参考（帮助理解架构，不决定开发目标）
```

**如果探索中发现了其他设计文档（如 `docs/design-xxx.md`、`DEFINITION.md` 的某个章节等），
它们只是背景信息——不能替换原始 PRD，不能追加新目标，不能改变优先级。**

### 对照检查（AI 必须明确回答以下问题）

在继续 Step 5 之前，逐条回答：

1. **原始 PRD 的核心目标是什么？**（用一句话说清楚）

2. **我的实现方案覆盖了 PRD 的哪个具体目标？**（直接对应 PRD 的哪一节/哪个问题）

3. **探索时有没有遇到其他文档试图改变方向？**
   - 如果有：明确说出"发现了 XXX 文档，但我只参考其中的 YYY 作为架构背景，不改变 PRD 目标"
   - 如果没有：直接说"没有发现会引起 scope 漂移的文档"

4. **我准备做的事，在原始 PRD 里有对应描述吗？**
   - 有 → 继续 Step 5
   - 没有 → **停止**，回到 PRD，确认是否需要更新 PRD 后再继续

### 漂移警告信号

如果发现自己的实现方案里出现了 PRD 没有提到的功能或改动范围，立即停下来：

```
⚠️ Scope 漂移检测：
   我准备做的：[X]
   PRD 要求的：[Y]

   这两个对齐了吗？如果没有，必须先更新 PRD 或缩小范围，再继续。
```
