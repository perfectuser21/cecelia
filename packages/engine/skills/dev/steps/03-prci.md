# Step 3: PR+CI — 提交 + 等待 CI

> 原 Step 08（PR创建）+ Step 09（CI监控）合并。

---

## 3.1 提交 PR（原 Step 08）

> commit + push + PR（版本号由 auto-version.yml 自动处理）
> **创建 PR 后继续 3.2 等待 CI，Stop Hook 控制完成条件**

**Task Checkpoint**: `TaskUpdate({ taskId: "3", status: "in_progress" })`

---

### 8.1 会话恢复检测

**先检测是否是恢复的会话**：

```bash
EXISTING_PR=$(gh pr list --head "$BRANCH_NAME" --state all --json number,url,state -q '.[0]' 2>/dev/null)

if [ ! -z "$EXISTING_PR" ]; then
  PR_STATE=$(echo "$EXISTING_PR" | jq -r '.state')

  if [ "$PR_STATE" = "MERGED" ]; then
    echo "✅ PR 已合并，跳到 cleanup"
  elif [ "$PR_STATE" = "OPEN" ]; then
    echo "✅ PR 已存在，跳到 CI"
  fi
fi
```

---

### 8.2 版本号 — 不要手动 bump

**版本号由 `auto-version.yml` 在合并到 main 后自动处理。PR 里不做任何版本操作。**

auto-version.yml 自动处理的文件（5 个）：
- `packages/brain/package.json`
- `packages/brain/package-lock.json`
- `.brain-versions`
- `DEFINITION.md`（Brain 版本行）
- `packages/brain/VERSION`

**commit 消息前缀决定 bump 类型**（auto-version 从 squash merge 的 commit message 解析）：

| commit 前缀 | 版本变化 |
|-------------|----------|
| fix: | patch (+0.0.1) |
| feat: | minor (+0.1.0) |
| feat!: / BREAKING: | major (+1.0.0) |
| 其他（docs:、test:、chore:） | 不 bump |

**禁止在 PR 中做以下操作**：
- ❌ `npm version patch/minor/major`
- ❌ 手动改 package.json 版本号
- ❌ 手动改 .brain-versions / VERSION / DEFINITION.md 版本
- ❌ 运行 `check-version-sync.sh`

**原因**：手动 bump 会导致并行 PR 版本冲突，浪费 10+ 分钟解决冲突 + 重跑 CI。

### 8.3.5 ⛔ 自检：Engine 版本文件完整性（commit 前必须通过）

**如果本次改动涉及 `packages/engine/` 下的任何文件，必须检查 7 个版本文件全部同步更新。**

```bash
CHANGED_ENGINE=$(git diff --name-only origin/main 2>/dev/null | grep "^packages/engine/" | head -1)

if [[ -n "$CHANGED_ENGINE" ]]; then
    echo "🔍 检测到 Engine 改动，验证版本 7 文件同步..."
    ERRORS=0
    CHANGED=$(git diff --name-only origin/main 2>/dev/null)

    check_file() {
        if ! echo "$CHANGED" | grep -q "^${1}$"; then
            echo "❌ 未修改: $1"
            return 1
        fi
        echo "✅ $1"
        return 0
    }

    check_file "packages/engine/package.json"         || ERRORS=1
    check_file "packages/engine/package-lock.json"    || ERRORS=1
    check_file "packages/engine/VERSION"              || ERRORS=1
    check_file "packages/engine/.hook-core-version"   || ERRORS=1
    check_file "packages/engine/regression-contract.yaml" || ERRORS=1
    check_file "packages/engine/ci-tools/VERSION"     || ERRORS=1
    # 根目录 package-lock.json（engine 条目）
    check_file "package-lock.json"                    || ERRORS=1

    if [[ $ERRORS -gt 0 ]]; then
        echo ""
        echo "⛔ Engine 版本文件不完整！"
        echo "   必须同时更新 7 个文件（少一个 L2 Consistency Gate 就失败）："
        echo "   packages/engine/package.json"
        echo "   packages/engine/package-lock.json"
        echo "   packages/engine/VERSION"
        echo "   packages/engine/.hook-core-version"
        echo "   packages/engine/regression-contract.yaml"
        echo "   packages/engine/ci-tools/VERSION"
        echo "   package-lock.json（根目录，engine 条目）"
        echo ""
        echo "   还需要在 feature-registry.yml 新增 changelog 条目，并运行："
        echo "   bash packages/engine/scripts/generate-path-views.sh"
        exit 1
    fi

    echo "✅ Engine 版本 7 文件全部已修改"
else
    echo "ℹ️  非 Engine 改动，跳过版本文件检查"
fi
```

```bash
/usr/bin/git push -u origin HEAD
```

### 8.6 创建 PR

```bash
BASE_BRANCH=$(git branch -r | grep -q 'origin/develop' && echo "develop" || echo "main")
gh pr create --base $BASE_BRANCH --title "feat: <功能描述>" --body "## Summary
- <主要改动>

## Test
- [x] 本地测试通过

## SYSTEM BEHAVIOR CHANGE
Before: [系统之前怎么跑 / 此处填写变更前的系统行为，若无行为变化填 N/A]
After:  [系统现在怎么跑 / 此处填写变更后的系统行为]
Log:    [关键日志事件 / 如 task dispatched, brain tick, API response 等]

## UNIMPLEMENTED
- [本 PR 故意不做的事，明确范围边界]

---
Generated by /dev workflow"
```

**行为声明规则**：
- `delivery_type=behavior-change` → SYSTEM BEHAVIOR CHANGE 必须填写实际内容（不能填 N/A）
- `delivery_type=code-only` → SYSTEM BEHAVIOR CHANGE 可填 N/A，但 UNIMPLEMENTED 必须填写
- 所有 PR 必须包含 `## SYSTEM BEHAVIOR CHANGE` 和 `## UNIMPLEMENTED` 两段

### 8.7 完成后

**标记步骤完成（PR 创建后）**：

```bash
BRANCH_NAME=$(git rev-parse --abbrev-ref HEAD 2>/dev/null || echo "")
DEV_MODE_FILE=".dev-mode.${BRANCH_NAME}"
sed -i "s/^step_3_prci: pending/step_3_prci: in_progress/" "$DEV_MODE_FILE"
echo "✅ PR 已创建，进入 CI 监控阶段"
```

**PR 创建后，立即继续 3.2 等待 CI 并合并**。

Stop Hook 完成条件：
1. PR 已创建 ✅（刚创建）
2. CI 通过？→ 3.2 检查
3. PR 已合并？→ 3.2 执行

**不要在 PR 创建后停止**，继续执行 3.2。

---

## 3.2 CI 监控（原 Step 09）

> 等待 CI 通过，**不在此处合并 PR**
> CI 通过后执行 Step 4（写 LEARNINGS → push → 合并 PR），合并在 Step 4 里完成，不回到 Step 3
> **Stop Hook 控制：CI 未通过或 PR 未合并时阻止会话结束（exit 2）**

---

### 流程

```
PR 创建 → 等待 CI → CI 失败？本地复现 → 修复 → 本地全绿 → push → CI 通过 → Step 4
                          ↑                                                          ↓
                    Stop Hook exit 2                                      push LEARNINGS 到功能分支
```

---

### 9.1 检查 CI 状态

```bash
BRANCH_NAME=$(git rev-parse --abbrev-ref HEAD)

# 等待 CI 启动
sleep 30

# 查询 CI 状态
RUN_INFO=$(gh run list --branch "$BRANCH_NAME" --limit 1 --json status,conclusion,databaseId)
CI_STATUS=$(echo "$RUN_INFO" | jq -r '.[0].status')
CI_CONCLUSION=$(echo "$RUN_INFO" | jq -r '.[0].conclusion')

echo "CI 状态: $CI_STATUS"
echo "CI 结论: $CI_CONCLUSION"
```

---

### 9.2 CI 状态处理

| CI 状态 | 动作 |
|---------|------|
| queued / in_progress | 等待，继续检查 |
| failure | **先本地复现**，修复后才 push（见 9.2b） |
| success | ⚠️ 执行 Step 4 (Learning)，不在此处合并 |

---

### 9.2b CI 失败修复（本地优先规则）

> **核心规则：CI 失败后，禁止直接 push 修复。必须先本地复现，本地全绿后才 push。**

#### 强制执行顺序

```
① 分析 CI 失败原因
  ↓
② 本地复现（运行对应本地检查）
  ↓
③ 修复代码
  ↓
④ 本地验证通过（才允许 push）
  ↓
⑤ push
```

#### ① 分析失败原因

```bash
RUN_ID=$(echo "$RUN_INFO" | jq -r '.[0].databaseId')
FAIL_LOG=$(gh run view "$RUN_ID" --log-failed 2>&1 | head -80)
echo "$FAIL_LOG"

FAILED_CHECKS=$(gh run view "$RUN_ID" --json jobs \
    -q '[.jobs[] | select(.conclusion == "failure") | .name] | join(", ")' 2>/dev/null || echo "unknown")
echo "失败的 checks: $FAILED_CHECKS"
```

**记录 incident（分析前必须先记录）**：

```bash
INCIDENT_FILE=".dev-incident-log.json"
ENTRY=$(jq -n \
    --arg ts "$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
    --arg checks "$FAILED_CHECKS" \
    --arg log "$(echo "$FAIL_LOG" | head -20)" \
    '{timestamp: $ts, step: "3-prci", type: "ci_failure", description: ("CI 失败: " + $checks), error: $log, resolution: ""}')
if [[ -f "$INCIDENT_FILE" ]]; then
    jq --argjson e "$ENTRY" '. += [$e]' "$INCIDENT_FILE" > /tmp/incident_tmp.json && mv /tmp/incident_tmp.json "$INCIDENT_FILE"
else
    jq -n --argjson e "$ENTRY" '[$e]' > "$INCIDENT_FILE"
fi
```

#### ② 本地复现（按失败类型选择对应检查）

根据 `$FAILED_CHECKS` 判断失败类型，运行对应的本地检查：

| CI 失败 check | 本地复现命令 |
|---------------|-------------|
| `facts-check` | `node scripts/facts-check.mjs` |
| `manifest-sync` | `node packages/brain/scripts/generate-manifest.mjs --check` |
| `brain-test`（覆盖率/测试） | `cd packages/brain && npm test` |
| `version-check`（engine） | `cd packages/engine && npm run version-check` |
| `dod-check` | `node packages/engine/scripts/devgate/check-dod-mapping.cjs` |
| **Brain 相关（含 facts/manifest/version）** | `bash scripts/local-precheck.sh --force` |
| **通用兜底** | 先 `bash scripts/local-precheck.sh`，再跑对应 package 的 `npm test` |

**⛔ 禁止**：本地复现步骤未执行就直接 push。

#### ③ 修复代码

根据本地复现的错误信息修复代码。

#### ④ 本地验证全绿（才允许 push）

```bash
# 确认本地预检通过
bash scripts/local-precheck.sh

echo "✅ 本地验证通过，准备 push"
```

**只有所有本地检查 exit 0，才能执行 push。**

#### ⑤ push

```bash
git add -A
git commit -m "fix: <具体修复内容>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>"

git push origin HEAD
```

**修复后继续等待 CI**，Stop Hook 会在会话尝试结束时检查条件。

---

### 9.3 CI 通过后 → 先执行 Step 4

**⚠️ 重要：CI 通过后不立刻合并 PR**

原因：需要在功能分支上写 LEARNINGS.md 并 push，让 PR 自动包含 LEARNINGS 变更。

```bash
echo "✅ CI 通过！"
echo "→ 执行 Step 4 (Learning)：写 LEARNINGS → push 到功能分支 → 合并 PR"
```

**继续 → Step 4 (Learning)**

---

### 9.4 Stop Hook 完成条件

| 条件 | 状态 | Stop Hook 行为 |
|------|------|---------------|
| PR 未创建 | ❌ | exit 2（继续创建 PR）|
| CI 失败 | ❌ | exit 2（继续修复）|
| CI 进行中 | ⏳ | exit 2（继续等待）|
| CI 通过但未合并 | ❌ | exit 2（继续合并）|
| **PR 已合并** | ✅ | exit 0（完成！）|

---

### 禁止行为

- ❌ CI 失败后不本地复现就直接 push 修复
- ❌ CI 失败后停止不管
- ❌ PR 创建后就结束
- ❌ 等待用户手动处理

### 正确行为

- ✅ CI 失败 → 分析原因 → **本地复现** → 修复 → 本地全绿 → push → 继续等待
- ✅ CI 通过 → 执行 Step 4 (Learning)，在 Step 4 里写 LEARNINGS + 合并 PR
- ✅ PR 合并（在 Step 4 内完成）→ 继续 Step 5

---

### 完成后

**CI 通过后立刻执行 Step 4（PR 还未合并，step_3_prci 在 Step 4 合并后才标记 done）**

```bash
BRANCH_NAME=$(git rev-parse --abbrev-ref HEAD 2>/dev/null || echo "")
DEV_MODE_FILE=".dev-mode.${BRANCH_NAME}"
sed -i "s/^step_3_prci: in_progress/step_3_prci: done/" "$DEV_MODE_FILE"
echo "✅ Step 3 完成标记已写入 .dev-mode"
```

**Task Checkpoint**: `TaskUpdate({ taskId: "3", status: "completed" })` ← Step 4 合并后执行

**继续 → Step 4 (Learning)**

写 LEARNINGS.md → push 到功能分支 → 合并 PR（PR 自动包含 LEARNINGS）
