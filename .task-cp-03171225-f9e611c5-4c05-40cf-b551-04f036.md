# Task Card: fix(executor): syncOrphanTasksOnStartup 孤儿改 requeue

## 需求（What & Why）
**功能描述**: 修复 `syncOrphanTasksOnStartup`，区分可重试孤儿（Brain 重启中断）和真实失败孤儿，将可重试的改为 queued 而非 failed。

**背景**: Brain 重启时，executor.js 的 `syncOrphanTasksOnStartup` 将所有 `in_progress` 任务无条件标记为 `failed`，导致静默数据丢失。正常的 Brain 重启会中断执行中的任务，这些任务应当被重新排队，而非直接标记为失败。

**不做什么**: 不改变 `startup-recovery.js` 的 `runStartupRecovery` 逻辑；不修改 liveness probe 路径；不新增数据库列。

## 成功标准
> [ARTIFACT] 产出物 / [BEHAVIOR] 运行时行为 / [GATE] 门禁

1. [ARTIFACT] `syncOrphanTasksOnStartup` 内包含 requeue 逻辑分支，且有 `watchdog_retry_count` 判断
2. [BEHAVIOR] Brain 重启后，之前 `in_progress` 且 `watchdog_retry_count < QUARANTINE_AFTER_KILLS` 且无 `error_message` 的任务出现在 `queued` 状态
3. [BEHAVIOR] `watchdog_retry_count >= QUARANTINE_AFTER_KILLS` 或有 `error_message` 的任务仍保持 `failed`
4. [GATE] CI 全部通过

## 实现方案

**要改的文件**:
- `packages/brain/src/executor.js`：修改 `syncOrphanTasksOnStartup` 函数（约第 2308-2384 行）
- `packages/brain/src/__tests__/executor-startup-sync.test.js`：新建单元测试

**Scope 锚定**:
- 在 orphan 路径（no matching process）中，读取 `task.payload?.watchdog_retry_count` 和 `task.error_message`
- `const QUARANTINE_AFTER_KILLS = 2`（已在函数外定义，直接引用）
- 可重试条件：`watchdog_retry_count < QUARANTINE_AFTER_KILLS && !errorMessage`
- requeue 时：UPDATE status='queued', payload 中 watchdog_retry_count++, error_message='requeued after brain restart'
- 真实失败时：保持现有 UPDATE status='failed' 逻辑
