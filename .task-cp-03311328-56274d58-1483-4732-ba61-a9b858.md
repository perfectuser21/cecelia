## 背景

Brain 调度 KR 进度诊断任务。当前现象：`/api/brain/okr/current` 返回的 5 个 objectives 中，4 个无 key_results（进度不可见），仅 `6b3b84b1`（Cecelia 系统稳定）的 3 个 KR 可见。实际存在 4 个内容 KR（属于 objective `4a2c8a08` ZenithJoy 自媒体）但完全不出现在 API 结果中。

---

## 根本原因诊断

### 根因 1：OKR API LIMIT 5 排除了有 KR 的 objective

`GET /api/brain/okr/current`（`okr-hierarchy.js` L347-353）查询：
```sql
SELECT id, title, status, description
FROM objectives
WHERE status != 'archived'
ORDER BY created_at DESC
LIMIT 5
```

2026-03-25 创建了 4 个新 objective（均无 KR），占满 LIMIT 5，导致 `4a2c8a08`（创建于 2026-03-23，有 4 个 KR）排在第 6 位，被截断不可见。

### 根因 2：内容生成 vs 自动发布 verifier 查询相同

PR #1763（migration 208）修复了空表问题后，两个 KR 均使用：
```sql
SELECT COUNT(*)::numeric as count FROM content_publish_jobs
WHERE created_at > NOW() - INTERVAL '30 days'
```
语义不同的 KR 用同一指标，`内容生成`（AI 产出内容）和`自动发布`（成功发布）无法区分。

---

## 修复方案

### Fix 1：`packages/brain/src/routes/okr-hierarchy.js`

将 objectives 查询改为：只返回有活跃 KR 的 objectives，移除 LIMIT 5：

```sql
SELECT DISTINCT o.id, o.title, o.status, o.description
FROM objectives o
JOIN key_results kr ON kr.objective_id = o.id
WHERE o.status != 'archived'
  AND kr.status != 'archived'
ORDER BY o.created_at DESC
```

### Fix 2：Migration `209_differentiate_kr_verifier_queries.sql`

- **自动发布 KR**（4b4d2262）：改为只计 `status = 'completed'` 的 publish_jobs（真实发布成功数）
- **内容生成 KR**（65b4142d）：改为计所有非失败的 publish_jobs（`status != 'failed'`，内容产出量）
