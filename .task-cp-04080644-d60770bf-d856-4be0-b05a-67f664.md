# Task Card: 认证层故障后遗症排查 + Scanner 认证健康度探针

## 需求（What & Why）

**功能描述**: 
- 分析认证层故障根因（account3 auth_fail_count_24h = 166）
- 验证凭据恢复状态
- 在 `credential-expiry-checker.js` 中添加 `scanAuthLayerHealth()` 实时失败率探针
- 在 `tick.js` 中集成该探针（每次 credential-check 时顺带执行）

**背景**:
account3 OAuth token 在今日（2026-04-08）大量失败（166次/24h），引发级联效应：
- 派给 account3 的所有 agent 任务无法启动 → watchdog liveness_dead → quarantined
- 大批 Pipeline Rescue 任务（worktree 路径不存在）堆积
- 已有 `proactiveTokenCheck` 可检测 token 过期并熔断，但缺少"实时失败率监控"探针

**根本原因**:
1. `proactiveTokenCheck` 检查 token 文件的 `expiresAt` 字段，但 codex provider 使用自身 API key，两者失败方式不同
2. Brain 的 `recoverAuthQuarantinedTasks` 仅恢复 `failure_class = 'auth'` 的任务，Pipeline Rescue 类型被正确跳过

**不做什么**: 
- 不修改 proactiveTokenCheck（已有 #2069 修复）
- 不手工操作 quarantined 任务（自动恢复机制已足够）
- 不修改 Pipeline Rescue 的生命周期

## 成功标准

> [BEHAVIOR] `scanAuthLayerHealth()` 在近1小时 auth 失败 ≥ 5次时创建 P1 告警任务

## 实现方案

**要改的文件**:
- `packages/brain/src/credential-expiry-checker.js` — 新增 `scanAuthLayerHealth()` 函数（约60行）
- `packages/brain/src/tick.js` — import + 在 credential-check 块内调用

**逻辑**:
1. 查询 `tasks` 表中 `updated_at > NOW() - INTERVAL '1 hour'` AND `payload->>'failure_class' = 'auth'`，按 `dispatched_account` 分组统计
2. 任意账号近1小时失败 ≥ `AUTH_FAIL_RATE_THRESHOLD`（默认5）→ 创建 P1 告警
3. 防重告警：`6 hours` 内已有同类告警任务则跳过

**不改什么**: 其他 credential 检查逻辑，proactiveTokenCheck
