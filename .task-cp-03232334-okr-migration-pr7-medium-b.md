# Task Card: OKR 业务代码迁移 PR7 — MEDIUM 组B（4个文件）

## 需求（What & Why）

**功能描述**：
将 planner.js / self-drive.js / similarity.js / validate-okr-structure.js 中对旧 `goals` / `projects` 表的直接查询迁移到新 OKR 七层表（visions / objectives / key_results / okr_projects / okr_scopes / okr_initiatives）。

**背景**：
migration 177-182 已建好新 OKR 表体系，双向同步触发器（migration 181）保证旧表写入会自动同步到新表。UUID 完全相同。项目目标是让所有业务代码读取新表，彻底不依赖 goals/projects 表，最终在 PR9 删除旧表。

**迁移规则**：
- `goals.id == objectives/key_results.id`（UUID 相同，直接替换）
- `projects.id == okr_projects/okr_scopes/okr_initiatives.id`（UUID 相同，直接替换）
- `status: in_progress → IN ('active', 'in_progress')`（新表含 active 状态）
- `projects.name → title`（新 okr_* 表字段名变化）
- `goals.type` 路由：vision→visions / area_okr→objectives / area_kr|global_kr→key_results
- `projects.type` 路由：project→okr_projects / scope→okr_scopes / initiative→okr_initiatives
- `project_kr_links` 保留不变（IDs 相同，仍可 JOIN 新表）

**4 个文件的迁移方向**：
1. `planner.js` — 多 type 分支查询（objectives/keyResults/projects/initiatives），需 UNION ALL across vision/global_kr/area_okr/project/scope/initiative
2. `self-drive.js` — getKRProgress() 读 active KR（goals → key_results）、getActiveProjects()（projects → okr_projects）、getGoalIdForArea()（goals → key_results/objectives）、executeAdjustmentAction() 中 UPDATE goals/projects（继续用旧表即可，触发器同步）
3. `similarity.js` — 向量相似度搜索 projects.id，JOIN project_kr_links 获取 kr_id（改为 JOIN okr_projects + project_kr_links，或 UNION ALL 三张 okr 项目表）
4. `validate-okr-structure.js` — 递归查 goals tree（parent_id 链）+ projects 树验证，WITH RECURSIVE 需改用新表

**不做什么**：
- 不修改 goals/projects 表的 INSERT/UPDATE（触发器自动同步，无需修改写操作）
- 不删除旧表（PR9 才做）
- 不修改 project_kr_links 表（IDs 相同，保留）
- handlePlanInput() 中已有的 INSERT INTO goals/projects 不改（触发器保证双写）
