# Task Card: Scanner 内嵌能力识别规则补充

## 需求（What & Why）

**功能描述**: 梳理并修复 Scanner 对 Brain 内嵌能力的判定逻辑，重点覆盖 stage=3 的能力，避免把已运行的内嵌流程误判为孤岛；补充识别规则和回归用例。

**背景**: PR #1479 已将 `dev-workflow`、`self-healing`、`self-healing-immunity` 加入 `BRAIN_ALWAYS_ACTIVE`，修复了具体实例。但缺乏系统性规则：stage≥3 代表"已部署运行"，其状态应为 `dormant`（休眠未触发）而非 `island`（孤岛/不存在）。若未来新增 stage=3 能力，仍可能被误判。

**不做什么**: 不改变 `BRAIN_ALWAYS_ACTIVE` / `BRAIN_EMBEDDED_SOURCES` / `INFRA_DEPLOYED_CAPABILITIES` 三个白名单的现有条目。

## 成功标准

- [ ] `determineStatus` 接受 `stage` 参数，stage≥3 时返回 `dormant` 而非 `island`
- [ ] 新增回归测试：stage=3 无活动记录的能力返回 `dormant` 而非 `island`
- [ ] 新增回归测试：stage<3 无活动记录的能力仍返回 `island`（原有行为保持）
- [ ] 所有现有测试继续通过

## 验收条件（DoD）

- [ ] [PRESERVE] 现有 BRAIN_ALWAYS_ACTIVE 能力（如 `dev-workflow`）仍标记为 `active`（优先级高于 stage 规则）
  Test: tests/packages/brain/src/__tests__/capability-scanner.test.js

- [ ] [PRESERVE] 现有 INFRA_DEPLOYED_CAPABILITIES 能力仍标记为 `active`（infra_deployed 路径不受影响）
  Test: tests/packages/brain/src/__tests__/capability-scanner.test.js

- [ ] [ARTIFACT] `determineStatus` 函数接受第五个参数 `stage`，stage≥3 时 island→dormant
  Test: tests/packages/brain/src/__tests__/capability-scanner.test.js

- [ ] [BEHAVIOR] stage=3 + 无技能活动 + 无表数据的能力 → status=dormant（不是 island）
  Test: tests/packages/brain/src/__tests__/capability-scanner.test.js

- [ ] [BEHAVIOR] stage=1 + 无技能活动 + 无表数据的能力 → status=island（保持原有行为）
  Test: tests/packages/brain/src/__tests__/capability-scanner.test.js

- [ ] [GATE] 所有现有测试通过
  Test: manual:npm test --prefix packages/brain

   - `determineStatus(hasSkillActivity, successRate, hasTableData, lastActivity, stage)` 加第五个参数
   - 在 island 判断处加 `&& (!stage || stage < 3)` 条件
   - `evaluateCapability` 调用时传入 `cap.current_stage`
