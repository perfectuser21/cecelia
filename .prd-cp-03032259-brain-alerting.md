## PRD - Brain 分级报警系统（P0/P1/P2/P3）

**需求来源**: 用户反映 Cecelia 24/7 运行但出问题不知道；当前 notifier.js 只有 P0 一档

**功能描述**:
给 Brain 加四级报警机制，让不同严重程度的异常走不同通知渠道：
- P0：系统完全不工作（服务宕机、DB 断连、连续失败>5次）→ 立即飞书推送
- P1：核心功能降级（LLM 超时连续、任务队列积压、内存超限）→ 每小时汇总飞书
- P2：非核心异常（单次任务失败、非关键接口报错）→ 每日汇总
- P3：信息性记录 → 只写日志，不推送

**涉及文件**:
- `packages/brain/src/alerting.js` ← 新建，核心报警模块
- `packages/brain/src/notifier.js` ← 改造，接入分级
- `packages/brain/src/tick.js` ← 在关键点插入报警调用
- `packages/brain/src/executor.js` ← 任务失败时调用报警
- `packages/brain/src/server.js` ← 注册报警路由
- `packages/brain/src/routes/alerting.js` ← 新建路由（GET 状态 + POST 测试）
- `packages/brain/src/__tests__/alerting.test.js` ← 新建测试

**成功标准**:
1. Brain 任务连续失败 5 次 → 飞书收到 P0 报警
2. 单次任务失败 → 不推送（P2，只记录）
3. GET /api/brain/alerting/status → 返回当前各级别积压数量
4. POST /api/brain/alerting/test?level=P0 → 能触发测试报警

**非目标**:
- 不改 P0 飞书渠道的现有行为（只是抽象成 alerting.js 调用）
- 不做 P1 邮件通知（飞书已够）
- 不做报警静默/升级（CRITICAL：不过度设计）

**异常处理规范**:

| 异常场景 | 级别 | 通知方式 |
|---------|------|---------|
| DB 连接断开 | P0 | 立即飞书 |
| 任务连续失败 > 5 次 | P0 | 立即飞书 |
| Brain 服务重启 | P0 | 立即飞书 |
| LLM 连续超时 > 3 次 | P1 | 每小时汇总 |
| 任务队列积压 > 20 | P1 | 每小时汇总 |
| 单次任务失败 | P2 | 每日汇总 |
| Memory > 1.5GB | P1 | 每小时汇总 |
| 非关键 API 报错 | P2 | 每日汇总 |
