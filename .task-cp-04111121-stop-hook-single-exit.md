# Task Card: stop hook 单一出口 — 删除 ready_to_merge，CI 通过后自动合并

## 需求（What & Why）

**功能描述**: devloop-check.sh 条件 6（CI 通过 + step_4 done）当前返回 `ready_to_merge` 中间状态，  
stop-dev.sh 将其解释为 exit 2 并输出"请手动执行 gh pr merge"，导致用户需要手动介入或 Claude 卡住等待。  
应删除该中间状态，改为直接自动合并。

**背景**: 曾有 P0 修复故意禁用自动合并（"合并失败时由用户处理"），但这违反了"整个流程只有一个 exit 0 出口"的原则。  
CI 通过 + step_4 done = 条件全满足，应无人工介入直接合并。

**不做什么**:
- 不改 harness 模式路径（0.5 通道，独立逻辑）
- 不改 cleanup_done 出口逻辑
- 不改 CI in_progress / CI failed 的 block 路径

## 成功标准

- [x] devloop-check.sh 不再包含 `ready_to_merge` 字符串
- [x] CI 通过 + step_4 done → devloop_check 直接执行 `gh pr merge --squash` 并返回 `status: done`
- [x] 合并失败时 → 返回 `status: blocked`，action 指示 Claude 自行修复，不涉及用户
- [x] CI in_progress 的 action 为空（不触发 stop-dev.sh "立即执行"后缀）

## 验收条件（DoD）

- [x] [ARTIFACT] devloop-check.sh 不含 ready_to_merge 字符串
  Test: manual:node -e "const c=require('fs').readFileSync('packages/engine/lib/devloop-check.sh','utf8');if(c.includes('ready_to_merge'))process.exit(1);console.log('OK')"

- [x] [BEHAVIOR] CI 通过 + step_4 done 路径返回 status:done（通过 stop-hook-exit-codes.test.ts 覆盖）
  Test: tests/hooks/stop-hook-exit-codes.test.ts

- [x] [BEHAVIOR] CI in_progress 路径的 action 字段为空字符串或不包含"手动"
  Test: manual:node -e "const c=require('fs').readFileSync('packages/engine/lib/devloop-check.sh','utf8');const m=c.match(/CI 进行中[\s\S]{0,300}?action/);if(m&&m[0].includes('手动'))process.exit(1);console.log('OK')"

## 实现方案

**要改的文件**:
- `packages/engine/lib/devloop-check.sh` — 条件 6：删除 ready_to_merge，改为 gh pr merge 自动合并
- `packages/engine/lib/devloop-check.sh` — CI in_progress action 改为空字符串（仅 reason）
- `packages/engine/tests/hooks/stop-hook-exit-codes.test.ts` — 补 CI passed + step_4 done → auto-merge 测试
- `packages/engine/tests/scripts/devloop-check-gates.test.ts` — 补 ready_to_merge 已删除的断言

**受影响函数**: `devloop_check()` 条件 6（L282-311）

**不改什么**: harness 通道（0.5）、stop-dev.sh 主逻辑（仅 devloop-check.sh）
