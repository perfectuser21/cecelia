# PRD: 飞书群聊身份识别无权限修复

## Bug

飞书 app (cli_a92eb1709ff81bdf) 缺少权限：
- `contact:user:readonly` → 联系人 API code=99991672
- `im:chat:readonly` → 群成员列表 API code=99991672

`getGroupMembers` 在 `data.code !== 0` 时静默返回 `[]`（无日志），
导致 `feishu_users` 表永远是 0 行，所有群成员被识别为"用户"/"访客"。

## 修复

### 1. getGroupMembers 加诊断日志

- `data.code !== 0` 时加 `console.warn` 输出失败码 + chatId
- 成功时加 `console.log` 输出成员数量

### 2. getFeishuUserName staleCache 路径加 openId 日志

- `联系人 API 返回 code=...` 的 warn 日志加上 `openId`
- 便于从日志确认 Alex 的 open_id

### 3. FEISHU_OWNER_OPEN_IDS 环境变量支持

在 `getFeishuUserName` 中，DB 缓存查询之前加一个检查：

```javascript
const ownerOpenIds = (process.env.FEISHU_OWNER_OPEN_IDS || '').split(',').map(s => s.trim()).filter(Boolean);
if (ownerOpenIds.includes(openId)) {
  return { name: '徐啸', user_id: 'owner', relationship: 'owner' };
}
```

### 4. docker-compose.yml 加 FEISHU_OWNER_OPEN_IDS 环境变量

```yaml
- FEISHU_OWNER_OPEN_IDS=${FEISHU_OWNER_OPEN_IDS:-}
```

## 使用流程

1. 部署后 Alex 在飞书群聊发任意消息
2. 查看 Brain 日志：`docker logs cecelia-node-brain 2>&1 | grep openId`
3. 找到 `openId=ou_xxx` → 这就是 Alex 的 open_id
4. 在 `.env.docker` 中设置 `FEISHU_OWNER_OPEN_IDS=ou_xxx`
5. `docker compose restart node-brain` → 重启后 Alex 被识别为 owner
