# PRD: feat(capture) — 建立 Capture 基础设施

## 背景

Cecelia 现有数据库混合存储了系统数据（Brain 自动产生）和用户数据（用户主动输入），缺乏区分机制。
同时，用户通过飞书、Dashboard、日记等多渠道产生想法/任务，目前没有统一的捕获入口。

## 目标

1. 建立 `captures` 表 —— 统一收件箱，支持飞书/Dashboard/日记多来源捕获
2. 为 areas/objectives/key_results/okr_projects 添加 `owner` 字段 —— 区分 user 数据 vs cecelia 系统数据
3. 提供 Dashboard Quick Capture 输入框（所有页面顶部）
4. 提供 `/gtd/inbox` 收件箱页面 —— 查看和管理未处理的 captures

## 成功标准

- captures 表存在于 DB，含 id/content/source/status/area_id/project_id/extracted_to/owner/created_at/updated_at 字段
- areas/objectives/key_results/okr_projects 表均含 owner 字段（默认 'user'）
- GET /api/captures 返回 captures 列表
- POST /api/captures 创建新 capture 返回 201
- PATCH /api/captures/:id 更新 status 返回 200
- Dashboard /gtd/inbox 页面可访问、显示收件箱列表
- Layout 顶部有 Quick Capture 输入框，按 Enter 或点击提交后在 inbox 可见
- Brain 迁移文件已创建（198_captures_table.sql）
- GTDInbox 组件文件存在（GTDInbox.tsx）
- QuickCapture 组件文件存在（QuickCapture.tsx）

## 范围

### DB（Migration）
- 新增 `captures` 表
- ALTER areas/objectives/key_results/okr_projects 添加 `owner VARCHAR(20) DEFAULT 'user'`

### API（apps/api）
- `GET /api/captures` — 列表（支持 ?status=inbox）
- `POST /api/captures` — 创建（content, source='dashboard', area_id?）
- `PATCH /api/captures/:id` — 更新（status, area_id, extracted_to）

### Dashboard UI（apps/dashboard）
- `QuickCapture` 组件 — 固定在页面顶部，输入框 + 提交
- `/gtd/inbox` 页面 — 列表展示 captures，支持状态筛选

## 不在本次范围

- Brain 自动消化（extraction/digestion）逻辑
- 飞书 webhook 接入
- Capture 与 OKR 的关联编辑 UI
