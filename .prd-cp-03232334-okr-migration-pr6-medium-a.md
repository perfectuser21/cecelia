# Task Card: OKR 业务代码迁移 PR6 — MEDIUM 组A（4个文件）

## 需求（What & Why）

**功能描述**：将 4 个 MEDIUM 复杂度的 Brain 源码文件从引用旧 `goals`/`projects` 表迁移到新 OKR 7-表体系（visions/objectives/key_results/okr_projects/okr_scopes/okr_initiatives）。

**背景**：
- migration 177 建立了新 OKR 7表体系
- migration 179 完成了数据迁移（goals/projects → 新表）
- 触发器保证了双向同步（旧表写→新表同步，新表写→旧表同步）
- UUID 完全相同（goals.id == visions/objectives/key_results.id）
- PR1-PR5 已迁移其他批次文件

**涉及文件**：
1. `decomposition-checker.js` — 检查 pending KR 是否需要拆解，查询 goals WHERE type IN ('area_kr','kr') → key_results
2. `okr-tick.js` — processReadyGoals() 触发分解，查询 goals WHERE status='ready' → objectives/key_results 分支处理
3. `project-activator.js` — 激活/去激活 projects，LEFT JOIN goals → okr_projects
4. `project-compare.js` — 对比 projects 评分，LEFT JOIN goals → LEFT JOIN objectives/okr_projects

**迁移规则**：
- `goals.id` == `visions/objectives/key_results.id`（按 type 路由）
- `status: in_progress` → `IN (active, in_progress)`（新表用 active 代替 in_progress 作主状态）
- `projects.name` → `okr_projects.title`
- `goals.type` 路由：vision→visions / area_okr→objectives / area_kr|kr→key_results
- `projects.type` 路由：project→okr_projects / scope→okr_scopes / initiative→okr_initiatives

**不做什么**：
- 不删除 goals/projects 表（双写期，触发器同步）
- 不修改 API 契约
- 不迁移测试/日志字符串中的表名引用

## 实现方案（探索后填写）

**要改的文件**：
- `packages/brain/src/decomposition-checker.js` — Check A/B/C: `FROM goals WHERE type IN (...)` → `FROM key_results`；UPDATE goals 保持（触发器同步双写）
- `packages/brain/src/okr-tick.js` — `FROM goals WHERE status = $1` → 分两路：type=vision→visions，type=area_okr→objectives，type=area_kr/kr→key_results；getGoalsByStatus 改为 getKRsByStatus + getObjectivesByStatus 等
- `packages/brain/src/project-activator.js` — `LEFT JOIN goals g` → `LEFT JOIN okr_projects op`（因为 project-activator 查的是 project层级的 priority）
- `packages/brain/src/project-compare.js` — `FROM projects p LEFT JOIN goals g` → `FROM projects p LEFT JOIN okr_projects op`

**Scope 锚定**：只改 SQL 查询中的表名和列名；functions/exports 不变；测试 mock 对应更新。
