# PRD：修复嘴巴↔大脑通信断层

## 背景

Cecelia 的嘴巴（orchestrator-chat.js）和大脑（Brain Node.js）之间存在多个断点，
导致嘴巴无法通过 call_brain_api 工具获取实时数据，Brain 完成任务后也无法通知用户。

## 目标

让嘴巴真正能"问大脑、大脑执行、结果回来告诉用户"的完整链路跑通。

## 需求

### 1. 修复 callWithHistory JSON 解析 bug

**现状**：`if (trimmed.startsWith('{'))` 条件只处理 LLM 输出以 `{` 开头的情况。
当 LLM 输出"自然语言文字 + JSON 尾巴"时，解析跳过，reply = 整段原文（含 JSON），发给用户。

**修复**：用 `lastIndexOf('{')` 找最后一个 JSON 块，支持"文字 + JSON"格式。

### 2. 新增 GET /api/brain/feishu/groups

查询 unified_conversations 中已知的飞书群（group_id 不为空），
返回群列表 + 最近活跃时间，供嘴巴通过 call_brain_api 查询。

### 3. 新增 POST /api/brain/feishu/send

接收 `{group_id, text}` 或 `{open_id, text}`，
调用 getTenantAccessToken + sendFeishuMessage 主动发消息，
供嘴巴通过 call_brain_api 派发。

### 4. 修复 buildManifestBlock 假示例端点

当前注释里写了 `/api/brain/feishu/groups`（404），误导 LLM。
修复后改为真实存在的示例。

### 5. 配置 FEISHU_OWNER_OPEN_IDS

填入 Alex 的 open_id（ou_b472f4ce7a9bf227b9f6ad4e557fb687），
让 Brain 完成任务后 proactive-mouth → notifier → sendFeishu 能推给用户。

## 成功标准

- callWithHistory 正确解析"文字+JSON"格式，reply 不再含原始 JSON
- GET /api/brain/feishu/groups 返回已知群列表
- POST /api/brain/feishu/send 成功发送飞书消息
- 嘴巴问"查一下飞书群"时，call_brain_api 能拿到实际数据并回答
