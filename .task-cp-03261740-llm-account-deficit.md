# Task Card: 改进 LLM 账号选择算法（进度对齐 Deficit 排序）

## 需求（What & Why）

**功能描述**: 改进 Codex（`executor.js`）和 Claude Code（`account-usage.js`）的账号选择算法，从"最低使用率优先"改为"进度对齐（deficit）优先"。

**背景**:
5个 Codex 账号和 3个 Claude Code 账号各自拥有 7天滚动配额窗口，但窗口重置时间不同步。
当前算法只看当前使用率，会导致"配额窗口快到期的账号明明还有余量却被最后选"的浪费。
新算法按已过时间比例推算应达到的目标使用率，优先消耗落后于目标（deficit 最大）的账号，
使每个账号在窗口结束时尽量接近 100%。

**不做什么**:
- 不改变 5h session 窗口的 gate filter 逻辑（仍用 >95% 过滤）
- 不改变 spending cap、cascade 降级、isAccountEligibleForTier 等其他筛选逻辑
- 不修改 DB schema 或 API 调用频率

## 成功标准

1. [ARTIFACT] `executor.js selectBestAccountFromLocal` 使用 7d deficit 排序：
   - 5h primary_window.used_percent > 95% → 跳过
   - `deficit = (elapsed_secs/(7*24*3600)*100) - secondary_window.used_percent`，其中 `elapsed_secs = 7*24*3600 - secondary_window.reset_after_seconds`
   - 排序：deficit DESC（最落后先用），同 deficit 时 5h 用量 ASC

2. [ARTIFACT] `account-usage.js selectBestAccount` 使用 7d deficit 排序：
   - 5h 已有 gate（`isAccountEligibleForTier` 中的 `five_hour_pct < 80`，保持不变）
   - `deficit = (elapsed_ms/(7*24*3600*1000)*100) - sevenDayPct`，其中 `elapsed_ms = Date.now() - (new Date(sevenDayResetsAt) - 7*24*3600*1000)`
   - 排序：deficit DESC，同 deficit 时 ePct ASC

3. [ARTIFACT] 单元测试文件 `packages/brain/src/__tests__/llm-account-deficit.test.js` 覆盖 deficit 计算逻辑（≥5 个用例）

4. [BEHAVIOR] 现有测试套件 `npm test` 全部通过

5. [GATE] CI 全部通过
