# Task Card: 重构 advanceContentPipeline（复杂度 33 → <10）

## 需求（What & Why）

**功能描述**: 将 `packages/brain/src/content-pipeline-orchestrator.js` 中的 `advanceContentPipeline` 函数圈复杂度从 33 降至 10 以下。

**背景**: Brain 代码复杂度扫描发现该函数嵌套分支过多（4 个 if-task_type 块 + 内联三元运算符 + 多个 `||` 逻辑链），导致可读性差、测试覆盖困难。

**不做什么**:
- 不改变函数的外部行为和签名
- 不修改测试文件（测试必须全部通过，零改动）
- 不修改 `orchestrateContentPipelines`、`executeQueuedContentTasks` 等其他函数

## 实现方案

**要改的文件**: `packages/brain/src/content-pipeline-orchestrator.js`（仅此一个文件）

**重构策略**:
1. 提取 `_loadPipelineContext(taskId, dbPool)` → 负责加载 task/pipeline 数据 + 早期返回（3 个守卫条件）
2. 提取 `_handleResearchComplete(ctx, dbPool)` → 创建 content-generate
3. 提取 `_handleGenerateComplete(ctx, dbPool)` → 创建 content-review
4. 提取 `_handleReviewComplete(ctx, taskStatus, findings, dbPool)` → review 判断 + 重试/导出
5. 提取 `_handleExportComplete(ctx, dbPool)` → 创建发布任务 + 标记完成
6. 使用 dispatch map `STAGE_HANDLER_MAP` 替代 4 个 if-else 分支
7. `advanceContentPipeline` 精简为：加载上下文 → dispatch → 返回

**Scope 锚定**: 仅改 `advanceContentPipeline` 及其下方提取出来的新私有辅助函数；`_createNextStage`、`_isReviewPassed`、`_createPublishJobs`、`_stageLabel` 保持不变。
