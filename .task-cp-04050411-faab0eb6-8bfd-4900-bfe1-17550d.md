# Task Card: 修复 content pipeline 子任务被 dispatch 系统抢占导致 Codex quota 耗尽

## 需求（What & Why）

**功能描述**: 在 `tick.js` 的 `selectNextDispatchableTask` 查询中，将所有 content pipeline 子任务类型（`content-research`/`content-copywriting`/`content-copy-review`/`content-generate`/`content-image-review`）排除在 dispatch 系统之外，确保它们只由本地 `executeQueuedContentTasks` executor 执行。

**根因**:
- Content pipeline 子任务设计上由 Brain 本地 executor（`executeQueuedContentTasks`）执行，使用 Claude API via `callLLM`
- 但 dispatch 系统（`dispatchNextTask`）没有排除这些类型（只排除了 `content-export`）
- Dispatch 抢先把这些任务标记 `in_progress` 并发给 Codex（Xian 机器），Codex quota 耗尽 → 任务失败 → quarantine
- 导致 20+ 任务 quarantined，4 个活跃 pipeline 阻塞，内容生成 KR 进度停滞

**不做什么**: 不改动 content-export（已排除）、不改动 content_publish（正确应该走 dispatch 到 US 发布）

## 成功标准

- 新创建的 content pipeline 子任务不再被 dispatch 系统分配给 Codex
- 本地 executor 可以正常执行这些任务

## 实现方案

**要改的文件**: `packages/brain/src/tick.js`（第 696 行附近）

**要改的内容**:
```sql
-- 原来（只排除 content-export）:
AND t.task_type != 'content-export'

-- 改为（排除所有 content 子任务）:
AND t.task_type NOT IN ('content-export', 'content-research', 'content-copywriting', 'content-copy-review', 'content-generate', 'content-image-review')
```

**不改什么**: `content_publish`（浏览器发布需要 dispatch）、`content-pipeline`（主任务可以正常排队）
