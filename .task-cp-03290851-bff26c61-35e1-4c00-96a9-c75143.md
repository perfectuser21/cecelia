# Task Card: Playwright Evaluator 层 — 端到端行为验证

## 需求（What & Why）

**功能描述**：
在 /dev Stage 3（CI 通过后）增加 Playwright Evaluator 步骤，由 Claude subagent 读取 Task Card 中的 [BEHAVIOR] DoD 条目，动态生成并执行端到端验证脚本，逐条核实运行中的 Brain API 和 Dashboard 行为是否符合预期。失败时返回详细反馈给主 agent 修复，形成"CI通过 → 行为验证 → 修复 → 再验证"闭环。

**背景**：
现有 /dev 流程中，CI 通过后直接进入 Stage 4 写 Learning + 合并。但 CI 只做静态检查（格式/映射/语法），无法验证运行时行为是否符合 DoD [BEHAVIOR] 描述的预期。Playwright Evaluator 填补这个空缺。

**不做什么**：
- 不修改 spec_review / code_review_gate 的 seal 机制（保持现有防伪体系）
- 不在 CI 流水线内运行 Playwright（仅本地执行）
- 不对 [ARTIFACT] / [GATE] 条目做行为验证
- Brain API 不可用时直接 SKIP（pass-through），不报错

## 成功标准

- `packages/engine/lib/devloop-check.sh` 新增条件 3.5：CI 通过后检查 playwright_evaluator seal 文件（`.dev-gate-pw.<branch>`），FAIL → blocked，PASS/无seal → pass-through
- `packages/engine/skills/dev/steps/03-integrate.md` 新增步骤 3.3，描述触发 Playwright Evaluator subagent 的完整流程（检查服务可用性 → 提取 [BEHAVIOR] 条目 → 调用 subagent → 解析结果 → 写 seal）
- `packages/workflows/skills/playwright-evaluator/SKILL.md` 新增 Playwright Evaluator skill 定义，描述如何读取 Task Card、动态生成 API/UI 测试、执行并写 seal 文件

**devloop-check.sh 条件 3.5 设计**：
```
条件 3.5（在 CI PASS 后、条件 5 step_4_ship 前插入）：
  seal 文件 .dev-gate-pw.<branch> 存在？
    → verdict=FAIL → blocked（Playwright Evaluator 验证失败，需修复代码）
    → verdict=PASS → 继续
    → 不存在 → pass-through（subagent 尚未运行，允许继续；Brain 不可用时也走此路径）
```

**playwright-evaluator SKILL.md 核心职责**：
1. 读取 Task Card 的 [BEHAVIOR] 条目列表
2. 检查 Brain API (localhost:5221/api/brain/health) 可用性
   → 不可用 → 写 seal: {"verdict":"SKIP"}, 输出警告，退出
3. 对每条 [BEHAVIOR] 条目动态生成验证代码（curl/node fetch 对应 API 条目，Playwright 对应 UI 条目）
4. 逐条执行，记录 pass/fail 结果
5. 写 seal 文件 .dev-gate-pw.<branch>: {"verdict":"PASS"|"FAIL", "results":[...], "issues":[...]}
```
