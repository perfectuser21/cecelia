# Task Card: [SelfDrive] 修复错误黑洞：错误传播、重试与可观测性

## 需求（What & Why）

**功能描述**：
为当前高失败率问题补齐错误上报、重试策略和链路日志，减少静默失败，提升整体成功率与 MTTR。

**背景**：
根据任务历史，两个关键失败路径被确认：

1. **exit_code=1 不触发重试（最严重）**：
   当 dev 任务以 `exit_code=1` 失败时，cecelia-run 返回 `result = { failure_class: 'code_error', exit_code: 1, ... }`。但 `classifyDevFailure()` 无法解析这个对象结构（对 `failure_class` 字段视而不见），返回 `UNKNOWN → retryable: false`。导致 retry_count=0 的 code_error 任务直接沉底，**本该重试 3 次却 0 次重试**。

2. **Dev 重试不同步 DB retry_count 字段**：
   dev 重试路径只更新 `payload.retry_count`，不更新 `tasks.retry_count`（DB 字段），导致 `max_retries` 检查失效，重试统计不准。

3. **进程日志未纳入 error_message**：
   当 `result=null` 时，`error_message` 仅含 `exit_code` 和 stderr，但 Claude 进程的详细错误日志在 `/tmp/cecelia-{taskId}.log` 中，未被读取，造成调试黑洞。

**不做什么**：
- 不改变隔离/熔断逻辑
- 不改变 quarantine.js 的分类逻辑
- 不改变看门狗超时参数

## 成功标准

1. [BEHAVIOR] `classifyDevFailure` 在 `result.failure_class='code_error'` 时返回 `retryable: true`（retry < MAX_DEV_RETRY）
2. [BEHAVIOR] `classifyDevFailure` 在 `result=null + exitCode=1` 时返回 code_error 可重试结果
3. [ARTIFACT] dev 重试路径的 UPDATE SQL 包含 `retry_count = retry_count + 1`
4. [BEHAVIOR] result=null 时 `error_message` 尝试读取进程日志（/tmp/cecelia-{taskId}.log）并追加
5. [GATE] 所有现有测试通过（dev-failure-classifier.test.js + 新增测试）
