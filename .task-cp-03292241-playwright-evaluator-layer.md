# Task Card: Playwright Evaluator 层 — 端到端行为验证

## 需求（What & Why）
**功能描述**: 在 /dev Stage 3（CI 通过之后、Stage 4 之前）增加 Playwright Evaluator subagent 步骤。Evaluator 读取 Task Card 中的 DoD [BEHAVIOR] 条目，动态生成 Playwright 测试脚本，逐条对运行中的 Brain API（localhost:5221）和 Dashboard（localhost:5211）执行端到端验证。失败则返回详细反馈给主 agent 修复。

**背景**: 当前 /dev pipeline 在 CI 通过后直接进入 Stage 4 Ship。CI 只做静态检查和单元测试，无法验证运行时行为（如 API 端点是否正常返回、Dashboard 页面是否渲染正确）。增加 Playwright Evaluator 层可以在合并前验证 DoD BEHAVIOR 条目的真实运行状态。

**技术方案**: Evaluator 使用 curl + node 脚本验证 API 行为，使用已有的 CDP 连接方式验证 Dashboard 页面。不引入新的 Playwright npm 依赖。

**不做什么**:
- 不改动 Stage 1/Stage 2 的逻辑
- 不改动 CI L1-L4 流水线
- 不引入新的 Playwright npm 依赖（复用已有 CDP 基础设施）
- 不替代现有 spec_review / code_review_gate 机制

## 成功标准

- Stage 3 CI 通过后，devloop-check 自动触发 Playwright Evaluator 检查
- Evaluator 从 Task Card 提取 [BEHAVIOR] 条目并生成对应测试脚本
- 逐条执行验证，输出 pass/fail 结果
- Brain API /health 等基础端点覆盖
- 失败时产生结构化反馈（seal 文件），主 agent 可据此修复

## 实现方案
**要改的文件**:
- `packages/engine/lib/devloop-check.sh` — 在条件 4（CI completed+success）之后、条件 5（PR merged 检查）之前，新增条件 4.5：playwright_evaluator seal 文件验证。遵循 spec_review/code_review_gate 相同的 seal 验证模式
- `packages/engine/skills/dev/steps/03-integrate.md` — 在 3.2（CI 监控）之后新增 3.3 Playwright Evaluator 步骤
- `packages/engine/scripts/devgate/playwright-evaluator.sh` — 新建 evaluator 脚本
- `packages/engine/features/feature-registry.yml` — 注册新 feature
- Engine 版本 bump（6 文件）

**架构插入点**:
- devloop-check.sh 新条件位置：条件 4（CI success）和条件 5（PR merged）之间
- .dev-mode 字段名：`playwright_evaluator_status: pass|fail|pending`
- seal 文件命名：`.dev-gate-evaluator.{branch}`（JSON 格式，含 verdict/branch/timestamp/issues）
- 与现有 seal 机制一致：seal 存在+PASS→继续，seal 存在+FAIL→blocked，无 seal+.dev-mode 有 pass→自认证拦截

**Scope 锚定**: packages/engine/ 内部改动，不涉及 Brain/Dashboard 代码
