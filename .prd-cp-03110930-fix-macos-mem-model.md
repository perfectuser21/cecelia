## PRD - macOS 内存管理模型全面重构

**需求来源**: Cecelia 从 Linux 迁移到 Mac mini 后，内存管理模型完全失效。
Linux 的 os.freemem() ≈ MemAvailable（含可回收缓存），macOS 的 os.freemem() 只是
真正空闲页面（运行时常态 <200MB）。导致 Brain 长期误判内存过载、Tick 停派、
任务全部进隔离区的系统性故障。同时 Planner 去重逻辑不排除 quarantined 状态，
导致隔离任务被无限重建（昨晚 36 个僵尸任务根因）。

**功能描述**:
1. 内存度量：改用 macOS 原生 `vm.memory_pressure` sysctl 信号（0=正常/1=警告/2=危急/3=危机）
   作为主要压力信号，代替基于 os.freemem() 的计算。Linux 保持不变。
2. 容量模型：引入 used_ratio = (active+wired)/total 作为辅助指标，
   在 memory_pressure 无法获取时作为 fallback。
3. Planner 去重：dedup SQL 加入 'quarantined' 状态，防止隔离任务被重建。
4. 回滚 PR #812 错误实现：getAvailableMemoryMB() 用 free+inactive 会让
   Brain 在 active 已满时仍允许派发，改为正确实现（基于 memory_pressure 或 used_ratio）。

**涉及文件**:
- `packages/brain/src/platform-utils.js` — 新增 getMacOSMemoryPressure(), 修正 getAvailableMemoryMB()
- `packages/brain/src/executor.js` — checkServerResources() 改用新指标
- `packages/brain/src/planner.js` — dedup SQL 加 quarantined

**成功标准**:
## 成功标准
1. macOS 上 Brain 不再因 os.freemem() < 4057MB 而误报内存过载
2. `sysctl vm.memory_pressure` = 0 时，Brain effectiveSlots > 0
3. Planner 不再为已 quarantined 的任务重建新任务
4. PR #812 的 getAvailableMemoryMB() 被修正为不会导致过度派发

**非目标**:
- 不改 Linux 路径行为
- 不动 MEM_PER_TASK_MB 具体数值
- 不改 Tick 开关状态

### 异常处理

| 异常场景 | 处理方式 | 报警级别 |
|----------|----------|----------|
| sysctl vm.memory_pressure 失败 | fallback 到 used_ratio 计算 | P3 |
| vm_stat 解析失败 | fallback 到 os.freemem() | P3 |
