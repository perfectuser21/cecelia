# PRD: 修复 decomp-checker 层级命名约束

## 背景

Brain 在自动拆解 KR → Project 和 Project → Initiative 时，存在命名混乱问题：
- 创建了名为 "KR3-I1: 任务失败自动 RCA 实现" 的 Project（Initiative 粒度命名）
- 创建了名称与 KR 相同的"容器 Project"（如 KR 叫"自我学习闭环"，Project 也叫"自我学习闭环"）
- 导致 OKR 层级结构混乱，Project 和 Initiative 语义边界模糊

## 目标

在 `decomposition-checker.js` 的 Check 4 和 Check 5 的任务描述中，
添加明确的命名约束，让秋米在创建 Project/Initiative 时遵守正确的命名和粒度规范。

## 需求

### Check 4（KR → Project 关联）description 新增约束

```
⛔ 命名规范（CRITICAL）：
- 禁止使用 "KR3-I1:"、"I1:" 等 Initiative 编号前缀作为 Project 名称
- 禁止创建名称与 KR 相同或高度相似的"容器 Project"
- Project 名称必须描述具体功能模块：如"RCA 自动触发实现"、"Learning 策略更新系统"
- 时间验证：每个 Project 应是 1-2 周的工作量（不是 1-3 小时）
- 一个 KR 建议创建 1-3 个 Project，不超过 3 个
```

### Check 5（Project → Initiative 拆解）description 新增约束

```
⛔ 命名规范（CRITICAL）：
- Initiative 名称可以使用 I1/I2/I3 编号前缀（如"I1: RCA 触发机制设计"）
- Initiative 时间范围：1-3 小时，对应 7-15 个 Task
- 一个 Project 建议拆解为 3-8 个 Initiative
- Initiative 应是具体可执行的功能切片，不是整个 Project 的重复
```

## 验收标准

1. Check 4 的 task description 包含 Project 命名禁止规则
2. Check 5 的 task description 包含 Initiative 命名规范
3. 相关测试验证 description 包含约束文本
4. 不改变现有功能逻辑，只增加 description 约束文本
