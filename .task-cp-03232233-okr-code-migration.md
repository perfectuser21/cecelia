# Task: OKR 业务代码迁移（批次1）

## 背景

Migration 177 创建了新 OKR 表（visions/objectives/key_results/okr_projects/okr_scopes/okr_initiatives），Migration 179 迁移数据（UUID 保留），Migration 181 增加双向同步触发器，Migration 182 增加运营列。

现在需要将业务代码从旧表（goals/projects）迁移到新表。本 PR 迁移 3 个影响最大且逻辑最清晰的文件。

## 范围（3 个文件）

### 1. `packages/brain/src/kr-verifier.js`

- `JOIN goals g ON g.id = v.kr_id WHERE g.status = 'in_progress'` → `JOIN key_results g ON g.id = v.kr_id WHERE g.status IN ('active', 'in_progress')`
- `UPDATE goals SET progress = $1, metadata = ... WHERE id = $3` → `UPDATE key_results SET ...`
- `resetAllKrProgress` 中的 `JOIN goals` 和 `UPDATE goals` → `key_results`
- **理由**：UUID 完全保留，key_results 与 goals(area_kr/global_kr) 一一对应，直接替换即可

### 2. `packages/brain/src/orchestrator-realtime.js`

- `SELECT status, count(*)::int as cnt FROM goals GROUP BY status` → UNION 查询 7 张新 OKR 表的状态分布
- **理由**：realtime 状态面板需要展示完整 OKR 状态，旧表数据将逐步停止写入

### 3. `packages/brain/src/heartbeat-inspector.js`

- `SELECT title, progress FROM goals WHERE status = 'in_progress' ORDER BY priority ASC LIMIT 3` → `SELECT title, progress FROM key_results WHERE status IN ('active', 'in_progress') ORDER BY priority ASC LIMIT 3`
- **理由**：heartbeat 展示活跃 KR 进度，key_results 是进度追踪的主表

## 不做什么

- 不修改 initiative-closer.js（parent 链路复杂，含 project_kr_links）
- 不修改 kr-completion.js（objectives/key_results 语义映射复杂）
- 不修改 okr-tick.js（goals 泛型查询，需要全量分析）
- 不修改 kr-progress.js（依赖 project_kr_links）
- 不新增 DB migration（纯代码变更）
- 不修改 selfcheck.js（schema version 保持 182）

## 成功标准

1. kr-verifier.js 中不再引用 `FROM goals` / `UPDATE goals`，改为 `key_results`
2. orchestrator-realtime.js 状态查询改为 UNION 全部新 OKR 表
3. heartbeat-inspector.js active_okrs 查询改为 `key_results`
4. CI L1/L2/L3 全通过
