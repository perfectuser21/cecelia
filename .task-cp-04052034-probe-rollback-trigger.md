# Task Card：监控闭环 — capability-probe 连续失败自动触发回滚

## 背景

capability-probe.js 每小时探测 10 条核心链路。当探针失败时，当前仅触发 auto-fix 任务（创建 dev 任务），**不做自动回滚**。
如果核心链路（db、dispatch）连续失败，说明 Brain 本身可能损坏，此时 auto-fix 任务也无法正常执行，需要兜底的**自动回滚机制**。

## 目标

在 `capability-probe.js` 的 `runProbeCycle()` 中：

1. **持久化连续失败计数**：利用 `cecelia_events` 表（DB 级别，Brain 重启后不丢失）查询历史失败记录
2. **触发阈值（保守）**：
   - 同一探针**连续** 3 次探针周期均失败 → 触发回滚
   - 单次失败只记录 + 告警，不回滚
3. **触发回滚**：执行 `scripts/brain-rollback.sh`，捕获退出码
4. **告警增强**：飞书告警附加回滚触发原因 + 执行结果（成功/失败/脚本不存在）

## 技术方案

### 连续失败检测

查询最近 N 次 capability_probe 事件，检查指定探针是否在每次结果中都失败：

```sql
SELECT payload FROM cecelia_events
WHERE event_type = 'capability_probe'
ORDER BY created_at DESC
LIMIT 3
```

然后从每次 payload 的 `probes` 数组里找该探针的 `ok` 字段，判断是否连续失败。

### 回滚执行

使用 Node.js `child_process.spawnSync` 同步执行 `brain-rollback.sh`，超时 90 秒：
- 退出码 0 → 回滚成功
- 非零 → 回滚失败，日志打印 stderr
- 脚本不存在 → 降级：只告警不崩溃

### 告警

使用 `alerting.raise('P0', ...)` 发送飞书 P0 告警，区分：
- 连续失败触发回滚（含探针名称 + 失败次数）
- 回滚执行结果

## 变更文件

- `packages/brain/src/capability-probe.js` — 新增 `checkConsecutiveFailures()` + 回滚触发逻辑
- `packages/brain/src/__tests__/capability-probe-rollback.test.js` — 单元测试

## 约束

- 回滚前必须打印明确触发原因（探针名、连续失败次数、时间戳）
- 不影响现有 auto-fix 逻辑（回滚是额外兜底，两者并行）
- 回滚调用是同步的（防止 Brain 进程在回滚过程中继续出问题）
