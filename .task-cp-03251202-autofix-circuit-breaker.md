# Task Card: P0 修复 — RCA→auto-fix 熔断机制失效（不稳定 signature + 漏查活跃任务）

## 需求（What & Why）

**功能描述**：修复 `capability-probe.js` 中使用 `Date.now()` 导致 signature 不稳定，使 `dispatchToDevSkill` 的熔断守卫（MAX_AUTO_FIX_ATTEMPTS=3）永远无法生效；同时补充对 `queued/in_progress` 活跃任务的检查，防止在修复任务执行过程中重复创建。

**背景**：前一轮修复（cp-03242032-p0p1-arch-fixes）给 `auto-fix.js` 加了 3 次失败上限守卫，但 `capability-probe.js` 每次调用 `dispatchToDevSkill` 时使用 `probe_${f.name}_${Date.now()}` 作为 signature，`Date.now()` 使每次 signature 唯一，守卫查询永远返回 0 条匹配，死循环无法被熔断。同时守卫只查 `status='failed'`，不查 `queued/in_progress`，导致正在修复中仍可重复创建任务。

**不做什么**：不修改 `monitor-loop.js`（使用 `generateErrorSignature` 生成稳定 hash，无此问题）；不重构熔断架构（P1）；不改 `MAX_AUTO_FIX_ATTEMPTS` 值。

## 成功标准

1. [ARTIFACT] `capability-probe.js` 的 signature 使用稳定格式 `probe_${f.name}`，不含 `Date.now()`
2. [ARTIFACT] `auto-fix.js` 的 `dispatchToDevSkill` 守卫同时检查 `queued` 和 `in_progress` 状态任务（有活跃任务则跳过创建）
3. [BEHAVIOR] 同一 probe 失败重复触发时，只创建一个 auto-fix 任务，后续触发被熔断跳过
4. [GATE] CI 全部通过
