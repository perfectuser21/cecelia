# Task Card: Sprint Contract 断点续跑 — 压缩后从正确轮次恢复

## 需求（What & Why）

**功能描述**：sprint-contract-loop.sh 每轮结束后写 checkpoint，恢复时读 .sprint-contract-state.{branch} 自动判断已完成几轮、从正确轮次继续，不重复已完成的轮次。支持 --resume 标志或自动检测。

**背景**：上下文压缩发生在 Sprint Contract 两轮之间时，.sprint-contract-state 记录了 round，但 agent 上下文丢失不知道已到哪一轮，重新开始导致重复对抗。现有脚本每次被调用时会读取 state 文件的 round 值，但没有机制告知 main agent "已经收敛过了，不用重跑"。

**核心问题**：
1. state 文件存在 + blocker_count == 0 时，main agent 上下文压缩后不知道是否已收敛
2. 主 agent 的 while true 循环（01-spec.md Step 4）在压缩恢复后无法判断应该从哪轮继续
3. 已删除 seal 文件（用于下一轮）但上下文丢失，导致重新从第 0 轮开始

**不做什么**：不改变 Sprint Contract 的核心双独立提案架构；不修改 Generator/Evaluator subagent 协议。

## 成功标准

1. [ARTIFACT] sprint-contract-loop.sh 新增 --resume 标志，读 state 文件判断是否已收敛
2. [BEHAVIOR] 调用 `bash sprint-contract-loop.sh BRANCH --resume` 时：若 state 文件存在且 blocker_count==0，脚本直接 exit 0 并打印"已收敛，跳过"；若 state 文件不存在或未收敛，正常执行
3. [ARTIFACT] 01-spec.md Step 4 增加恢复检测逻辑：while 循环开始前调用 --resume 检查，若已收敛直接跳过整个 Sprint Contract
4. [GATE] CI 全部通过
