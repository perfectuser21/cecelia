## 问题描述

调查KR进度全为0%的根本原因：
1. 检查 `/api/brain/okr/current` API 为何返回空 key_results
2. 追踪 objectives → key_results 的数据关联，验证进度聚合是否正确
3. 分析 [内容生成] 等KR的0%是数据延迟还是度量配置错误
4. 确认 [管家闭环] 为何能正确显示40%，其他KR为何不能

## 诊断结论

**根因 1**：`/api/brain/okr/current` 查询用 `ORDER BY created_at DESC LIMIT 5`，
最新创建的 4 个 objectives 均无 KR，导致有 4 个 KR 的 `4a2c8a08`（ZenithJoy 自媒体）排第 6 被截断。

**根因 2**：migration 208 修复后，`内容生成` 和 `自动发布` 两个 KR verifier 用了同一个 SQL，
无法区分"AI内容产出"和"平台发布成功"两个不同指标。

## 修复范围

1. `packages/brain/src/routes/okr-hierarchy.js` — 改 `/current` 端点只返回有活跃 KR 的 objectives
2. `packages/brain/migrations/209_differentiate_kr_verifier_queries.sql` — 区分内容生成 vs 自动发布的 SQL

## 成功标准

- `/api/brain/okr/current` 包含所有有活跃 KR 的 objectives（2个：Cecelia 系统稳定 + ZenithJoy 自媒体）
- ZenithJoy 自媒体的 4 个内容 KR 可见
- 内容生成 和 自动发布 KR 使用不同 SQL
