## PRD - cleanup.sh 完成后自动触发 worktree-gc

**需求来源**: Brain Task 0d253352 — cleanup.sh 完成后自动触发 worktree-gc，不再靠别人
**功能描述**: 在 cleanup.sh 的 `_mark_cleanup_done()` 执行完成后，从主仓库后台触发 `worktree-gc.sh`，自动清理已合并的 worktree（包括刚完成的这个）。目前 cleanup.sh 完成后注释说"GC 将由 worktree-gc.sh 从主仓库执行"，但实际上没有人主动调用它，导致 worktree 残留。
**涉及文件**:
- `packages/engine/skills/dev/scripts/cleanup.sh` — 添加 GC 触发逻辑
- `packages/engine/VERSION` — 版本 bump
- `packages/engine/package.json` — 版本 bump
- `packages/engine/package-lock.json` — 版本 bump
- `packages/engine/.hook-core-version` — 版本 bump
- `packages/engine/regression-contract.yaml` — 版本 bump
- 根 `package-lock.json` — engine 条目版本 bump
- `packages/engine/scripts/feature-registry.yml` — 新增 changelog 条目

**非目标**: 不修改 worktree-gc.sh 本身；不修改 stop-dev.sh；不改变 GC 逻辑

## 成功标准

1. cleanup.sh 的 `_mark_cleanup_done` 完成后，自动从主仓库路径后台执行 `worktree-gc.sh`
2. GC 以后台方式运行（`&`），不阻塞 cleanup 流程
3. GC 必须 `cd "$MAIN_WT"` 再运行（从主仓库执行）
4. 如果 `worktree-gc.sh` 不存在或 `$MAIN_WT` 为空，静默跳过，不报错
5. Engine 版本 bump 6 个文件全部同步
6. feature-registry.yml 新增 changelog 条目
7. PR title 包含 `[CONFIG]` tag

### 异常处理 & 报警规范

| 异常场景 | 处理方式 | 报警级别 | eventType |
|----------|----------|----------|-----------|
| GC 脚本不存在 | 静默跳过，打印 INFO | P3 | N/A |
| MAIN_WT 为空 | 静默跳过，打印 WARN | P3 | N/A |
| GC 后台进程失败 | 独立进程，cleanup 不感知 | P3 | N/A |
